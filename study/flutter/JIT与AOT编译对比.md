# JIT 与 AOT 编译对比

## 一、一句话区分

```
JIT —— 边运行边编译，启动慢但预热后快，灵活（支持热重载）
AOT —— 运行前全编译好，启动快、性能稳定，但丧失运行时灵活性
```

## 五、JIT 的内部机制 —— 分层编译

现代 JIT 不是"一次性编译"，而是层层递进：

```
                JIT 分层编译（以 Android ART / V8 为例）

  调用次数
     │
     │                                 ┌─────────────────┐
     │                         非常多   │ 第3层: JIT 优化   │
     │                     ┌──────────▶│ - 激进内联         │
     │                     │           │ - 去虚拟化         │
     │                     │           │ - 逃逸分析         │
     │             较多     │           │ - 循环展开         │
     │          ┌──────────┤           └─────────────────┘
     │          │ 第2层:   │
     │          │ 快速JIT  │  ← 基于 profile 数据优化
     │          │ 简单内联  │
     │   少量   └──────────┤
     │ ┌────────┐          │
     │ │ 第1层: │          │
     │ │ 解释器 │  ← 逐条解释字节码，收集方法调用计数
     │ └────────┘          │
     │                     │
     └─────────────────────┴───────────────────────────────▶ 代码复杂度

  关键点：
  - 99% 的代码走解释器就够了（初始化代码、异常路径等）
  - 只有 1% 的热点代码需要深度优化
  - 运行时 Profiler 根据实际数据做 PGO（Profile-Guided Optimization）
```

```java
// 一个方法的"升级"过程示意（以 ART 为例）

public int hotMethod(int a, int b) {
    return a + b;  // 看似简单
}

// 阶段1（解释器）:
//   每条指令 push/pop 操作数栈，一次加法耗时 ~100 周期

// 阶段2（快速 JIT）:
//   直接编译为几条 ARM 指令，加法耗时 ~5 周期

// 阶段3（优化 JIT + 内联后）:
//   hotMethod 被内联到调用方，消除了方法调用开销
//   如果参数是常量，加法在编译期完成，耗时 ~0 周期
```

## 六、AOT 的优化手段 —— 时间换空间

AOT 因为编译不占用"运行时"，可以花更多时间做全局优化：

```
                  AOT 全程序优化管线

  Dart 源码
     │
     ▼
  ┌─────────────────────────┐
  │ 1. TFA (Type Flow Analysis) │
  │    全局类型流分析            │
  │    推导每个变量、返回值的      │
  │    具体类型（而非 interface）  │
  └─────────────┬───────────┘
                ▼
  ┌─────────────────────────┐
  │ 2. Devirtualization     │
  │    去虚拟化               │
  │    原本的接口调用          │
  │    obj.method()           │
  │    → 变成具体类的方法调用   │
  │    ConcreteClass.method() │
  └─────────────┬───────────┘
                ▼
  ┌─────────────────────────┐
  │ 3. Inlining             │
  │    内联                   │
  │    将小方法体直接嵌入调用处  │
  │    消除方法调用的栈帧开销   │
  └─────────────┬───────────┘
                ▼
  ┌─────────────────────────┐
  │ 4. Constant Folding     │
  │    常量折叠               │
  │    x = 2 + 3 → x = 5    │
  │    编译期计算完           │
  └─────────────┬───────────┘
                ▼
  ┌─────────────────────────┐
  │ 5. Dead Code Elimination│
  │    死代码消除             │
  │    未调用的方法、未使用的   │
  │    分支全部移除           │
  └─────────────┬───────────┘
                ▼
  ┌─────────────────────────┐
  │ 6. 生成目标平台机器码      │
  │    ARM / ARM64 / x64    │
  └─────────────────────────┘
```

**AOT 的一个关键能力：TFA（类型流分析）**

```dart
// 这段代码在 JIT 和 AOT 下的处理完全不同

abstract class Animal { void speak(); }
class Dog extends Animal {
  @override void speak() => print('Woof');
}

void makeSpeak(Animal a) {
  a.speak();  // 虚调用？还是具体调用？
}

void main() {
  makeSpeak(Dog());  // 实际传入的就是 Dog
}

// JIT 下：
//   makeSpeak 中的 a.speak() 是虚调用——通过 vtable 查找
//   运行时才能确定是哪个类的 speak

// AOT + TFA 下：
//   编译器分析整个程序的调用链，发现 makeSpeak 只被传入 Dog
//   于是优化：
//     1. makeSpeak 参数从 Animal 推断为 Dog
//     2. speak 调用从虚调用变为直接调用 Dog.speak
//     3. Dog.speak 足够小，被内联到 main 中
//     4. 最终 main 等价于 print('Woof')
```

## 八、混合策略 —— 为什么现代运行时都不"二选一"

```
         现代运行时的演进方向：混合编译
         ─────────────────────────────────

  第一阶段（过去）:
    Java → 解释器 → JIT（HotSpot C2）→ 全 AOT (Android 5-6)

  第二阶段（现在）:
    Java → C1 + C2 分层编译
    ART  → 解释器 + JIT + 充电时 AOT (profile-guided)
    V8   → Ignition + Sparkplug + Maglev + TurboFan
    Dart → Debug JIT + Release AOT（两个极端，Profile 是折中）

  第三阶段（趋势）:
    → 都往"解释器 + 多层JIT + 离线AOT缓存"方向走
    → 利用运行时数据（profiling）反哺 AOT 编译
    → 最大化两种模式的优点

  这个演进的原因：
    纯 JIT 问题：启动慢、内存大、不适合移动端
    纯 AOT 问题：编译慢（开发体验差）、无法利用运行时特征做优化
    混合方案  ：开发用 JIT → 发布用 AOT → 运行时收集 profile → 下次 AOT 更精准
```

## 九、Android ART 的 JIT+AOT 混合 —— 典型案例

```
  Android 7.0+ 的 ART 混合编译流程：

  ┌──────────┐
  │ 安装 APK  │
  └────┬─────┘
       │ 直接装 DEX（不编译）
       ▼
  ┌──────────┐    调用够多     ┌──────────────┐
  │ 解释执行   │ ────────────▶ │ JIT 编译热点   │
  │          │               │ 缓存到 jit-cache│
  └──────────┘               └───────┬──────┘
       │                             │
       │                    记录 profile 数据
       │                    （哪些方法被调用、
       │                     调用顺序、分支走向）
       │                             │
       │                    充电+空闲时触发
       ▼                             ▼
  ┌────────────────────────────────────────┐
  │         dex2oat  (AOT 编译)             │
  │  根据 profile 只编译"真正用到的代码"      │
  │  → 生成 .oat 文件 (ELF 格式的目标平台机器码) │
  └────────────────────────────────────────┘
       │
       ▼
  ┌──────────┐
  │ 下次启动   │ → 优先加载 .oat，跳过解释/JIT
  │ 直接跑机器码│
  └──────────┘

  效果：
  - 安装快（不用等 AOT 编译）
  - 越用越快（profile 引导的精准编译）
  - 不浪费空间（只编译用到的代码，不是整个 APK）
```

## 十一、面试题

**Q1: JIT 和 AOT 的本质区别是什么？**

> JIT（Just-In-Time）在**运行时**将字节码/IR 编译为机器码，编译器与程序共存，可利用运行时 profile 数据做针对性优化。AOT（Ahead-Of-Time）在**构建时**完成编译，运行时直接执行机器码，无编译开销，启动快。本质区别是"编译时机"——运行时 vs 构建时。

**Q2: 为什么 Flutter Debug 用 JIT，Release 用 AOT，而不是都用一种？**

> Debug 阶段需要热重载——修改代码后秒级看到效果，这依赖 JIT 的动态代码替换能力。Release 阶段追求启动速度和运行性能，不需要热重载，AOT 直接生成优化后的机器码，移除 VM 和编译器，包体积更小、内存更省。两种模式各取所长。

**Q3: JIT 为什么要分层编译？不分层行不行？**

> 不分层直接深度编译所有代码会非常慢（启动时卡住），而且绝大部分代码（初始化、异常处理等）只执行一两次，不值得优化。分层编译的策略是：先用解释器快速启动并收集热点信息，只对真正频繁执行的热点代码做深度优化。这保证了启动速度，又能优化到真正影响性能的代码。

**Q4: "AOT 性能一定优于 JIT" 这话对吗？**

> 不完全对。AOT 在启动速度和资源占用上确实优于 JIT。但 JIT 有运行时信息优势——它能根据实际数据做 PGO（如常数传播、分支预测），某些场景下 JIT 的峰值性能可能**超过** AOT。比如 Java HotSpot 的 C2 编译器的峰值性能通常优于提前编译。正确说法是：AOT 启动快、性能稳定；JIT 启动慢、有预热过程、但峰值性能可能更高。

**Q5: Android APK 为什么安装后越用越快？**

> Android 7.0+ 的 ART 采用 JIT + AOT 混合方案。刚安装时 DEX 文件用解释器跑（慢），系统收集热点代码的 profile 数据（哪些方法被频繁调用、走哪个分支），在设备充电+空闲时通过 dex2oat 工具将这些热点方法 AOT 编译为 .oat 机器码。下次启动时直接加载 .oat 跳过解释/JIT，启动变快、运行变流畅。这是一个"越用越了解你"的过程。

**Q6: JIT 的"热重载"和 AOT 的"Tree Shaking"分别是什么原理？**

> 热重载：Dart VM 监听文件变更，将修改的文件重新编译为 kernel dill，然后通过 Service Protocol 将新的函数体注入 VM，替换旧函数，同时保持 State 不变，Widget 树自动重建。Tree Shaking：AOT 编译器通过 TFA（类型流分析）确定整个程序的调用图，将从未被引用的类、方法、字段从编译产物中移除——dart:core 中的很多代码在 Hello World 里根本不会用到，全部被 shake 掉。
