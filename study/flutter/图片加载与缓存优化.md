# 图片加载与缓存优化——内存与卡顿的重灾区

> **面试核心问题**：一张 4000x3000 的网络图，显示在 80x80 的缩略图里，会占多少内存？如果是 100 张呢？怎么优化？本文把图片性能的每个环节说清楚。

---

## 一、图片在 Flutter 里经历了什么

```
网络/本地文件
    │
    ▼
┌─────────────┐
│  加载字节流   │   比如 3MB 的 JPG 压缩数据
└─────────────┘
    │
    ▼
┌─────────────┐
│  解码（dart:ui│   解压成位图：W × H × 4 字节 (RGBA)
│   instantiate│   4000×3000×4 = 48 MB ！！！
│   ImageCodec)│
└─────────────┘
    │
    ▼
┌─────────────┐
│  Image 对象  │   dart:ui.Image，持有 GPU 纹理
│  (GPU 纹理)  │
└─────────────┘
    │
    ▼
┌─────────────┐
│  ImageCache │   LRU 缓存，默认 100 张 / 100 MB
└─────────────┘
    │
    ▼
  RawImage widget
    │
    ▼
  paint 到 Canvas
```

**面试核心认知**：图片的"大小"分三个层面：
- **磁盘/网络大小**：JPG/PNG 压缩后，几百 KB
- **解码后内存大小**：W × H × 4，单位字节（RGBA 每像素 4 字节）
- **GPU 纹理大小**：等于解码后大小，占用 GPU 内存

**4000×3000 图片 = 48 MB 内存**，而不是几百 KB。这就是为什么图片轻松爆内存。

## 二、cacheWidth / cacheHeight：解码时降采样

### 2.1 错误写法

```dart
// ❌ 只在布局层改尺寸，解码出来还是 4000×3000
Image.network(
  url,
  width: 80,
  height: 80,
)
```

`width/height` 只影响绘制时的缩放，**解码后仍然是原图大小**。48 MB 的位图一分不少。

### 2.2 正确写法

```dart
// ✅ 解码时就按 160×160（2x 屏）采样
Image.network(
  url,
  width: 80,
  height: 80,
  cacheWidth: 160,   // = width × devicePixelRatio（大致即可）
  cacheHeight: 160,
)
```

**内存对比**：
- 不加：4000 × 3000 × 4 = **48 MB**
- 加了：160 × 160 × 4 = **100 KB**

300 倍差距。

### 2.3 cacheWidth/cacheHeight 的底层原理

```dart
// Image 源码的相关逻辑（简化）
Future<Codec> _loadAsync(...) async {
  final codec = await PaintingBinding.instance.instantiateImageCodecWithSize(
    buffer,
    getTargetSize: (width, height) {
      // 按 cacheWidth/cacheHeight 决定目标尺寸
      return ImmutableBuffer.targetSize(cacheWidth, cacheHeight);
    },
  );
  return codec;
}
```

Skia 在解码时直接按目标尺寸生成位图，使用 **bilinear/box filter** 降采样，内存和质量都可控。

### 2.4 cacheWidth 应该设多大？

```dart
final dpr = MediaQuery.of(context).devicePixelRatio;  // 2.0 / 3.0
final targetWidth = (displayWidth * dpr).round();
```

**经验法则**：
- 列表缩略图：`cacheWidth = displayWidth * 2`（2x DPR 够清晰）
- 大图预览：不设 cacheWidth，但限制 ImageCache 数量
- 小图标：优先用 SVG 或矢量字体图标

## 三、ImageCache：全局 LRU

### 3.1 默认配置

```dart
PaintingBinding.instance.imageCache;
// 默认：
//   maximumSize      = 1000  （最多 1000 张）
//   maximumSizeBytes = 100 MB（总内存 100 MB）
```

### 3.2 命中逻辑

```
Image.network(url) 流程：
  ①  生成 cacheKey（基于 url + scale + configuration）
  ②  查 ImageCache
      │
      ├─ 命中 → 直接拿 ImageStream 返回
      └─ 不命中 → 真正下载 + 解码 → 放入 ImageCache
  ③  LRU 驱逐：超过 maximumSize 或 maximumSizeBytes 时，
      从最久未使用的开始踢
```

### 3.3 调优

```dart
void main() {
  // 适合大图浏览 app（相册、漫画）
  PaintingBinding.instance.imageCache
    ..maximumSize = 200
    ..maximumSizeBytes = 200 * 1024 * 1024;  // 200 MB
  
  runApp(const MyApp());
}
```

**调优原则**：
- 小图多的 app（社交 feed）：`maximumSize` 调大（500-1000），bytes 不动
- 大图少的 app（相册预览）：`maximumSize` 调小（50-100），bytes 调大（200MB）
- 内存紧张：`imageCache.clear()` 配合 `didHaveMemoryPressure` 监听

### 3.4 低端机内存告警处理

```dart
class _App extends StatefulWidget with WidgetsBindingObserver {
  @override
  void initState() {
    super.initState();
    WidgetsBinding.instance.addObserver(this);
  }

  @override
  void didHaveMemoryPressure() {
    // 系统内存告警，清缓存
    PaintingBinding.instance.imageCache.clear();
    PaintingBinding.instance.imageCache.clearLiveImages();
  }
}
```

## 四、cached_network_image：磁盘缓存

Flutter 内置的 `Image.network` **只做内存缓存**，重启 App 就没了。生产环境基本都用 `cached_network_image`：

```dart
CachedNetworkImage(
  imageUrl: url,
  width: 80,
  height: 80,
  memCacheWidth: 160,           // 内存中解码尺寸
  memCacheHeight: 160,
  maxWidthDiskCache: 800,       // 磁盘缓存最大宽度
  placeholder: (_, __) => const _Skeleton(),
  errorWidget: (_, __, ___) => const Icon(Icons.broken_image),
  fadeInDuration: const Duration(milliseconds: 100),
)
```

```
访问图片流程：
  ①  查内存缓存（ImageCache）
       │
       ├─ 命中 → 返回
       └─ miss
  ②  查磁盘缓存（sqflite + 文件）
       │
       ├─ 命中 → 读文件 → 解码 → 放内存缓存 → 返回
       └─ miss
  ③  下载 → 写磁盘 → 解码 → 放内存缓存 → 返回
```

## 五、GIF 和长图的特殊处理

### 5.1 GIF 的内存陷阱

```
一张 GIF，200 帧，每帧 500×500
  → 解码内存 = 500 × 500 × 4 × 200 = 200 MB ！！
```

Flutter 的 `Image.network` 解码 GIF 时会**把所有帧都解到内存**，播放时按帧切换。长 GIF 极易爆内存。

**优化**：
- 长 GIF 转成 WebP 动图（支持更好的压缩）
- 超长动图考虑转视频用 `video_player`
- 使用 `cached_network_image` 的 `ImageCacheManager` 控制 concurrent

### 5.2 长图（高度 10000+）

```
长图解码：100 × 10000 × 4 = 4 MB（相对还好）
但 GPU 纹理有最大尺寸限制（通常 4096 或 8192），超过会：
  - iOS：降级或不显示
  - Android：抛异常
```

**解决**：
- 长图切片显示（每段 4000 高度）
- 使用 `PhotoView` 库支持超长图

## 七、按需加载：只在需要时解码

### 7.1 滑动列表里的预加载

```dart
class _ImageItem extends StatelessWidget {
  final String url;
  const _ImageItem({required this.url});

  @override
  Widget build(BuildContext context) {
    return Image.network(
      url,
      cacheWidth: 300,
      cacheHeight: 300,
      // 使用 frameBuilder 控制加载动画
      frameBuilder: (_, child, frame, wasSyncLoaded) {
        if (wasSyncLoaded) return child;
        return AnimatedOpacity(
          opacity: frame == null ? 0 : 1,
          duration: const Duration(milliseconds: 200),
          child: child,
        );
      },
    );
  }
}
```

### 7.2 预加载

```dart
// 在列表滑动到位时，提前预加载下 N 张
precacheImage(NetworkImage(url), context);
```

`precacheImage` 会把图片放进 ImageCache，等真正 `Image.network` 时秒显。

**用法**：在 `didChangeDependencies` 里提前预加载详情页的图。

## 八、自定义 ImageProvider：进阶场景

如果你需要：
- 从加密文件读
- 从内存 Uint8List 读
- 自己实现磁盘缓存

可以继承 `ImageProvider`：

```dart
class EncryptedImageProvider extends ImageProvider<EncryptedImageProvider> {
  final String path;
  EncryptedImageProvider(this.path);

  @override
  Future<EncryptedImageProvider> obtainKey(ImageConfiguration config) {
    return SynchronousFuture(this);
  }

  @override
  ImageStreamCompleter loadImage(EncryptedImageProvider key, ImageDecoderCallback decode) {
    return MultiFrameImageStreamCompleter(
      codec: _loadCodec(decode),
      scale: 1.0,
    );
  }

  Future<Codec> _loadCodec(ImageDecoderCallback decode) async {
    final bytes = await _readAndDecrypt(path);
    final buffer = await ImmutableBuffer.fromUint8List(bytes);
    return decode(buffer);
  }
  
  @override
  bool operator ==(Object other) =>
      other is EncryptedImageProvider && other.path == path;
  @override
  int get hashCode => path.hashCode;
}
```

**关键**：`operator ==` 和 `hashCode` 必须正确实现，这决定了 ImageCache 是否命中。

## 九、图片加载性能的实战组合

### 9.1 社交 feed 的最佳实践

```dart
class PostImage extends StatelessWidget {
  final Post post;
  const PostImage({super.key, required this.post});

  @override
  Widget build(BuildContext context) {
    final dpr = MediaQuery.of(context).devicePixelRatio;
    final width = MediaQuery.of(context).size.width;
    
    return AspectRatio(
      aspectRatio: post.imageAspectRatio,  // 从接口返的比例，避免 layout 抖动
      child: CachedNetworkImage(
        imageUrl: post.coverUrl,
        fit: BoxFit.cover,
        memCacheWidth: (width * dpr).round(),  // 按实际分辨率解码
        placeholder: (_, __) => Container(
          color: Color(post.dominantColorInt),   // 服务端返主色占位
        ),
        errorWidget: (_, __, ___) => const Icon(Icons.broken_image),
      ),
    );
  }
}
```

**关键设计**：
1. `AspectRatio` 用接口返回的比例撑起占位，避免图片加载完成后整个列表抖动
2. `dominantColorInt` 用主色做渐入过渡，比灰色骨架更高级
3. `memCacheWidth` 按 DPR 计算，不浪费不模糊

### 9.2 相册大图预览

```dart
PhotoView(
  imageProvider: CachedNetworkImageProvider(url),
  loadingBuilder: (_, event) => CircularProgressIndicator(
    value: event == null ? null : event.cumulativeBytesLoaded / event.expectedTotalBytes!,
  ),
  minScale: PhotoViewComputedScale.contained,
  maxScale: PhotoViewComputedScale.covered * 3,
)
```

**注意**：预览不加 `cacheWidth`（用户就是要看清），但全局 ImageCache 要控制容量，避免多张大图叠加爆内存。

## 十、图片性能的坑

### 10.1 忘记 dispose ImageStream

很少直接用 `ImageStream`，但如果你自己写了 `ImageProvider.resolve(...)`，必须手动 `removeListener`，否则会泄漏。

### 10.2 Widget 切换导致重复下载

```dart
// ❌ 每次 setState 都重新构造 NetworkImage
Image(image: NetworkImage('https://...'))  
// 但 ImageCache 的 key 是基于 provider 的 == + hashCode 的，
// NetworkImage 这两个方法正确实现，所以实际不会重复下载
```

真正的坑是**自定义 ImageProvider 没实现 ==/hashCode**，每次都 miss cache。

### 10.3 在 build 里 new NetworkImage

```dart
// 性能没影响（NetworkImage 本身很轻），但代码风格上应该用 Image.network
```

### 10.4 SVG 的误区

Flutter 原生**不支持 SVG**。用 `flutter_svg` 第三方库。但要注意：
- SVG 每次渲染都是运行时解析 + 绘制，比栅格图慢
- 大量 SVG（比如 500 个图标）会导致渲染卡顿
- 解决：对静态 SVG 做一次栅格化缓存

## 十一、面试题精选

### Q1：一张 4000×3000 的 JPG 图片，在 Flutter 中显示占多少内存？

**答**：JPG 在磁盘上可能只有几百 KB，但**解码后内存 = 4000 × 3000 × 4 = 48 MB**。Flutter（dart:ui）按 RGBA 解码，每像素 4 字节。这就是图片性能问题的根源。

### Q2：`width/height` 和 `cacheWidth/cacheHeight` 的区别？

**答**：
- `width/height`：只影响绘制时的缩放，解码后的位图仍然是原图大小
- `cacheWidth/cacheHeight`：在解码阶段就按目标尺寸降采样，直接减少位图和 GPU 纹理的内存

列表里的缩略图必须设 `cacheWidth/cacheHeight`，否则一眼爆内存。

### Q3：Flutter 的 ImageCache 默认多大？怎么调？

**答**：默认 `maximumSize = 1000`（张数），`maximumSizeBytes = 100 MB`。可以通过 `PaintingBinding.instance.imageCache` 调整。调优原则：
- 小图多：张数调大
- 大图少：总字节调大
- 内存紧张：监听 `didHaveMemoryPressure` 主动 `clear()`

### Q4：`Image.network` 会缓存到磁盘吗？

**答**：**不会，只在内存**（ImageCache）。App 重启就没了。生产环境用 `cached_network_image` 库，它在内存缓存之外加了磁盘缓存（sqflite + 文件）。

### Q5：同一个 URL 的图片在两个地方用，会下载两次吗？

**答**：不会。`NetworkImage` 以 `url + scale` 作为 cacheKey，ImageCache 命中后两处共享同一个 `ImageStream`。这要求 `ImageProvider` 的 `operator ==` 和 `hashCode` 正确实现，自定义 Provider 很容易在这里踩坑。

### Q6：列表里滑动到大图就卡，怎么排查？

**答**：五步走
1. DevTools → Memory 看 ImageCache 大小，是不是爆了
2. 加 `cacheWidth/cacheHeight`，解码尺寸降下来
3. 列表 `cacheExtent` 减小，避免预解码太多
4. 改用 `cached_network_image`，首次下载后走磁盘
5. 图片 CDN 侧也裁剪好，不要客户端拉原图再降采样

### Q7：GIF 为什么容易爆内存？怎么优化？

**答**：Flutter 解码 GIF 会把**所有帧**解到内存。200 帧 500×500 的 GIF 就是 200 MB。优化：
- 用 WebP 动图替代 GIF
- 长动画转视频用 `video_player`
- 短 GIF 限制帧数

### Q8：`FadeInImage` 和自己写 `AnimatedOpacity` 有什么区别？

**答**：`FadeInImage` 是官方封装，提供 placeholder → image 的渐入，内部用 `ImageStream` 监听加载完成再切换。自己写 `AnimatedOpacity` + `frameBuilder` 可以实现同样效果，还能做骨架屏 → 图片的更细控制。

### Q9：图片 CDN 返回的分辨率应该客户端决定还是服务端决定？

**答**：客户端根据屏幕 DPR + 显示尺寸告诉 CDN，CDN 直接返裁剪好的图。比如 `https://cdn.com/img.jpg?w=320&h=320`。这样：
- 客户端不用传原图 → 省带宽
- 不用 cacheWidth 降采样（反正服务端已经裁好）
- GPU 纹理更小

阿里云 OSS、七牛、AWS S3 + CloudFront 都支持 URL 参数裁剪。

### Q10：`precacheImage` 适合什么场景？

**答**：在即将进入的页面预热图片缓存。典型场景：
- 列表 item 里预加载详情页 banner
- 轮播图提前加载下一张
- 游戏/漫画预加载下一页

用法：`precacheImage(NetworkImage(url), context)` 放在 `didChangeDependencies` 里。

### Q11：为什么图片切页动画时第一次会卡？

**答**：典型的 Shader 编译卡顿（Shader Jank）。Hero 动画伴随的 saveLayer/模糊/圆角 clip 触发 Skia 编译新 Shader。解决：启用 Impeller（iOS 已默认，Android 逐步启用），或旧版本用 SkSL 预热。

### Q12：`Image.asset` 和 `Image.network` 性能差别？

**答**：
- `Image.asset`：读本地资源，无网络、无磁盘缓存需要，只解码一次放 ImageCache
- `Image.network`：额外有网络下载 + 解码，不做磁盘缓存（要自己加 `cached_network_image`）

都会走同一个 ImageCache。资源图记得用 flavors 做 `1.0x/2.0x/3.0x`，Flutter 自动按 DPR 选。

## 十二、图片性能优化 checklist

```
开发时：
  ✅ 列表图片必加 cacheWidth/cacheHeight
  ✅ 确定比例的用 AspectRatio 撑占位
  ✅ 网络图用 cached_network_image（别用原生 Image.network）
  ✅ placeholder / errorWidget 必写
  ✅ GIF 考虑换 WebP

上线前：
  ✅ 调整 ImageCache 大小匹配 app 特性
  ✅ 监听 didHaveMemoryPressure 清缓存
  ✅ 低端机降级策略（DPR = 1.0x，或用更低分辨率的 URL）
  ✅ CDN URL 带裁剪参数
  ✅ 打包时排除未使用的 asset 图

排查时：
  ✅ DevTools Memory → filter `ImageCache`
  ✅ flutter run --trace-systrace 看 decodeImage 耗时
  ✅ imageCache.liveImageCount 看活跃图片数
```

**一句话总结**：**按需解码、按屏缓存、按场景清理**。
