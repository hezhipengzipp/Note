# Flutter 混合开发方案

---

## 前言：两个方向

```
方向一：原生 App 里嵌入 Flutter 页面（Add-to-App）
  → 已有的 Android/iOS 项目，部分页面用 Flutter 写
  → 大厂最常见的场景

方向二：Flutter 里嵌入原生控件（PlatformView）
  → Flutter 页面中需要地图、WebView 等原生组件
  → 无法用 Skia 自绘的场景
```

---

## 一、方向一：原生嵌入 Flutter（Add-to-App）

### 1.1 基本接入方式

```
原生 Android App
┌──────────────────────────────────────────┐
│                                          │
│  NativeActivity ──→ NativeActivity       │
│       │                                  │
│       │  点击"去 Flutter 页面"             │
│       ▼                                  │
│  FlutterActivity / FlutterFragment       │
│  ┌────────────────────────────────────┐  │
│  │  FlutterEngine                     │  │
│  │  ┌──────────────────────────────┐  │  │
│  │  │  Dart VM                     │  │  │
│  │  │  运行 Flutter UI             │  │  │
│  │  │  自绘在 SurfaceView 上       │  │  │
│  │  └──────────────────────────────┘  │  │
│  └────────────────────────────────────┘  │
│       │                                  │
│       │  返回                             │
│       ▼                                  │
│  NativeActivity                          │
│                                          │
└──────────────────────────────────────────┘
```

```kotlin
// Android 端：最简单的接入方式
// 方式一：直接启动 FlutterActivity
startActivity(
    FlutterActivity
        .withNewEngine()           // 创建新 Engine
        .initialRoute("/home")     // 指定 Flutter 初始路由
        .build(context)
)

// 方式二：用 FlutterFragment 嵌入已有 Activity
val flutterFragment = FlutterFragment
    .withNewEngine()
    .initialRoute("/settings")
    .build()

supportFragmentManager.beginTransaction()
    .replace(R.id.container, flutterFragment)
    .commit()
```

### 1.2 FlutterEngine 管理——核心问题

**问题**：Engine 启动很慢（初始化 Dart VM + 编译 Dart 代码），首次打开 Flutter 页面会白屏 1~2 秒。

```
不预热（❌）：

用户点击 ──→ 创建 Engine ──→ 初始化 Dart VM ──→ 运行 Dart ──→ 渲染
              └──── 白屏 1~2 秒 ────────────────┘


预热 Engine（✅）：

App 启动时 ──→ 提前创建 Engine ──→ 放入缓存
                                    │
用户点击 ──→ 从缓存取 Engine ──→ 直接渲染（几乎无白屏）
```

```kotlin
// Application.onCreate() 中预热
class MyApp : Application() {
    override fun onCreate() {
        super.onCreate()
        
        // 预热 Engine 并缓存
        val engine = FlutterEngine(this).apply {
            // 预先执行 Dart 入口点，Dart 代码开始运行
            dartExecutor.executeDartEntrypoint(
                DartExecutor.DartEntrypoint.createDefault()
            )
        }
        // 放入全局缓存
        FlutterEngineCache.getInstance()
            .put("main_engine", engine)
    }
}

// 使用缓存的 Engine 打开页面（秒开）
startActivity(
    FlutterActivity
        .withCachedEngine("main_engine")  // 用缓存而非新建
        .build(context)
)
```

### 1.3 单 Engine vs 多 Engine

```
单 Engine（主流方案）：
┌─────────────────────────────────────┐
│           唯一的 FlutterEngine        │
│           唯一的 Dart VM              │
│           唯一的 Isolate              │
│                                     │
│  Flutter 页面 A ──┐                  │
│  Flutter 页面 B ──┼─ 共享同一个 Engine │
│  Flutter 页面 C ──┘                  │
│                                     │
│  优点：内存占用小（~40MB 只算一次）     │
│  缺点：页面间状态隔离需要自己管理       │
│        多页面共享一个 Navigator        │
└─────────────────────────────────────┘

多 Engine（Flutter 3.x 支持）：
┌────────────────┐  ┌────────────────┐
│  Engine A       │  │  Engine B       │
│  Dart VM (共享)  │  │  Dart VM (共享)  │
│  Isolate A      │  │  Isolate B      │
│  Flutter 页面 A  │  │  Flutter 页面 B  │
│                 │  │                 │
│  ~40MB          │  │  ~180KB 增量     │
└────────────────┘  └────────────────┘
              ↑
    Flutter 3.x 的轻量级 Engine
    共享 Dart VM，每个额外 Engine 只增加 ~180KB
    页面天然隔离
```

## 二、混合导航栈——最大难题

原生页面和 Flutter 页面交替跳转时，导航栈管理非常复杂：

```
用户操作路径：
  原生 A → Flutter B → 原生 C → Flutter D → 返回 → 返回 → 返回

导航栈：
┌──────────────────────────────┐
│  Flutter D   ← 栈顶          │
│  原生 C                      │
│  Flutter B                   │
│  原生 A      ← 栈底          │
└──────────────────────────────┘

问题：
· Flutter 有自己的 Navigator 栈
· Android 有自己的 Activity/Fragment 栈
· 两个栈怎么同步？返回键谁处理？
· Flutter B 和 Flutter D 是同一个 Engine 吗？
```

### 方案对比

#### 方案一：每个 Flutter 页面一个 FlutterActivity（简单粗暴）

```
原生 A(Activity) → Flutter B(FlutterActivity) → 原生 C(Activity)

  优点：导航栈天然和 Android 一致，返回键自动处理
  缺点：单 Engine 下多个 FlutterActivity 共享同一个 Dart 环境
        多 Engine 下内存开销大（老版本 Flutter）
```

#### 方案二：FlutterBoost（闲鱼开源，业界主流）

```
┌─────────────────────────────────────────────────┐
│                FlutterBoost                       │
│                                                  │
│  原生侧                      Flutter 侧           │
│  ┌──────────────┐           ┌──────────────┐     │
│  │ BoostContainer│           │ BoostNavigator│    │
│  │ (Activity/VC) │←─ 同步 ─→│ (Flutter路由)  │    │
│  └──────────────┘           └──────────────┘     │
│                                                  │
│  核心思路：                                        │
│  · 每个 Flutter 页面对应一个原生容器                  │
│    （Activity / ViewController）                   │
│  · 共享单个 FlutterEngine                          │
│  · 页面切换时把 Engine 的渲染 attach 到当前容器       │
│  · 原生和 Flutter 的路由表统一管理                    │
└─────────────────────────────────────────────────┘

跳转流程：
原生 A ──open("/flutterB")──→ 创建 BoostContainer(Activity)
                               │
                               ├─ attach Engine 到这个 Activity
                               ├─ Flutter 侧 push "/flutterB" 路由
                               └─ 渲染 Flutter 页面 B

Flutter B ──open("/nativeC")──→ 通过 Channel 通知原生
                                 │
                                 └─ 原生侧 startActivity(NativeC)

返回时：
原生 finish() → detach Engine → Flutter pop → 回到上一个容器
```

#### 方案三：Flutter 3.x 多 Engine（官方推荐的新方向）

```dart
// Flutter 3.x: 每个原生容器一个轻量级 Engine
// 不再需要 FlutterBoost 这种中间层

// Android 端
class PageA : FlutterActivity() {
    // 自动使用一个轻量级 Engine
    // 和其他页面的 Engine 共享 Dart VM
    // 额外内存开销只有 ~180KB
}

class PageB : FlutterActivity() {
    // 另一个轻量级 Engine
    // 天然隔离，不用操心状态冲突
}
```

### 方案选型总结

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| 单 Engine + 单 Activity | 最简单 | 不支持混合栈 | Flutter 占主体的 App |
| FlutterBoost | 成熟稳定，大厂验证 | 引入三方依赖，有学习成本 | 已有大型原生 App |
| 多 Engine (3.x) | 官方支持，天然隔离 | 页面间通信需走 Channel | 新项目或渐进迁移 |

## 三、方向二：Flutter 嵌入原生控件（PlatformView）

Flutter 自绘引擎画不了地图、WebView 等原生控件，需要"开个洞"嵌入。

### 3.1 两种实现方式

#### Virtual Display（虚拟显示）

```
Flutter 层                          原生层
┌──────────────────┐              ┌──────────────────┐
│  Flutter UI       │              │  创建原生 View     │
│  ┌──────────┐    │              │  (MapView)         │
│  │ 纹理占位   │←──│── Texture ──│  渲染到 VirtualDisplay│
│  │           │    │    ID       │  → 生成纹理 ────────│→ 传给 Flutter
│  └──────────┘    │              └──────────────────┘
└──────────────────┘

原生 View → 渲染到虚拟屏幕 → 生成纹理 → Flutter 拿纹理当图片绘制

优点：Flutter 完全掌控合成，无线程问题
缺点：触摸事件需要转发，可能有键盘/无障碍问题
```

#### Hybrid Composition（混合合成，推荐）

```
┌──────────────────────────────────────┐
│              Android View 层级        │
│                                      │
│  ┌─ FlutterSurfaceView ──────────┐  │
│  │  Flutter UI (Skia 绘制)        │  │
│  │       ┌──────────────┐        │  │
│  │       │  原生 MapView  │ ← 直接 │  │
│  │       │  (真实View)    │   嵌入  │  │
│  │       └──────────────┘        │  │
│  │  Flutter UI (Skia 绘制)        │  │
│  └────────────────────────────────┘  │
└──────────────────────────────────────┘

原生 View 直接插入 Android View 层级，和 Flutter Surface 混合显示

优点：原生 View 是真实的，触摸/键盘/无障碍都正常
缺点：Android 10 以下有性能开销（需要额外 Surface 同步）
```

### 3.2 PlatformView 代码示例

```dart
// Flutter 侧：声明一个 PlatformView
class NativeMapView extends StatelessWidget {
  @override
  Widget build(BuildContext context) {
    return AndroidView(
      viewType: 'native-map-view',  // 和原生注册的 ID 对应
      creationParams: {'lat': 39.9, 'lng': 116.4},
      creationParamsCodec: StandardMessageCodec(),
    );
  }
}
```

```kotlin
// Android 侧：注册原生 View 工厂
class MapViewFactory : PlatformViewFactory(StandardMessageCodec.INSTANCE) {
    override fun create(context: Context, id: Int, args: Any?): PlatformView {
        val params = args as Map<String, Any>
        return NativeMapPlatformView(context, params)
    }
}

class NativeMapPlatformView(context: Context, params: Map<String, Any>) 
    : PlatformView {
    
    private val mapView = MapView(context).apply {
        // 用 params 里的经纬度初始化地图
    }
    
    override fun getView(): View = mapView
    override fun dispose() { mapView.onDestroy() }
}

// 在 FlutterEngine 中注册
flutterEngine.platformViewsController
    .registry
    .registerViewFactory("native-map-view", MapViewFactory())
```

### 3.3 Virtual Display vs Hybrid Composition

| | Virtual Display | Hybrid Composition |
|---|---|---|
| 原理 | 原生 View 渲染到虚拟屏幕生成纹理 | 原生 View 直接插入 View 层级 |
| 触摸事件 | Flutter 转发，可能不准 | 原生直接处理，完全正常 |
| 键盘输入 | 有问题（WebView 输入框） | 正常 |
| 无障碍 | 需要额外适配 | 原生支持 |
| 性能 | 有纹理拷贝开销 | Android 10+ 性能好 |
| 推荐度 | 兜底方案 | **首选** |

```
┌─────────────────────────────────────────────────────────────┐

│        │                   │                   │           │
│        └───────────┬───────┘                   │           │
│                    │                           │           │
│            MethodChannel                       │           │
│            EventChannel                        │           │
│                    │                           │           │
│        ┌───────────┴───────────────────────────┘           │
│        │                                                   │
│        ▼                                                   │
```

## 五、面试回答模板

**Q: Flutter 混合开发有哪些方案？怎么选型？**

> 混合开发分两个方向。原生嵌入 Flutter 用 Add-to-App 方案，核心是 FlutterEngine 管理——通过 FlutterEngineCache 预热避免白屏。导航栈管理是最大难点：简单场景可以每个 Flutter 页面对应一个 FlutterActivity；复杂的混合栈场景可以用 FlutterBoost，它让每个 Flutter 页面对应一个原生容器，共享单 Engine，路由统一管理；Flutter 3.x 之后多 Engine 方案成为新方向，多个轻量级 Engine 共享 Dart VM，每个额外 Engine 只增加约 180KB，天然隔离。Flutter 嵌入原生控件用 PlatformView，首选 Hybrid Composition 方式，原生 View 直接插入 View 层级，触摸和键盘都正常；Virtual Display 作为兜底方案。

**Q: PlatformView 的两种模式有什么区别？**

> Virtual Display 是把原生 View 渲染到一块虚拟屏幕上生成纹理，Flutter 拿这个纹理当图片绘制，好处是 Flutter 完全掌控合成，缺点是触摸事件需要转发且有键盘问题。Hybrid Composition 是把原生 View 直接插入 Android View 层级，和 Flutter Surface 混合显示，触摸、键盘、无障碍都正常工作，是目前的首选方案，缺点是 Android 10 以下有额外的 Surface 同步开销。
