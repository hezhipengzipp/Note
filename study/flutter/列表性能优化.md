# 列表性能优化——从 ListView 到 Sliver 的完整打法

> **面试高频场景 TOP 1**：列表卡。本文覆盖 `ListView` 三种用法的区别、`itemExtent / prototypeItem / cacheExtent` 的作用、`keepAlive` 的代价、Sliver 家族的适用场景，以及高刷/嵌套/懒加载/分页的实战套路。

---

## 一、ListView 的三种用法：性能差异巨大

```dart
// ① ListView(children: [...])       一次性构建所有 item
// ② ListView.builder(itemBuilder)   按需构建（懒加载）
// ③ ListView.separated              builder + 分隔线
```

### 1.1 三者的差异

```
ListView(children: [...])
  │
  ▼
  构造时立刻创建 List<Widget>，全部 1000 个 item 都生成
  ↓
  SliverList 收到这个 List，用 SliverChildListDelegate 包装
  ↓
  问题：1000 个 Widget 对象 + 1000 个 RenderObject 实例

ListView.builder(itemBuilder: ...)
  │
  ▼
  SliverChildBuilderDelegate(itemBuilder)
  ↓
  仅在滑入视口 ± cacheExtent 区间时才调用 itemBuilder
  ↓
  不可见的 item 不创建 Widget、不创建 RenderObject
```

### 1.2 什么时候可以用 `ListView(children: [...])`

```
item 数量 < 20 且每个 item 很轻 → 可以用 children 模式，省得写 builder
其他情况一律 builder
```

### 1.3 源码速读

```dart
// listview.dart 片段（简化）
class ListView extends BoxScrollView {
  ListView({List<Widget>? children, ...}) 
    : childrenDelegate = SliverChildListDelegate(children, ...);

  ListView.builder({required IndexedWidgetBuilder itemBuilder, int? itemCount, ...})
    : childrenDelegate = SliverChildBuilderDelegate(itemBuilder, childCount: itemCount, ...);
}
```

核心差异在 `ChildDelegate`：
- `SliverChildListDelegate` 持有完整 `List<Widget>`
- `SliverChildBuilderDelegate` 持有 `itemBuilder` 函数，按需调用

## 二、itemExtent / prototypeItem：避免重复测量

### 2.1 不固定高度的代价

默认情况下，列表滑动时每个新 item 都要：
1. 父 SliverList 下发 Constraints
2. item 自己做 layout，测量自己的高度
3. 上报 Size 给父节点累加

```
        ┌─────────────────┐
Slider →│  layout item 1  │ 3ms
  ↓     ├─────────────────┤
        │  layout item 2  │ 2ms
        ├─────────────────┤
        │  layout item 3  │ 4ms
        └─────────────────┘
总计 9ms + build + paint，一帧可能吃掉全部预算
```

### 2.2 itemExtent：主轴方向固定尺寸

```dart
ListView.builder(
  itemExtent: 80,            // 每个 item 高度固定 80
  itemCount: 1000,
  itemBuilder: (_, i) => _Item(i),
)
```

`itemExtent` 带来两个收益：

```
① 跳过测量
   父 Sliver 直接按 itemExtent 计算 item 位置
   不需要 layout 子节点来知道高度

② O(1) 定位
   scrollTo(index: 500) 时：
     offset = 500 * 80 = 40000 ← 直接算
   不用像变长列表那样 "从头累加到第 500 项"
```

### 2.3 prototypeItem（3.10+）：用原型推断尺寸

```dart
ListView.builder(
  prototypeItem: const _Item(index: 0),   // 用这个 item 的尺寸作为所有项的尺寸
  itemCount: 1000,
  itemBuilder: (_, i) => _Item(i),
)
```

适合"所有 item 结构类似但高度不是硬编码常量"的场景。

### 2.4 三种情况怎么选

| 情况 | 怎么办 |
|---|---|
| 每个 item 高度相同且已知 | `itemExtent: 80` |
| 每个 item 高度相同但难算（字体、padding） | `prototypeItem:` |
| 每个 item 高度不同 | 不设，接受测量成本 |

## 三、cacheExtent：视口外预渲染的缓冲区

```
                 ┌─────────────────┐
                 │                 │
                 │   cacheExtent   │   ← 视口上方，提前构建 250dp
                 │      上         │
     视口上沿  ──────────────────
                 │                 │
                 │                 │
                 │    可见区域     │
                 │                 │
                 │                 │
     视口下沿  ──────────────────
                 │                 │
                 │   cacheExtent   │   ← 视口下方，提前构建 250dp
                 │      下         │
                 └─────────────────┘
```

**默认值**：`cacheExtent = 250.0`（dp）

```dart
ListView.builder(
  cacheExtent: 500,  // 扩大预渲染范围
  // ...
)
```

### 3.1 面试题："cacheExtent 越大越好吗？"

**不是**。权衡：
- **变大**：滑动更不容易白屏（用户看到"快滑到底了"还有内容）
- **变大的代价**：内存占用变高、首次 layout 时间变长（初始要构建更多 item）

**建议值**：
- 列表项轻量（纯文字）：500-1000
- 列表项有大图：200-300 足够
- 60fps 流畅滑动需要至少能覆盖 "一次 fling 的 100ms 滑动距离"

## 四、AutomaticKeepAliveClientMixin：让 item 不被回收

### 4.1 场景：包含 StatefulItem 的列表

```dart
class _VideoItem extends StatefulWidget {...}

class _VideoItemState extends State<_VideoItem> {
  VideoPlayerController? _controller;
  
  @override
  void initState() {
    super.initState();
    _controller = VideoPlayerController.network(widget.url)..initialize();
  }
  // ...
}
```

**问题**：item 滑出视口后被销毁，controller 也跟着 dispose；滑回来时重新 initialize，视频重新加载。

### 4.2 keepAlive 方案

```dart
class _VideoItemState extends State<_VideoItem>
    with AutomaticKeepAliveClientMixin {  // ← mixin
  
  @override
  bool get wantKeepAlive => true;          // ← 告诉父 Sliver 保留我
  
  @override
  Widget build(BuildContext context) {
    super.build(context);                  // ← 必须调
    return VideoPlayer(_controller!);
  }
}
```

### 4.3 代价

```
不开 keepAlive：
  视口外的 State 被销毁 → 内存低 → 回来时重建
  
开 keepAlive：
  所有滑过的 item State 都留着 → 内存高 → 回来时瞬开

1000 个 item 的长列表，全开 keepAlive → 可能撑爆内存
```

**使用原则**：只对"**重建代价 > 内存代价**"的 item 开 keepAlive（视频、复杂输入状态、WebView）。

### 4.4 Selective KeepAlive

```dart
// 只对特定 index 保活
@override
bool get wantKeepAlive => widget.index < 5;
```

## 五、Sliver 家族：解决混合滚动的唯一方案

### 5.1 为什么需要 Sliver

```
需求：一个滚动页面，包含
  - 顶部图片（视差滚动）
  - 吸顶的 TabBar
  - 一个 GridView
  - 一个 ListView

用 SingleChildScrollView + Column 的问题：
  Column 要知道 child 高度 → GridView/ListView shrinkWrap 计算全部 item
  → 失去懒加载 → 卡 + 内存爆炸 ❌

正解：把它们都变成 Sliver，统一交给 CustomScrollView 调度
```

### 5.2 Sliver 家族

```
CustomScrollView
  slivers: [
    SliverAppBar              ← 可折叠的 AppBar
    SliverPersistentHeader    ← 吸顶的分组头
    SliverList / SliverFixedExtentList
    SliverGrid
    SliverToBoxAdapter        ← 把普通 Widget 塞进 Sliver 体系
    SliverPadding
    SliverFillRemaining       ← 占据剩余空间
  ]
```

### 5.3 一个典型结构

```dart
CustomScrollView(
  slivers: [
    SliverAppBar(
      expandedHeight: 200,
      pinned: true,
      flexibleSpace: FlexibleSpaceBar(
        title: const Text('Home'),
        background: Image.network(bannerUrl, fit: BoxFit.cover),
      ),
    ),
    const SliverToBoxAdapter(child: _CategoryTabs()),
    SliverGrid(
      gridDelegate: const SliverGridDelegateWithFixedCrossAxisCount(crossAxisCount: 2),
      delegate: SliverChildBuilderDelegate(
        (_, i) => _GridItem(index: i),
        childCount: 20,
      ),
    ),
    SliverList(
      delegate: SliverChildBuilderDelegate(
        (_, i) => _ListItem(index: i),
        childCount: 100,
      ),
    ),
  ],
)
```

### 5.4 Sliver 性能优势

- 每个 Sliver 独立 layout，互不拖累
- Sliver 内部的 item 依然懒加载
- 统一的滚动物理（惯性、bouncing），用户体验一致

## 六、嵌套滚动的两个坑

### 6.1 在 Column 里塞 ListView

```dart
// ❌ 报错 or 不滚
Column(
  children: [
    const Text('Header'),
    ListView.builder(...),   // 父 Column 要知道高度，ListView 无法确定
  ],
)

// ✅ 方案 1：Expanded
Column(
  children: [
    const Text('Header'),
    Expanded(child: ListView.builder(...)),
  ],
)

// ✅ 方案 2：shrinkWrap + 禁用滚动（内部列表很短时）
Column(
  children: [
    const Text('Header'),
    ListView.builder(
      shrinkWrap: true,                         // 自己算高度
      physics: const NeverScrollableScrollPhysics(),  // 禁用自己的滚动
      ...
    ),
  ],
)
```

**警告**：`shrinkWrap: true` 会强制 ListView 一次性 layout 全部 item，**失去懒加载**。只能用于 item 数量小的场景。

### 6.2 两个可滚动区域嵌套

```dart
// NestedScrollView：SliverAppBar + TabBarView(内部也滚动)
NestedScrollView(
  headerSliverBuilder: (_, __) => [
    SliverAppBar(...),
  ],
  body: TabBarView(
    children: [
      CustomScrollView(slivers: [...]),
      CustomScrollView(slivers: [...]),
    ],
  ),
)
```

## 七、分页加载：scrollController + 阈值触发

```dart
class _FeedPageState extends State<FeedPage> {
  final _controller = ScrollController();
  final _items = <Item>[];
  bool _loading = false;
  bool _noMore = false;
  int _page = 0;

  @override
  void initState() {
    super.initState();
    _controller.addListener(_onScroll);
    _loadMore();
  }

  void _onScroll() {
    // 距离底部 500px 时触发加载
    if (_controller.position.pixels >= 
        _controller.position.maxScrollExtent - 500) {
      _loadMore();
    }
  }

  Future<void> _loadMore() async {
    if (_loading || _noMore) return;
    _loading = true;
    final newItems = await api.fetch(page: _page++);
    if (mounted) {
      setState(() {
        _items.addAll(newItems);
        _noMore = newItems.isEmpty;
        _loading = false;
      });
    }
  }

  @override
  void dispose() {
    _controller.dispose();
    super.dispose();
  }

  @override
  Widget build(BuildContext context) {
    return ListView.builder(
      controller: _controller,
      itemCount: _items.length + 1,
      itemBuilder: (_, i) {
        if (i == _items.length) {
          return _noMore 
            ? const _NoMoreIndicator() 
            : const _LoadingIndicator();
        }
        return _ItemTile(item: _items[i]);
      },
    );
  }
}
```

**关键细节**：
1. 阈值（这里 500px）要大于一屏 fling 的距离，否则用户会在底部等待
2. 加载状态要防抖：`if (_loading || _noMore) return;`
3. `mounted` 检查防止异步回来时页面已销毁

### 8.1 一个"重型 item"的完整优化

```dart
class _Item extends StatelessWidget {

              ],

        ],

    );
  }
```


### 8.2 ListView.builder 的 addRepaintBoundaries

```dart
ListView.builder(
  addRepaintBoundaries: true,   // 默认 true，每个 item 包一层 RepaintBoundary
  addAutomaticKeepAlives: true, // 默认 true，响应 KeepAlive 协议
  addSemanticIndexes: true,     // 默认 true，无障碍
  ...
)
```

**什么时候关？**
- item 极其简单（单个 Text），RepaintBoundary 本身的开销比收益大 → `addRepaintBoundaries: false`
- 列表项需要复杂语义（自定义 Semantics）→ `addSemanticIndexes: false`

## 九、面试题精选

### Q1：`ListView` 和 `ListView.builder` 底层有什么区别？

**答**：
- `ListView(children: [...])` 用的是 `SliverChildListDelegate`，构造时立即生成完整 `List<Widget>`
- `ListView.builder` 用的是 `SliverChildBuilderDelegate`，内部按需调用 `itemBuilder`，只构建视口 + cacheExtent 范围内的 item

当 item 数量大于 20 或 item 本身较重时，必须用 builder，否则会有大量无效 Widget/RenderObject 创建。

### Q2：`itemExtent` 为什么能提升性能？

**答**：
1. 跳过 layout：父 Sliver 直接按 itemExtent 计算每个 item 的位置，不需要 layout 子节点
2. 快速定位：scrollToIndex 时 `offset = index * itemExtent`，O(1) 复杂度
3. 快速渲染：可以并行布局，因为位置不依赖前面 item 的实际尺寸

### Q3：`cacheExtent` 是什么？默认值多少？

**答**：视口上下各自的预渲染距离，单位 dp，默认 250。越大滑动越不容易白屏，但内存占用高、首次加载慢。对纯文字列表可以开到 1000，对图片/视频列表建议 200-300。

### Q4：ListView 嵌套在 Column 里为什么会报错？

**答**：Column 要求 child 有确定高度，但 ListView 默认是 `Expanded` 行为，希望占满父约束。解决：
1. 用 `Expanded(child: ListView(...))` 让 Column 分配剩余空间
2. 用 `ListView(shrinkWrap: true, physics: NeverScrollableScrollPhysics())` 让 ListView 自测，但会失去懒加载，仅适合短列表

### Q5：`AutomaticKeepAliveClientMixin` 会带来什么副作用？

**答**：让 item 的 State 不随滑出视口而销毁，好处是回来时能秒开（保留视频/输入状态），代价是内存占用随滑过的 item 数线性增长。长列表全开会 OOM，应该只对重建代价高的 item 选择性开启。

### Q6：什么是 Sliver？为什么要有它？

**答**：Sliver 是 Flutter 的"可滚动单元"。它和普通 Widget 的区别是用 `SliverConstraints`（包含 scrollOffset、remainingPaintExtent 等）来布局，能让多个 Sliver 在同一个 Viewport 里协同滚动。用途：当你需要 SliverAppBar + GridView + ListView 混合滚动时，必须用 Sliver，否则失去懒加载。

### Q7：`shrinkWrap: true` 什么情况下能用？

**答**：只在内部列表的 item 数量确定且很小（比如 < 10）时用。它会强制一次性 layout 全部 item，失去 builder 的懒加载优势。常见场景：对话框里显示几个选项、卡片内的小列表。

### Q8：为什么列表滑动时每个 item 里的 `DateTime.now()` 导致滑动变卡？

**答**：`DateTime.now()` 是系统调用，虽然不慢但在 build 方法里调用会导致：① item 的 `operator ==` 对不上（时间一直变），触发不必要的 rebuild；② 如果配合 `DateFormat` 还会有字符串解析开销。优化：时间值在 Model 里冻结，或者预先格式化。

### Q9：`ListView.builder` 的 `addRepaintBoundaries` 是干什么的？

**答**：默认 `true`，会给每个 item 自动包一层 `RepaintBoundary`。好处是 item 独立成 Layer，某个 item 重绘不会影响其他 item。对复杂 item 很有用，但极轻的 item（单 Text）加了反而因为 Layer 开销得不偿失。

### Q10：如何处理"列表分页 + 上拉加载 + 下拉刷新"？

**答**：
```dart
RefreshIndicator(
  onRefresh: _refresh,
  child: ListView.builder(
    controller: _controller,      // 监听滚动触发加载
    itemCount: _items.length + 1, // +1 放底部指示器
    itemBuilder: (_, i) {
      if (i == _items.length) return const _LoadingOrEnd();
      return _Item(item: _items[i]);
    },
  ),
)
```

关键点：
1. 阈值触发而非到底触发（距离底部 500px 就开始加载）
2. 加锁防止重复请求（`if (_loading) return`）
3. `mounted` 检查异步安全

### Q11：嵌套滚动怎么做？

**答**：用 `NestedScrollView` 或 `CustomScrollView`。
- `NestedScrollView`：外层滚动 + 内层可滚动 body（典型是 TabBarView），处理吸顶/折叠 header
- `CustomScrollView`：多个 Sliver 共享同一个 Viewport，所有东西都是同一个滚动系统

避免：多个 `SingleChildScrollView` 嵌套，或 `ListView` 嵌套，都会有手势冲突。

### Q12：超长列表（10 万+）怎么优化？

**答**：
1. **必须 builder**，绝不 children 模式
2. `itemExtent` 或 `prototypeItem` 让定位 O(1)
3. `ListView.builder` + `ValueKey` 让跳转和增删更稳定
4. 数据分段加载，前端只持有 "已浏览 + 预加载" 的数据
5. 极端情况用 `InfiniteListView` 第三方库或 `TwoDimensionalScrollView`（3.16+）
6. 考虑 backend pagination + 本地 SQLite 缓存，而不是一次性拉全量到内存

## 十、决策树：遇到列表性能问题怎么办

```
卡顿现象？
 │
 ├─ 首次渲染慢
 │    └─ 用了 ListView(children:) 吗？ → 改 ListView.builder
 │       item 太重？ → 拆 Widget、const、cacheWidth
 │
 ├─ 滑动掉帧
 │    ├─ item 高度不固定？ → itemExtent / prototypeItem
 │    ├─ 大图？ → cacheWidth/cacheHeight、fadeInDuration
 │    ├─ 滤镜/阴影？ → 降低分辨率、避免 saveLayer
 │    └─ RepaintBoundary 开着吗？ → builder 默认开，自定义的记得加
 │
 ├─ 来回滑动时 State 丢失
 │    └─ 该 item 重建代价高吗？ → AutomaticKeepAliveClientMixin
 │
 ├─ 页面首屏白一下才出
 │    └─ cacheExtent 太小？ → 调大到 500-1000
 │
 └─ 分页/下拉刷新卡
      └─ 异步请求前做了同步计算？ → 搬到 Isolate
         setState 范围太大？ → 用 ValueListenableBuilder 缩小
```

记住一句话：**懒加载 + 固定尺寸 + 绘制隔离 + 按需解码**，是列表性能四大件。
