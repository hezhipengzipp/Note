# 动画与渲染管线优化——Vsync / Shader Jank / Impeller

> **面试核心**：Flutter 渲染管线有哪些阶段？UI Thread 和 Raster Thread 分别干什么？Shader 编译为什么卡？Impeller 相比 Skia 强在哪？本文把渲染和动画两大板块彻底讲透。

---

## 一、Flutter 渲染管线：一帧的完整旅程

### 1.1 时间线视图

```
 Vsync 信号到达
 │
 ├─ UI Thread（Dart） ─────────────────────────────────
 │   ① Animate        ~0.5ms    推进动画值
 │   ② Build          ~2ms      调用 build()
 │   ③ Layout         ~2ms      RenderObject layout
 │   ④ Paint          ~1ms      生成 Layer Tree
 │   ⑤ Composite      ~0.5ms    Scene 构建
 │                    ────────
 │                    预算 8ms（60fps）或 4ms（120fps）
 │
 └─ Raster Thread（C++/GPU） ─────────────────────────
     ⑥ Skia/Impeller 栅格化    ~3ms
     ⑦ GPU 提交                 ~2ms
                               ────────
                                预算 8ms

 ↓  16.6ms 内完成 → 60fps 流畅
    16.6ms ~ 33ms → 掉 1 帧
    超过 33ms     → 明显卡顿
```

### 1.2 两条线程的分工

```
┌─────────────────────────────────────────────────────────────┐
│  Platform Thread    主线程，处理原生消息                      │
│                                                             │
│  UI Thread          Dart 代码运行的地方                       │
│                     - build / layout / paint                │
│                     - 产出 Layer Tree                       │
│                                                             │
│  Raster Thread      渲染专用线程                             │
│                     - 消费 Layer Tree                       │
│                     - Skia/Impeller 光栅化                  │
│                     - GPU 指令提交                          │
│                                                             │
│  IO Thread          文件/网络 IO                             │
└─────────────────────────────────────────────────────────────┘
```

**关键认知**：
- 两条线程**并行**工作：UI Thread 画第 N 帧的 Layer Tree 的同时，Raster Thread 在栅格化第 N-1 帧
- 卡顿要定位"卡在哪条线程"：
  - UI Thread 超时 → build/layout 问题 → 优化 Widget
  - Raster Thread 超时 → 绘制问题 → 避免 saveLayer、优化 Shader

## 二、Vsync：帧信号的驱动器

```
屏幕刷新率 60Hz → 每 16.6ms 一次 Vsync

硬件 Vsync ──────┬──────┬──────┬──────┬──
                │      │      │      │
                ▼      ▼      ▼      ▼
           ┌───────────────────────────┐
           │ Choreographer (Android)    │
           │ CADisplayLink (iOS)        │
           └───────────────────────────┘
                │
                ▼
           ┌───────────────────────────┐
           │ Flutter Engine             │
           │ SchedulerBinding           │
           └───────────────────────────┘
                │
                ▼
           onBeginFrame    onDrawFrame
           (动画推进)       (build/layout/paint)
```

**Ticker 怎么拿到 Vsync**：

```dart
class _MyState extends State<MyWidget> with SingleTickerProviderStateMixin {
  late final _controller = AnimationController(vsync: this, ...);
}

// this 实现了 TickerProvider，提供 createTicker
// 背后 Ticker 注册到 SchedulerBinding.instance.scheduleFrameCallback()
// 每帧被唤醒执行 tick(Duration elapsed)
```

## 三、Shader Jank：Flutter 最臭名昭著的性能问题

### 3.1 问题现象

```
用户第一次：
  - 点按钮出涟漪 → 第一次卡 1 帧
  - 页面切换 → 第一次卡 2-3 帧
  - 弹出 Modal → 第一次卡 1 帧

之后就不卡了
```

### 3.2 根因

```
Skia 使用 OpenGL/Metal，每个绘制指令组合需要对应的 Shader 程序
  │
  ▼
第一次绘制新组合：
  Dart 指令 → Skia 生成 GLSL/MSL 源码 → GPU 编译 → 缓存
                                       ↑
                              耗时 10-100ms，就是这里卡
  │
  ▼
之后：直接用已缓存的 Shader → 快
```

具体触发场景：
- 首次页面切换动画（涉及新的 blur / clip 组合）
- 首次按下按钮的涟漪效果
- 首次 Hero 动画
- 首次模糊/滤镜

### 3.3 解决方案 1：SkSL 预热（旧方案）

```bash
# ① 用代表性操作录制 Shader
flutter run --profile --cache-sksl
# 手动触发各种动画、切页、弹窗等
# 退出 App，产生 sksl.json 文件

# ② 打包时带上 sksl.json
flutter build apk --bundle-sksl-path=flutter_01.sksl.json
```

运行时 Flutter 在启动阶段预编译这些 Shader。

**缺点**：
- 需要录制所有场景，漏了的还是会卡
- 不同机型/驱动可能 Shader 不兼容
- 维护成本高

### 3.4 解决方案 2：Impeller（新方案，3.7+）

```
Skia 架构：
  Dart 绘制指令
       │
       ▼
  Skia 生成 Shader 源码（首次遇到时）
       │
       ▼
  GPU 编译 Shader ← 卡顿根源
       │
       ▼
  绘制

Impeller 架构：
  Dart 绘制指令
       │
       ▼
  已预编译的固定 Shader 集合 ← 编译期完成，运行时零成本
       │
       ▼
  绘制
```

**Impeller 的核心改变**：
- 用**预定义的、有限的** Shader 集合，不再按需生成
- 编译期就把所有 Shader 编译好，打包进 App
- 运行时 GPU 直接用，消除首次编译开销

**平台支持**（2026/04 时间点）：
- iOS：默认启用（3.10+ 稳定）
- Android：Flutter 3.16+ Vulkan 设备默认启用，旧设备 fallback 到 Skia
- macOS：支持
- Windows/Linux：开发中

**开启方式**：

```xml
<!-- Android: AndroidManifest.xml -->
<meta-data
  android:name="io.flutter.embedding.android.EnableImpeller"
  android:value="true" />
```

```xml
<!-- iOS: Info.plist，3.10+ 默认开启 -->
<key>FLTEnableImpeller</key>
<true/>
```

### 3.5 Impeller 的代价

- **包体积**：+几 MB（包含预编译 Shader）
- **兼容性**：老设备 Vulkan 支持不全，可能有渲染差异
- **某些效果**：Skia 专有效果可能在 Impeller 暂不支持，需要 fallback

## 四、saveLayer：最昂贵的绘制原语

### 4.1 什么是 saveLayer

```dart
canvas.saveLayer(bounds, paint);
// ... 绘制内容
canvas.restore();
```

作用：把一段绘制的结果**单独画到一张离屏位图**，然后整体合成到主画布。

```
不 saveLayer：
  canvas → 直接往主帧缓冲区画 → 快

saveLayer：
  canvas → 创建一张临时位图（GPU 内存分配！）
        → 把后续绘制都画到这张位图上
        → restore 时把临时位图合成回主画布
        → 极慢（涉及 GPU 内存分配、帧缓冲切换、合成开销）
```

### 4.2 什么场景会触发 saveLayer

```
显式触发：
  Opacity(opacity: 0.5) 包住的子树            ← 著名坑
  ClipRRect + 复杂 child                     ← 有抗锯齿 clip
  ColorFilter / ImageFilter                  ← 滤镜
  BackdropFilter                             ← 背景模糊

隐式触发：
  Material 的 InkWell 涟漪
  Hero 动画某些场景
  圆角带阴影的复杂 Container
```

### 4.3 Opacity 的替代方案

```dart
// ❌ 触发 saveLayer，昂贵
Opacity(
  opacity: 0.5,
  child: ComplexChild(),
)

// ✅ 单张图用 Image 的 opacity 参数，不触发 saveLayer
Image.asset('icon.png', color: Colors.white.withOpacity(0.5))

// ✅ 文字直接改 color 的 alpha
Text('hi', style: TextStyle(color: Colors.black.withOpacity(0.5)))

// ✅ 单色块直接改 color
Container(color: Colors.red.withOpacity(0.5))

// ✅ 复杂子树必须做透明度动画时，用 FadeTransition
FadeTransition(
  opacity: _anim,
  child: const ComplexChild(),
)
// FadeTransition 内部也用 saveLayer，但只在需要时触发
```

### 4.4 ClipRRect 的优化

```dart
// ❌ 带抗锯齿的圆角裁剪 → saveLayer
ClipRRect(
  borderRadius: BorderRadius.circular(16),
  child: Image.network(url),
)

// ✅ 用 DecoratedBox 的圆角 + BoxDecoration.image
DecoratedBox(
  decoration: BoxDecoration(
    borderRadius: BorderRadius.circular(16),
    image: DecorationImage(
      image: NetworkImage(url),
      fit: BoxFit.cover,
    ),
  ),
)
// 无 saveLayer，直接在绘制时做圆角
```

## 五、AnimatedBuilder：正确的动画写法

### 5.1 反例：用 setState 做动画

```dart
// ❌ 每一帧都重建整棵树
class _MyState extends State<MyWidget> with SingleTickerProviderStateMixin {
  late final _anim = AnimationController(vsync: this, ...)
    ..addListener(() => setState(() {}))   // 每帧 setState
    ..forward();

  @override
  Widget build(BuildContext context) {
    return Scaffold(
      body: Column(children: [
        const _StaticHeader(),                    // ❌ 跟着重建
        Transform.translate(
          offset: Offset(_anim.value * 100, 0),
          child: const _Box(),
        ),
        const _StaticFooter(),                    // ❌ 跟着重建
      ]),
    );
  }
}
```

### 5.2 正解：AnimatedBuilder

```dart
// ✅ 只有 Transform.translate 那一小块重建
class _MyState extends State<MyWidget> with SingleTickerProviderStateMixin {
  late final _anim = AnimationController(vsync: this, ...)..forward();

  @override
  Widget build(BuildContext context) {
    return Scaffold(
      body: Column(children: [
        const _StaticHeader(),
        AnimatedBuilder(
          animation: _anim,
          builder: (_, child) => Transform.translate(
            offset: Offset(_anim.value * 100, 0),
            child: child,
          ),
          child: const _Box(),                    // ✅ 传 child，不重建
        ),
        const _StaticFooter(),
      ]),
    );
  }
}
```

**两个优化点**：
1. 只有 `AnimatedBuilder.builder` 返回的 Widget 会重建，外层不动
2. `child` 参数传入，避免在每帧内重建 `_Box`

### 5.3 已封装的 Animated*

能用就用现成的：

| 功能 | Widget |
|---|---|
| 位移 | `SlideTransition` |
| 淡入淡出 | `FadeTransition` |
| 旋转 | `RotationTransition` |
| 缩放 | `ScaleTransition` |
| 尺寸 | `SizeTransition` |
| 复合 | `AnimatedBuilder` + 自定义 |
| 简单状态切换 | `AnimatedContainer` / `AnimatedOpacity` / `AnimatedAlign` |
| 列表增删 | `AnimatedList` |

### 5.4 隐式动画 vs 显式动画

```
隐式动画（Implicit）：
  AnimatedContainer(duration: ..., color: _color)
  改一下 _color，自己算插值
  简单、一行代码
  适合状态切换（颜色变、尺寸变）

显式动画（Explicit）：
  AnimationController + Transition
  完全控制、可暂停、可反向、可组合
  适合复杂/交互式动画
```

## 六、Hero 动画性能优化

### 6.1 Hero 原理

```
页面 A                    页面 B
┌──────┐                 ┌──────┐
│      │                 │ ┌──┐ │
│ Hero │ ─── 过渡 ───►   │ │  │ │
│ tag  │                 │ └──┘ │
└──────┘                 └──────┘

过渡过程：
  ① 把 A 的 Hero 和 B 的 Hero 都临时挂到 Overlay
  ② Navigator 路由动画中，Hero 位置做 Tween
  ③ 动画结束，Overlay 里的移除，B 页面正常显示
```

### 6.2 性能坑

- Hero 的 `child` 是复杂 Widget → 过渡时重建开销大
- `flightShuttleBuilder` 没写 → 用默认，复杂子树整个拖走
- 两个 Hero 的比例差别大 → 过渡时缩放渲染压力大

### 6.3 优化手法

```dart
Hero(
  tag: 'avatar-$userId',
  flightShuttleBuilder: (_, anim, dir, from, to) {
    // 过渡期间用轻量替身
    return Material(
      color: Colors.transparent,
      child: CircleAvatar(backgroundImage: NetworkImage(url)),
    );
  },
  child: CircleAvatar(backgroundImage: NetworkImage(url)),
)
```

## 七、滚动动画性能

### 7.1 滚动时避免复杂重绘

```dart
// ❌ 列表里每个 item 都有复杂阴影 + 模糊
ListView.builder(
  itemBuilder: (_, i) => Container(
    decoration: BoxDecoration(
      boxShadow: [BoxShadow(blurRadius: 20, color: Colors.black26)],  // 模糊阴影
    ),
    child: BackdropFilter(                                             // 背景模糊
      filter: ImageFilter.blur(sigmaX: 10, sigmaY: 10),
      child: ItemCard(),
    ),
  ),
)
```

滑动时每个可见 item 都 saveLayer + blur，Raster Thread 崩溃。

### 7.2 替代方案

- 静态阴影：用预渲染的 9-patch 图替代 `boxShadow`
- 背景模糊：视觉上用半透明色 + 低分辨率背景图代替 `BackdropFilter`
- 真要做毛玻璃效果：限制模糊区域为小面积

## 八、RepaintBoundary 在动画中的作用

```dart
class AnimatedClock extends StatefulWidget { ... }

// ❌ 没 boundary，时钟指针动，整个页面都 repaint
class _State extends State<Parent> {
  @override
  Widget build(BuildContext context) {
    return Column(children: [
      const HeavyHeader(),
      AnimatedClock(),              // 这货每秒都 repaint
      const HeavyFooter(),
    ]);
  }
}

// ✅ 加 boundary 隔离
Column(children: [
  const HeavyHeader(),
  const RepaintBoundary(child: AnimatedClock()),
  const HeavyFooter(),
])
```

DevTools 的 Performance Overlay 可以直观看出 Layer 是否在频繁 repaint（红色越深越频繁）。

## 九、Raster Cache：Skia 的静态 Layer 缓存

```
Skia/Impeller 的 Raster Cache 策略：
  连续 2 帧没变的 Layer
    ↓
  标记为 "稳定"
    ↓
  栅格化结果缓存到 GPU 纹理
    ↓
  下次直接用缓存纹理，跳过 Dart 层绘制
```

**触发条件**：
- 必须是 RepaintBoundary（有独立 Layer）
- 必须连续不变
- 尺寸不能太大（有上限，否则不缓存）

**怎么看缓存效果**：
```dart
// 开 Performance Overlay
MaterialApp(
  showPerformanceOverlay: true,
  ...
)
```
绿色块表示有缓存命中。

## 十、Canvas 自绘性能要点

```dart
class _CustomPaint extends CustomPainter {
  @override
  void paint(Canvas canvas, Size size) {
    // ❌ 每次 paint new Path
    final path = Path()..moveTo(0, 0)..lineTo(size.width, size.height);
    
    // ❌ 每次 paint new Paint
    final paint = Paint()..color = Colors.red;
    
    canvas.drawPath(path, paint);
  }
  
  @override
  bool shouldRepaint(_) => true;    // ❌ 总是 repaint
}
```

### 10.1 正确做法

```dart
class _CustomPaint extends CustomPainter {
  final double progress;
  _CustomPaint(this.progress);
  
  // 静态对象提前创建
  static final _paint = Paint()..color = Colors.red..strokeWidth = 2;
  static final _path = Path();
  
  @override
  void paint(Canvas canvas, Size size) {
    _path.reset();
    _path.moveTo(0, size.height);
    _path.lineTo(size.width * progress, 0);
    canvas.drawPath(_path, _paint);
  }
  
  @override
  bool shouldRepaint(_CustomPaint old) => old.progress != progress;  // ✅ 只有 progress 变才重绘
}
```

### 10.2 大量 drawXxx 调用

```dart
// ❌ 逐个画 1000 个圆
for (var i = 0; i < 1000; i++) {
  canvas.drawCircle(points[i], r, paint);
}

// ✅ 合并成一个 drawPoints 或 drawVertices
canvas.drawPoints(PointMode.points, points, paint);
```

减少 Draw Call 数量，让 GPU 一次处理一批。

## 十一、120Hz 高刷设备的特殊考虑

```
60Hz：   每帧 16.6ms 预算
120Hz：  每帧 8.3ms 预算  ← 一半时间，同样的工作

问题：60fps 流畅的代码，120Hz 可能掉帧
```

iOS 的 ProMotion 屏幕会自动降频（静态时 24Hz，滚动时 120Hz），但 App 要主动声明支持：

```xml
<!-- Info.plist -->
<key>CADisableMinimumFrameDurationOnPhone</key>
<true/>
```

Android 12+ 对高刷支持更好，但同样需要 App 能在 8.3ms 内完成帧绘制。

## 十二、面试题精选

### Q1：Flutter 的渲染管线有哪些阶段？

**答**：7 个阶段，前 5 个在 UI Thread，后 2 个在 Raster Thread。
1. Animate（动画值推进）
2. Build（调用 Widget 的 build）
3. Layout（RenderObject 布局）
4. Paint（生成 Layer Tree）
5. Composite（Scene 构建）
6. Rasterize（Skia/Impeller 栅格化）
7. GPU 提交

任何一个阶段超出帧预算（60Hz 是 16.6ms，分给 UI 和 Raster 各 8ms 左右）就会掉帧。

### Q2：UI Thread 和 Raster Thread 的区别？

**答**：
- **UI Thread**：Dart 代码运行的地方，执行 build/layout/paint，产出 Layer Tree
- **Raster Thread**：C++ 代码运行的地方，消费 Layer Tree，调用 Skia/Impeller 栅格化并提交 GPU

两条线程并行：UI Thread 画第 N 帧时，Raster Thread 在栅格化第 N-1 帧。卡顿要判断卡在哪条线程上，修复方向不同。

### Q3：什么是 Shader Jank？怎么解决？

**答**：Skia 使用 OpenGL/Metal 时，每种新的绘制组合要首次生成并编译 Shader 程序，耗时 10-100ms，导致"首次动画、首次按按钮"卡顿。

两种解决方案：
1. **SkSL 预热**（旧）：用代表性操作录制 Shader，打包时一起带上，启动时预编译
2. **Impeller**（新）：用固定、有限的预编译 Shader 集合，彻底消除运行时 Shader 编译。iOS 默认开启，Android Vulkan 设备默认开启（3.16+）

### Q4：Impeller 相比 Skia 的核心改进？

**答**：
- Skia：运行时按需生成 Shader 源码 → GPU 编译 → 缓存。**首次必卡**
- Impeller：编译期就确定所有需要的 Shader，打包进 App。**零运行时编译**

代价是包体积略增，有些 Skia 专有效果需要 fallback。iOS 从 3.10 稳定默认，Android 从 3.16 逐步默认。

### Q5：`Opacity` 为什么昂贵？怎么优化？

**答**：`Opacity` 包住的子树会触发 `saveLayer`，Skia 把子树先绘制到离屏位图，再按透明度合成回主画布。saveLayer 涉及 GPU 内存分配、帧缓冲切换、合成开销，单次耗时 1-5ms。

替代方案：
- 单图：用 `Image(color: ..withOpacity)`
- 单色块/文字：直接改 color alpha
- 动画透明度：`FadeTransition`（只在需要时 saveLayer）

### Q6：什么是 `saveLayer`？什么场景会触发？

**答**：`saveLayer` 创建一张离屏位图，后续绘制都画到这张位图上，restore 时合成到主画布。触发场景：
- 显式：`Opacity`、`ClipRRect` + 复杂 child、`ColorFilter`、`BackdropFilter`
- 隐式：`InkWell` 涟漪、Hero 动画、复杂圆角 + 阴影的 Container

是 Flutter 绘制性能的头号杀手。DevTools 的 `debugRepaintRainbowEnabled` 和 raster 耗时可以识别。

### Q7：为什么动画推荐用 `AnimatedBuilder` 而不是 `setState`？

**答**：`setState` 让整个 State 的子树重建；`AnimatedBuilder` 只让 `builder` 返回的那块子树重建。复杂页面里用 setState 做动画，每一帧都重建大片 Widget，几乎必然掉帧。

配合 `child` 参数可以把静态子树排除在重建外：
```dart
AnimatedBuilder(
  animation: _anim,
  child: const ExpensiveChild(),   // 不重建
  builder: (_, child) => Transform.scale(scale: _anim.value, child: child),
)
```

### Q8：`AnimatedContainer` 和 `AnimationController + Transform` 的区别？

**答**：
- `AnimatedContainer`：隐式动画，自动算插值，一行代码搞定状态切换
- `AnimationController`：显式动画，可暂停、可反向、可链式组合，支持自定义 Curve

简单 UI 状态过渡用 AnimatedContainer，交互式/复杂动画用 AnimationController。

### Q9：Hero 动画性能优化有哪些？

**答**：
1. `flightShuttleBuilder` 用轻量替身，避免过渡期重建复杂子树
2. Hero 的 `child` 保持简单（一张图、一个头像就好）
3. 两个 Hero 的尺寸比例不要相差太大（缩放压力大）
4. 禁用不需要的 `placeholderBuilder`

### Q10：怎么判断卡顿是 UI Thread 还是 Raster Thread 问题？

**答**：DevTools Performance 面板
- 每一帧的条形图分两段：UI 蓝色，Raster 橙色
- 哪段超过 8ms 就是哪边的问题
- UI 超时 → Track Widget Rebuilds、检查 build 方法里的耗时
- Raster 超时 → 检查 saveLayer、Shader、图片解码

快速方法：手机上开 Performance Overlay（两条曲线，上面是 Raster，下面是 UI），超过红线就是超时。

### Q11：`RepaintBoundary` 在动画场景的作用？

**答**：让动画区域成为独立 Layer，动画变化只导致该 Layer repaint。没有 boundary 时，整个父 Layer 都要 repaint。

辅助作用：静态子树能被 Raster Cache 命中，直接复用 GPU 纹理。典型场景：时钟、加载转圈、带动画的卡片。

### Q12：`CustomPainter.shouldRepaint` 什么时候返回 true？

**答**：只有当影响绘制结果的参数变了才返回 true。错误写法 `return true` 会导致每次父 widget 重建都重绘。正确做法：

```dart
@override
bool shouldRepaint(_MyPainter old) =>
    old.color != color || old.progress != progress;
```

### Q13：为什么 120Hz 设备上 App 容易掉帧？

**答**：预算从 16.6ms 缩到 8.3ms，原本 60fps 流畅的代码可能超预算。优化方向：
1. 减少 build 里的昂贵操作
2. `const` + `RepaintBoundary` 缩小重建范围
3. 启用 Impeller 消除 Shader Jank
4. iOS 要在 Info.plist 开 `CADisableMinimumFrameDurationOnPhone`

### Q14：`BackdropFilter` 的毛玻璃效果为什么特别卡？

**答**：`BackdropFilter` 需要：
1. 把当前主画布的内容 copy 到离屏纹理
2. 对该纹理做高斯模糊（昂贵）
3. 合成回主画布

每帧都要做，列表里用就是灾难。优化：
- 限制模糊区域尺寸
- 用低分辨率的预模糊图 + 半透明色替代
- 静态毛玻璃用截图 + 预处理

### Q15：`flutter run --profile --trace-startup` 能干什么？

**答**：Profile 模式启动并追踪启动性能，生成 `start_up_info.json`，包含：
- engineEnterTimestampMicros
- timeToFirstFrameMicros
- timeToFirstFrameRasterizedMicros

用来量化启动优化的收益。注意 Debug 模式数据没参考价值，必须 Profile 或 Release。

## 十三、动画与渲染优化 checklist

```
动画：
  ✅ 复杂动画用 AnimatedBuilder，传 child 参数
  ✅ 简单状态切换用 AnimatedContainer 系列
  ✅ AnimationController 必须 dispose
  ✅ 动画区域加 RepaintBoundary

绘制：
  ✅ 避免 Opacity 包复杂子树 → 改用 FadeTransition / color alpha
  ✅ 避免 ClipRRect 包大图 → 改用 DecoratedBox + borderRadius
  ✅ 避免 BackdropFilter 在列表中 → 限制范围或预处理
  ✅ Canvas 自绘：Paint/Path 复用，shouldRepaint 精确判断

Shader：
  ✅ iOS 默认启用 Impeller
  ✅ Android：3.16+ 开 Impeller，旧版本做 SkSL 预热
  ✅ 避免动态生成复杂绘制组合

高刷：
  ✅ iOS 开 CADisableMinimumFrameDurationOnPhone
  ✅ 帧预算按最差情况（8.3ms）设计

排查：
  ✅ Performance Overlay 看两条曲线
  ✅ DevTools Timeline 看每帧分段
  ✅ debugRepaintRainbowEnabled 看 repaint 范围
```

**一句话总结**：**UI Thread 看 build，Raster Thread 看 saveLayer/Shader，动画用 AnimatedBuilder 或 Transition**。
