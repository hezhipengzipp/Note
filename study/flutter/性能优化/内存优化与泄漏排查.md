# 内存优化与泄漏排查——Dart GC 原理 + 泄漏清单

> **面试核心**：Dart 有 GC，但依然会泄漏。Controller 不 dispose、Stream 不 cancel、全局 map 持引用……本文先讲 Dart GC 怎么工作，再列出 Flutter 最常见的泄漏源 + 排查手段。

---

## 一、Dart 的 GC：分代 + 并发

### 1.1 两代堆

```
          ┌─────────────────────────────────────┐
          │                Dart Heap            │
          │                                     │
          │  ┌───────────────────────────────┐  │
          │  │   New Space（新生代）           │  │
          │  │   - 新创建的对象先进这里         │  │
          │  │   - 用 Scavenge（复制算法）     │  │
          │  │   - 快，频繁                    │  │
          │  └───────────────────────────────┘  │
          │                ↓ 存活多轮            │
          │                晋升到老年代           │
          │  ┌───────────────────────────────┐  │
          │  │   Old Space（老年代）           │  │
          │  │   - 长期存活对象                 │  │
          │  │   - 用 Mark-Sweep-Compact      │  │
          │  │   - 慢，偶发                    │  │
          │  └───────────────────────────────┘  │
          └─────────────────────────────────────┘
```

**为什么分代**：业界经验——"**大多数对象朝生暮死**"。新生代 GC 只扫很小一片区域，成本低。

### 1.2 Scavenge（新生代 GC）

```
  From Space                To Space
  ┌──────────┐             ┌──────────┐
  │  对象 A   │            │          │
  │  对象 B   │   ─────►   │  对象 A  │   扫描根引用，把"活"对象
  │  对象 C   │            │  对象 C  │   复制到 To Space
  │  对象 D   │            │          │   
  └──────────┘             └──────────┘   B、D 是垃圾，不复制
       │                        │
       └────────交换 ◄──────────┘
  清空 From Space（全部标为可用）
```

- 耗时：通常 < 1ms
- 触发：New Space 满（默认 2MB-8MB）
- UI 影响：基本无感

### 1.3 Mark-Sweep-Compact（老年代 GC）

```
Mark 阶段：从 root 出发，染色所有可达对象
      ↓
Sweep 阶段：把没染色的对象清掉，留下碎片
      ↓
Compact 阶段：把存活对象压缩到一端，消除碎片
```

- 耗时：通常 5-50ms
- 触发：老年代用量超过阈值
- UI 影响：**可能掉 1-3 帧**，这是面试爱问的点

### 1.4 并发 GC

```
主线程（UI）：   ──────[build][layout][paint]─────────────────
                          │
                          │ GC 触发
                          ▼
GC 线程（后台）: ──[Mark 并发]──[Sweep 并发]──[Compact STW]──
                                                    ↑
                                         只有这一小段需要暂停主线程
```

Dart VM 把 Mark/Sweep 阶段放到后台线程并发进行，只有 Compact 需要短暂的 Stop-The-World（STW）。

### 1.5 面试常问："Dart GC 会卡 UI 吗？"

**答**：
- 新生代 Scavenge 基本不卡
- 老年代 Mark-Sweep-Compact 的 **Compact 阶段**是 STW，如果老年代很大（比如 200MB）可能卡 10-50ms
- 对策：减少对象分配、避免短时间内产生大量中等寿命对象（容易晋升到老年代又很快死）

## 二、Flutter 内存泄漏的 10 种常见场景

### 2.1 Controller 没 dispose

```dart
// ❌ 泄漏
class _MyPageState extends State<MyPage> {
  final _controller = TextEditingController();    // 没 dispose
  // State 销毁后，_controller 仍被 mounted listener 链持有
}

// ✅ 正确
class _MyPageState extends State<MyPage> {
  late final _controller = TextEditingController();
  
  @override
  void dispose() {
    _controller.dispose();
    super.dispose();
  }
}
```

**泄漏源**：`TextEditingController`、`ScrollController`、`AnimationController`、`PageController`、`TabController`、`FocusNode`……

### 2.2 AnimationController 最容易爆

```dart
class _MyPageState extends State<MyPage> with SingleTickerProviderStateMixin {
  late final _anim = AnimationController(
    vsync: this,
    duration: const Duration(seconds: 1),
  );
  
  @override
  void dispose() {
    _anim.dispose();     // 必须！否则 Ticker 泄漏
    super.dispose();
  }
}
```

`Ticker` 注册到 `SchedulerBinding`，不 dispose 会：
- 每帧继续触发 tick 回调
- 防止整个 State 被 GC
- 抛 `Ticker was activated but never disposed` 异常

### 2.3 StreamSubscription 不 cancel

```dart
class _MyPageState extends State<MyPage> {
  StreamSubscription? _sub;
  
  @override
  void initState() {
    super.initState();
    _sub = eventBus.stream.listen((event) {
      if (mounted) setState(() {/*...*/});
    });
  }
  
  @override
  void dispose() {
    _sub?.cancel();     // ✅ 取消订阅
    super.dispose();
  }
}
```

### 2.4 定时器 Timer 不 cancel

```dart
class _MyPageState extends State<MyPage> {
  Timer? _timer;
  
  @override
  void initState() {
    super.initState();
    _timer = Timer.periodic(const Duration(seconds: 1), (_) {
      if (mounted) setState(() {/*...*/});
    });
  }
  
  @override
  void dispose() {
    _timer?.cancel();
    super.dispose();
  }
}
```

### 2.5 全局单例持有 Context / State

```dart
// ❌ 典型泄漏
class EventBus {
  static final instance = EventBus._();
  final List<BuildContext> _contexts = [];
  
  void register(BuildContext context) {
    _contexts.add(context);    // 页面销毁了，context 还在 list 里
  }
}
```

**修复**：用弱引用或 id 映射，而不是直接持有 Context。

### 2.6 匿名回调捕获 this

```dart
class _MyPageState extends State<MyPage> {
  @override
  void initState() {
    super.initState();
    GlobalSingleton.instance.onSomeEvent = () {
      setState(() {/*...*/});    // 闭包持有了 this（State 对象）
    };
  }
  
  @override
  void dispose() {
    GlobalSingleton.instance.onSomeEvent = null;    // 必须解绑
    super.dispose();
  }
}
```

### 2.7 WidgetsBindingObserver 不 remove

```dart
class _MyPageState extends State<MyPage> with WidgetsBindingObserver {
  @override
  void initState() {
    super.initState();
    WidgetsBinding.instance.addObserver(this);
  }
  
  @override
  void dispose() {
    WidgetsBinding.instance.removeObserver(this);    // 必须！
    super.dispose();
  }
}
```

### 2.8 ChangeNotifier 的 listener 不 remove

```dart
class _MyPageState extends State<MyPage> {
  @override
  void initState() {
    super.initState();
    userModel.addListener(_onUserChanged);
  }
  
  @override
  void dispose() {
    userModel.removeListener(_onUserChanged);   // 必须
    super.dispose();
  }
  
  void _onUserChanged() {/*...*/}
}
```

### 2.9 ImageCache 爆

不是"泄漏"但经常被误认为是。默认 100MB，加载大图列表时很容易撑满。

```dart
// 监听内存告警清掉
class _AppState extends State<App> with WidgetsBindingObserver {
  @override
  void didHaveMemoryPressure() {
    PaintingBinding.instance.imageCache.clear();
  }
}
```

### 2.10 Isolate 不 kill

```dart
// ❌ Isolate 忘了 kill，后台一直跑
final isolate = await Isolate.spawn(workFunction, sendPort);

// ✅ 用完了 kill
isolate.kill(priority: Isolate.immediate);
```

## 三、关键规则：State 的生命周期 hook 要对齐

```
initState()   → 订阅、创建 Controller、注册 observer
    │
    ▼
didChangeDependencies()  → 订阅依赖 InheritedWidget 的资源
    │
    ▼
build()                 → 绝不做副作用
    │
    ▼
didUpdateWidget(old)    → 旧 widget 的资源释放、新的创建
    │
    ▼
deactivate()            → 从树上摘下（可能还会重新挂）
    │
    ▼
dispose()               → 释放所有资源
```

**口诀**：**有 add 就有 remove，有 listen 就有 cancel，有 Controller 就有 dispose**。

## 四、LeakTracker：Flutter 官方泄漏检测

```dart
// Flutter 3.14+
import 'package:leak_tracker_flutter_testing/leak_tracker_flutter_testing.dart';

void main() {
  LeakTesting.enable();
  testWidgets('no leaks', (tester) async {
    await tester.pumpWidget(const MyPage());
    await tester.tap(find.byType(BackButton));
    await tester.pumpAndSettle();
    // 如果 MyPage 里有 Controller 没 dispose，测试会失败
  });
}
```

### 4.1 LeakTracker 原理

```
每个 dispose() 里调用：
   _FinalizableDisposable.disposalValidation(this);
        ↓
   注册到 WeakReference + Finalizer
        ↓
   GC 时检查：
     如果对象仍然活着但已经 dispose → 过度 dispose
     如果对象没被 dispose 又被 GC → 泄漏（它应该被 dispose）
```

## 五、DevTools Memory 面板实战

### 5.1 看什么

```
DevTools → Memory 面板
  ├─ Memory 图表（实时曲线）
  │   - Dart Heap
  │   - Raster Cache
  │   - Images (GPU)
  │   - Native
  │
  ├─ Diff Snapshot（差异对比）
  │   - 进入页面前 snapshot
  │   - 进入页面后 snapshot
  │   - 退出页面后 snapshot
  │   - 对比：什么对象没被回收？
  │
  └─ Allocation Profile（分配剖析）
      - 哪些类创建最多
      - 哪些类总占用最大
```

### 5.2 泄漏排查标准流程

```
① 录基准：进入 App，在首页 → Take Snapshot

② 反复进入/退出可疑页面 5 次
   - 进入 A 页 → Take Snapshot
   - 退出 A 页 → 点 "GC" → Take Snapshot
   - 重复 5 次

③ 对比 Snapshot
   - 如果每次退出后 A 相关对象数量都 +1，说明泄漏
   - Diff 功能可以直接看新增对象

④ 点击嫌疑对象 → "inbound references"
   - 顺着引用链找谁还持着它
   - 通常指向某个全局、某个 listener、某个 Controller
```

### 5.3 看内存构成

```
某次快照：
  Dart Heap:  120 MB
  Images:     250 MB   ← 图片缓存
  Raster:     40 MB    ← Skia 栅格化缓存
  Total:      410 MB
```

如果 Images 异常大 → 图片优化问题（cacheWidth、cache 数量）
如果 Dart Heap 异常大 → 对象泄漏
如果 Raster 异常大 → RepaintBoundary 用太多

## 六、Isolate 的内存管理

```
main Isolate（UI）                 worker Isolate
  ┌──────────┐                     ┌──────────┐
  │  Heap 1  │     消息传递         │  Heap 2  │
  │  (独立)   │  ←──SendPort──►    │  (独立)   │
  └──────────┘                     └──────────┘
```

**Dart Isolate 特性**：
- 堆独立，不共享内存（不像 Java 线程）
- 通过消息传递，消息是 deep copy（对大对象有成本）
- kill 后整个 Heap 释放

**使用要点**：
1. 用完必 kill，否则整个 Heap 驻留
2. 用 `Isolate.run()`（3.3+）代替手动 spawn，自动释放
3. 传大对象用 `TransferableTypedData`（零拷贝）

## 八、优化内存占用的手段

### 8.1 对象池

```dart
class _PathPool {
  static final _pool = <Path>[];
  
  static Path acquire() {
    if (_pool.isNotEmpty) return _pool.removeLast()..reset();
    return Path();
  }
  
  static void release(Path p) {
    if (_pool.length < 20) _pool.add(p);
  }
}
```

适合自定义 CustomPainter 里反复创建 Path/Paint 的场景。Flutter Framework 自身已经池化了很多对象，业务层极少需要。

### 8.2 懒加载大对象

```dart
class Config {
  static late final String _bigJson = _loadBigJson();    // 第一次访问才加载
  
  static String get bigJson => _bigJson;
}
```

### 8.3 用 Iterable 代替 List

```dart
// ❌ 创建整个 List
final doubled = list.map((e) => e * 2).toList();

// ✅ 按需计算，不产生中间 List
final doubled = list.map((e) => e * 2);
```

仅当只遍历一次时 Iterable 优于 List。

### 8.4 字符串拼接用 StringBuffer

```dart
// ❌ n 次拼接 = n 个新字符串
var result = '';
for (var s in list) result += s;

// ✅ 一个 buffer，最后 toString 一次
final sb = StringBuffer();
for (var s in list) sb.write(s);
final result = sb.toString();
```

## 九、面试题精选

### Q1：Dart 有 GC，为什么还会内存泄漏？

**答**：GC 只能回收"**不可达**"的对象。泄漏的本质是"**本该不可达但仍可达**"，比如：
- 全局单例持有 Context
- Controller 没 dispose，其内部的 listener 持有 State
- Stream 没 cancel，listener 持有 setState 闭包

GC 解决不了"程序错误地持有引用"。

### Q2：Dart 的 GC 会导致 UI 卡顿吗？

**答**：
- 新生代 Scavenge（< 1ms）：基本无感
- 老年代 Mark-Sweep：Mark 和 Sweep 并发执行，不卡
- 老年代 Compact（STW）：可能卡 5-50ms，老年代越大越明显

优化方向：减少对象分配、避免大量中等寿命对象（短寿命对象在新生代就被回收，长寿命对象晋升一次就稳定）。

### Q3：`AnimationController` 不 dispose 会怎样？

**答**：
1. `Ticker` 留在 `SchedulerBinding` 里，每帧继续触发
2. State 对象无法被 GC，形成泄漏
3. Flutter 在 Debug 模式下会抛 `Ticker was activated but never disposed` 异常

必须在 `dispose()` 里调 `animationController.dispose()`。

### Q4：`StatefulWidget` 的 `dispose` 和 `deactivate` 有什么区别？

**答**：
- `deactivate()`：从树上摘下时调用，可能只是暂时的（比如 Reparent）
- `dispose()`：永久销毁时调用，在 deactivate 之后

`dispose()` 是释放资源的正确位置。`deactivate` 里只做"暂时解绑"。

### Q5：怎么排查一个页面的内存泄漏？

**答**：DevTools Memory 面板流程
1. 首页 Take Snapshot（基准）
2. 进入可疑页面 → Take Snapshot
3. 退出页面，手动 GC → Take Snapshot
4. 用 Diff 功能对比，看有哪些对象没回收
5. 点可疑对象看 "inbound references"，顺着找谁持有它
6. 定位到代码后修复（加 dispose / removeListener / cancel）

### Q6：Isolate 的内存管理和普通线程有什么区别？

**答**：Isolate 每个都有独立 Heap，不共享内存（Java/Kotlin 线程共享堆）。好处是无锁、无数据竞争，代价是消息传递需要 deep copy（大对象拷贝成本高）。Isolate kill 后整个 Heap 释放，不需要手动清理内部对象。

### Q7：`ImageCache` 和"图片内存泄漏"是一回事吗？

**答**：不是。ImageCache 是**有上限的 LRU**，不算泄漏。但默认 100MB 对大图 app 可能撑满，看起来像"泄漏"实际是缓存策略问题。优化：调整 `maximumSizeBytes`、监听 `didHaveMemoryPressure` 清缓存。

### Q8：一个 ListView 里每个 item 都订阅了 EventBus，滑出视口后会自动取消订阅吗？

**答**：**不会**。`StatelessWidget` 没有 dispose 概念。应该用 `StatefulWidget`，在 `initState` 里订阅，在 `dispose` 里取消。或者更安全的做法是用 `StreamBuilder`（自动管理订阅生命周期）。

### Q9：闭包捕获 `this` 导致泄漏怎么办？

**答**：两个思路
1. **解除注册**：在 dispose 里把全局回调设为 null / 从 list 里 remove
2. **弱引用**：用 `WeakReference`（Dart 2.18+）包装，让 GC 可以回收

```dart
final ref = WeakReference(this);
Timer.periodic(Duration(seconds: 1), (t) {
  final state = ref.target;
  if (state == null) { t.cancel(); return; }
  state.setState(...);
});
```

### Q10：LeakTracker 怎么用？

**答**：
```dart
import 'package:leak_tracker_flutter_testing/leak_tracker_flutter_testing.dart';
void main() {
  LeakTesting.enable();
  // ...
}
```
在测试里开启后，任何 `Disposable` 对象（Controller、AnimationController 等）如果没 dispose 就被 GC，测试会失败。生产代码不开销，但所有 UI 测试都应该打开。

### Q11：`StreamBuilder` 和 `FutureBuilder` 会泄漏吗？

**答**：通常不会。Framework 在 State 的 dispose 里会自动 cancel 订阅。但要注意：
- 传入的 Stream 是 `broadcastStream`，Framework 订阅不会关闭 stream 本身，那要看 stream 的产生方
- 传入的 Future 如果内部持有异步任务，Future 完成时 Widget 已经销毁了，setState 可能报错

解决：Future 里的 setState 前加 `if (mounted)`。

### Q12：什么是"过度 dispose"（double dispose）？

**答**：同一个 Disposable 对象被 dispose 两次。常见原因：
- 父子 Widget 都持有同一个 Controller，都各自 dispose
- 异步任务回来后再次调用 dispose

LeakTracker 会检测这种情况。解决：谁创建谁 dispose，明确所有权。

## 十、内存优化 checklist

```
防御性编程：
  ✅ 所有 Controller 在 dispose 里释放
  ✅ 所有 StreamSubscription 在 dispose 里 cancel
  ✅ 所有 addListener 在 dispose 里 removeListener
  ✅ 所有 addObserver 在 dispose 里 removeObserver
  ✅ Timer 在 dispose 里 cancel
  ✅ 异步回调内 setState 前加 if (mounted)

检测：
  ✅ 单元测试启用 LeakTracker
  ✅ 关键页面用 DevTools Memory 做进/退对比
  ✅ CI 里跑 `flutter test` 有泄漏会报错

线上监控：
  ✅ 监听 didHaveMemoryPressure 清缓存
  ✅ 定时上报内存占用（可配合 APM 工具）
  ✅ OOM 率统计

图片相关：
  ✅ cacheWidth/cacheHeight 按需解码
  ✅ ImageCache 大小按 App 特性调整
  ✅ 低端机降级策略
```

**一句话总结**：**Dart GC 解决不了"错误持有"，泄漏靠代码规范 + LeakTracker + DevTools 三位一体**。
