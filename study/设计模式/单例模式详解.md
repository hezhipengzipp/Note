# 单例模式详解

## 核心思想

```
保证一个类在整个程序生命周期中只有一个实例
并提供全局访问点

适用场景：
  线程池、连接池、日志管理器、配置管理、缓存
```

---

## 一、饿汉式

```java
// 类加载时立即创建，线程安全，但可能浪费内存
public class Singleton {

    // 类加载时就创建实例
    private static final Singleton INSTANCE = new Singleton();

    private Singleton() { }   // 私有构造器，禁止外部 new

    public static Singleton getInstance() {
        return INSTANCE;
    }
}
```

```
优点：✅ 简单，线程安全（类加载由 JVM 保证）
缺点：❌ 类加载就创建，不用也占内存
```

---

## 二、懒汉式（线程不安全）

```java
// 第一次使用时才创建，但多线程下有问题
public class Singleton {

    private static Singleton instance;

    private Singleton() { }

    public static Singleton getInstance() {
        if (instance == null) {          // 多线程同时到这里 → 创建多个实例
            instance = new Singleton();
        }
        return instance;
    }
}
```

```
线程 A：if(instance == null) → true
线程 B：if(instance == null) → true（A 还没 new 完）
线程 A：instance = new Singleton()
线程 B：instance = new Singleton()   ← 又 new 了一个！❌
```

---

## 三、同步方法（线程安全，性能差）

```java
public class Singleton {

    private static Singleton instance;

    private Singleton() { }

    // 整个方法加锁，每次 getInstance() 都要竞争锁
    public static synchronized Singleton getInstance() {
        if (instance == null) {
            instance = new Singleton();
        }
        return instance;
    }
}
```

```
优点：✅ 线程安全
缺点：❌ 每次调用都加锁，99% 的情况实例已存在，锁完全是浪费
```

---

## 四、双重检查锁 DCL（推荐）

```java
public class Singleton {

    // volatile 关键，禁止指令重排序
    private static volatile Singleton instance;

    private Singleton() { }

    public static Singleton getInstance() {
        if (instance == null) {                    // 第一次检查（无锁）
            synchronized (Singleton.class) {
                if (instance == null) {            // 第二次检查（有锁）
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

### 为什么需要 volatile

```
instance = new Singleton() 不是原子操作，分三步：

① 分配内存空间
② 初始化对象
③ 将 instance 指向内存地址

JVM 可能重排序为 ① → ③ → ②

线程 A：执行 ①③（instance 不为 null，但对象未初始化）
线程 B：第一次检查 instance != null → 直接返回
线程 B：拿到的是未初始化的对象 → NPE ❌

volatile 禁止重排序，保证 ①②③ 顺序执行
```

### 为什么需要两次 if 检查

```
第一次检查（无锁）：
  实例已存在时直接返回，避免每次都加锁，提高性能

第二次检查（有锁内）：
  防止两个线程同时通过第一次检查后重复创建
  线程A 和 线程B 都通过第一层 if
  线程A 获得锁，创建实例，释放锁
  线程B 获得锁，再次检查 instance != null，直接返回
```

## 五、静态内部类（推荐，最优雅）

```java
public class Singleton {

    private Singleton() { }

    // 静态内部类：不会随外部类加载而加载
    private static class Holder {
        private static final Singleton INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return Holder.INSTANCE;    // 第一次调用时才加载 Holder 类
    }
}
```

```
原理：利用 JVM 类加载机制保证线程安全

外部类 Singleton 加载时，Holder 不会加载
第一次调用 getInstance() → 触发 Holder 类加载
类加载由 JVM 保证线程安全（ClassLoader 加锁）
INSTANCE 只会被创建一次

优点：
  ✅ 懒加载（用时才创建）
  ✅ 线程安全（JVM 保证）
  ✅ 无锁，性能好
  ✅ 代码简洁
```

## 六、枚举单例（最安全）

```java
public enum Singleton {
    INSTANCE;

    public void doSomething() {
        System.out.println("业务方法");
    }
}

// 使用
Singleton.INSTANCE.doSomething();
```

```
优点：
  ✅ 天然线程安全
  ✅ 防止反射攻击（反射无法创建枚举实例）
  ✅ 防止反序列化破坏单例
  ✅ 代码最简洁

Effective Java 推荐的写法
```

## 七、单例的破坏与防御

### 反射攻击

```java
// 攻击
Constructor<Singleton> c = Singleton.class.getDeclaredConstructor();
c.setAccessible(true);
Singleton s2 = c.newInstance();   // 强行 new 第二个实例

// 防御：构造器中检查
private Singleton() {
    if (instance != null) {
        throw new RuntimeException("禁止反射创建实例");
    }
}
```

### 反序列化攻击

```java
// 攻击：序列化再反序列化会创建新对象
ObjectOutputStream oos = new ObjectOutputStream(...);
oos.writeObject(Singleton.getInstance());
ObjectInputStream ois = new ObjectInputStream(...);
Singleton s2 = (Singleton) ois.readObject();  // 新实例！

// 防御：实现 readResolve()
private Object readResolve() {
    return instance;    // 反序列化时返回已有实例
}
```

## 九、Android 中的单例

```kotlin
// Kotlin object 关键字 = 天然单例
object AppConfig {
    val baseUrl = "https://api.example.com"
    fun init(context: Context) { ... }
}
AppConfig.init(this)

// Application 级别单例
class App : Application() {
    companion object {
        lateinit var instance: App
            private set
    }
    override fun onCreate() {
        super.onCreate()
        instance = this
    }
}

// 需要 Context 的单例（静态内部类）
class DatabaseManager private constructor(context: Context) {
    private object Holder {
        lateinit var INSTANCE: DatabaseManager
    }
    companion object {
        fun getInstance(context: Context): DatabaseManager {
            if (!Holder::INSTANCE.isInitialized) {
                synchronized(this) {
                    if (!Holder::INSTANCE.isInitialized) {
                        Holder.INSTANCE = DatabaseManager(context.applicationContext)
                    }
                }
            }
            return Holder.INSTANCE
        }
    }
}
```

## 总结

```
面试答法：

简单场景（不需懒加载）   → 饿汉式 / 枚举
需要懒加载 + 高性能      → 静态内部类（首选）
需要防反射/反序列化      → 枚举
多线程高并发             → DCL（volatile 不能少）

核心考点：
  DCL 为什么需要 volatile？    → 禁止指令重排，防止拿到未初始化对象
  DCL 为什么需要两次 if？      → 外层提高性能，内层保证线程安全
  静态内部类为什么线程安全？    → JVM 类加载机制保证
  枚举为什么防反射？            → 反射调用枚举构造器会抛异常
```
