# 代理模式详解

## 一、静态代理 vs 动态代理

### 核心区别一句话

```
静态代理：代理类由程序员手动编写，编译期确定
动态代理：代理类由 JVM 在运行时动态生成，无需手动编写
```

---

## 二、静态代理

```java
// 接口
public interface UserService {
    String findById(int id);
    void save(String name);
}

// 真实实现
public class UserServiceImpl implements UserService {
    public String findById(int id) { return "user_" + id; }
    public void save(String name)  { System.out.println("save: " + name); }
}

// 静态代理类：程序员手动编写，和目标类实现同一接口
public class UserServiceProxy implements UserService {

    private final UserService target;

    public UserServiceProxy(UserService target) {
        this.target = target;
    }

    @Override
    public String findById(int id) {
        System.out.println("前置：日志");
        String result = target.findById(id);
        System.out.println("后置：日志");
        return result;
    }

    @Override
    public void save(String name) {
        System.out.println("前置：日志");
        target.save(name);
        System.out.println("后置：日志");
    }
    // ↑ 每个方法都要手动写一遍，非常繁琐
}
```

### 静态代理的问题

```
UserService    有 2 个方法 → 代理类写 2 个方法
OrderService   有 5 个方法 → 代理类写 5 个方法
ProductService 有 8 个方法 → 代理类写 8 个方法

接口新增一个方法 → 代理类必须同步新增，否则编译报错

100 个 Service → 100 个 Proxy 类，全是重复模板代码
```

## 三、动态代理

### JDK 动态代理

```java
// 同一个 Handler 代理所有接口的所有方法
public class LogHandler implements InvocationHandler {

    private final Object target;

    public LogHandler(Object target) { this.target = target; }

    @Override
    public Object invoke(Object proxy, Method method, Object[] args)
            throws Throwable {
        System.out.println("前置：" + method.getName());
        Object result = method.invoke(target, args);
        System.out.println("后置：" + method.getName());
        return result;
    }
}

// 一行代码代理任意接口
UserService userProxy = (UserService) Proxy.newProxyInstance(
    UserServiceImpl.class.getClassLoader(),
    new Class[]{UserService.class},
    new LogHandler(new UserServiceImpl())
);

// 同一套代码代理不同 Service，不需要写 OrderServiceProxy 类
OrderService orderProxy = (OrderService) Proxy.newProxyInstance(
    OrderServiceImpl.class.getClassLoader(),
    new Class[]{OrderService.class},
    new LogHandler(new OrderServiceImpl())
);
```

### JDK 代理内部原理

```
Proxy.newProxyInstance() 做了什么？

① 根据接口列表动态生成 $Proxy0.class 字节码
② 加载进 JVM
③ 实例化并返回

生成的代理类大致长这样：

public final class $Proxy0 extends Proxy implements UserService {

    private static Method m_findById;
    private static Method m_save;

    static {
        m_findById = UserService.class.getMethod("findById", int.class);
        m_save     = UserService.class.getMethod("save", String.class);
    }

    public $Proxy0(InvocationHandler h) { super(h); }

    public String findById(int id) {
        return (String) h.invoke(this, m_findById, new Object[]{id});
    }

    public void save(String name) {
        h.invoke(this, m_save, new Object[]{name});
    }
}
```

```
调用链路

proxy.findById(1)
      ↓
$Proxy0.findById(1)
      ↓
handler.invoke(proxy, method, args)
      ↓
前置逻辑
      ↓
method.invoke(target, args)   // 反射调用真实方法
      ↓
后置逻辑
      ↓
return result
```

### JDK 代理的限制

```
必须基于接口
─────────────────────────────────────
原因：生成的 $Proxy0 已经 extends Proxy
      Java 单继承，无法再继承 UserServiceImpl
      所以目标类必须有接口
```

### CGLIB 动态代理

```java
// 目标类（不需要接口）
public class OrderService {
    public String createOrder(String item) {
        return "order_" + item;
    }
}

// MethodInterceptor：对应 JDK 的 InvocationHandler
public class LogInterceptor implements MethodInterceptor {
    @Override
    public Object intercept(Object obj, Method method,
            Object[] args, MethodProxy proxy) throws Throwable {
        System.out.println("before: " + method.getName());
        Object result = proxy.invokeSuper(obj, args);  // 调用父类方法
        System.out.println("after: " + method.getName());
        return result;
    }
}

// 生成代理（继承目标类，重写方法）
Enhancer enhancer = new Enhancer();
enhancer.setSuperclass(OrderService.class);
enhancer.setCallback(new LogInterceptor());
OrderService proxy = (OrderService) enhancer.create();
```

### CGLIB 的限制

```
final 类/方法无法代理
──────────────────────────────────────
final class OrderService → 无法继承 → ❌
public final String createOrder() → 无法重写 → ❌
```

## 四、核心对比

```
静态代理                          动态代理
──────────────────────────────────────────────────────
代理类由程序员编写                代理类由 JVM 运行时生成
编译期已确定                      运行期才生成 $Proxy0.class
每个接口需要一个代理类             一个 Handler 代理所有接口
接口新增方法 → 代理类必须修改      接口新增方法 → Handler 自动拦截
代码量大，重复模板多              代码量小，逻辑集中
方法调用直接调用                  方法调用通过反射（有性能损耗）
```

| | 静态代理 | JDK 动态代理 | CGLIB |
|---|---|---|---|
| 代理类来源 | 手动编写 | JVM 生成 | JVM 生成 |
| 要求 | 实现同一接口 | 必须有接口 | 类不能是 final |
| 性能 | 最快 | 稍慢（反射）| 稍慢（创建），调用快 |
| 灵活性 | 低 | 高 | 高 |

## 五、JDK vs CGLIB 选择策略

```
                  JDK 动态代理          CGLIB
──────────────────────────────────────────────────
依赖              JDK 内置             第三方库（Spring 已内置）
代理方式          实现接口             继承目标类
生成类            $Proxy0              OrderService$$Enhancer
Spring 默认       有接口用 JDK         无接口用 CGLIB
```

```java
// Spring 强制使用 CGLIB
@EnableAspectJAutoProxy(proxyTargetClass = true)
```

## 六、装饰器模式 vs 静态代理（容易混淆）

```java
// 静态代理：增强原有功能（日志、权限），不改变接口语义
public class UserServiceProxy implements UserService {
    private UserService target;
    public String findById(int id) {
        log();                         // 增强：加日志
        return target.findById(id);    // 功能不变
    }
}

// 装饰器：扩展功能，叠加组合，改变行为本身
public class BufferedInputStream extends InputStream {
    private InputStream in;
    public int read() {
        return bufferedRead();         // 增加缓冲，改变读取行为
    }
}
```

```
区别：
  代理   → 控制访问，增强非功能性逻辑，对调用方透明
  装饰器 → 扩展功能，功能叠加组合，调用方有意选择装饰
```

## 七、实际应用场景

```
Spring AOP       → 动态代理（日志、事务、权限）
MyBatis Mapper   → JDK 动态代理（接口 → SQL 执行）
Retrofit         → JDK 动态代理（接口方法 → 网络请求）
RPC 框架         → 动态代理（接口调用 → 远程调用）
Mock 测试        → 动态代理（Mockito）
Android          → 静态代理（装饰器 Context）
```

## 八、适用场景总结

```
静态代理适合：
  ✅ 代理逻辑因方法而异（每个方法增强不同）
  ✅ 接口稳定，方法数量少
  ✅ 性能要求极高，不能用反射

动态代理适合：
  ✅ 统一横切逻辑（日志、事务、权限、监控）
  ✅ 接口多、方法多，需要批量代理
  ✅ 运行时才知道代理哪个类

  有接口   → JDK 动态代理
  无接口   → CGLIB
  Spring  → 自动选择，也可强制 CGLIB
```
