# JVM 内存模型与垃圾回收

### 各区域详解

| 区域 | 线程 | 存什么 | 异常 |
|------|------|--------|------|
| **堆** | 共享 | 对象实例、数组 | OutOfMemoryError |
| **方法区/元空间** | 共享 | 类信息、常量、静态变量 | OutOfMemoryError |
| **虚拟机栈** | 私有 | 方法调用的栈帧（局部变量、操作数） | StackOverflowError / OOM |
| **本地方法栈** | 私有 | Native 方法的栈帧 | StackOverflowError |
| **程序计数器** | 私有 | 当前线程执行的字节码行号 | 唯一不会 OOM 的区域 |

## 二、堆的分代结构

```
堆（Heap）
┌──────────────────────────────────────────────────────┐
│                                                      │
│  新生代 (Young Generation)  1/3          老年代 (Old)  2/3
│  ┌────────────┬──────┬──────┐  ┌──────────────────┐ │
│  │   Eden     │  S0  │  S1  │  │                  │ │
│  │   8/10     │ 1/10 │ 1/10 │  │  长期存活的对象    │ │
│  │            │      │      │  │  大对象           │ │
│  │  新对象在这  │ 交替  │ 使用  │  │                  │ │
│  │  里分配     │      │      │  │                  │ │
│  └────────────┴──────┴──────┘  └──────────────────┘ │
│                                                      │
└──────────────────────────────────────────────────────┘

默认比例：
  新生代 : 老年代 = 1 : 2
  Eden : S0 : S1 = 8 : 1 : 1
```

## 三、对象的一生

```
1. 对象诞生
   new Object() → 分配在 Eden 区

2. 第一次 GC（Minor GC）
   Eden 满了 → 触发 Young GC
   存活的对象 → 复制到 S0，年龄 = 1
   Eden 清空

3. 第二次 GC
   Eden + S0 中存活的 → 复制到 S1，年龄 +1
   Eden 和 S0 清空
   （S0 和 S1 交替使用，始终有一个是空的）

4. 多次 GC 后
   年龄达到阈值（默认 15）→ 晋升到老年代

5. 特殊情况
   大对象（超过阈值）→ 直接分配到老年代
   Survivor 放不下  → 直接晋升老年代

6. 老年代满了 → 触发 Full GC（Stop The World，很慢）
```

```
Eden      S0        S1        Old
┌────┐  ┌────┐  ┌────┐  ┌──────────┐
│ABCD│  │    │  │    │  │          │   初始状态：对象在 Eden
└────┘  └────┘  └────┘  └──────────┘

  ↓ Minor GC（A、C 存活）

┌────┐  ┌────┐  ┌────┐  ┌──────────┐
│    │  │A(1)│  │    │  │          │   A、C 复制到 S0，年龄=1
│ EF │  │C(1)│  │    │  │          │   Eden 清空，新对象 E、F 进来
└────┘  └────┘  └────┘  └──────────┘

  ↓ Minor GC（A、E 存活）

┌────┐  ┌────┐  ┌────┐  ┌──────────┐
│    │  │    │  │A(2)│  │          │   A 复制到 S1，年龄=2
│ GH │  │    │  │E(1)│  │          │   S0 和 S1 交换角色
└────┘  └────┘  └────┘  └──────────┘

  ↓ ... 年龄到 15

┌────┐  ┌────┐  ┌────┐  ┌──────────┐
│    │  │    │  │    │  │  A(15)   │   A 晋升到老年代
└────┘  └────┘  └────┘  └──────────┘
```

## 四、怎么判断对象可以回收

### 引用计数法（JVM 不用）

```
每个对象有一个引用计数器：
  被引用 → 计数 +1
  引用断开 → 计数 -1
  计数 = 0 → 回收

问题：循环引用无法回收
  A.ref = B
  B.ref = A
  A、B 都不再被外部使用，但计数都是 1 → 永远不会回收
```

### 可达性分析（JVM 实际使用）

```
从 GC Roots 出发，沿引用链遍历
能到达的对象 → 存活
不可达的对象 → 可回收

GC Root
  │
  ├── Object A ──→ Object B ──→ Object C     ← 全部存活
  │
  └── Object D                                ← 存活

  Object E ──→ Object F                       ← 不可达，可回收
```

### GC Roots 有哪些

```
1. 虚拟机栈中的局部变量
   void method() {
       Object obj = new Object();  // obj 是 GC Root
   }

2. 方法区中的静态变量
   static Object sObj = new Object();  // sObj 是 GC Root

3. 方法区中的常量
   static final String CONST = "hello";  // CONST 是 GC Root

4. 本地方法栈中的 JNI 引用

5. 活跃线程（Thread 对象）

6. synchronized 锁持有的对象
```

## 五、四种引用类型

```
┌─────────────────┬─────────────────────────────────────┐
│ 强引用 (Strong)  │ Object obj = new Object()            │
│                 │ 永远不会被 GC，OOM 也不回收            │
│                 │ 要回收需手动 obj = null                │
├─────────────────┼─────────────────────────────────────┤
│ 软引用 (Soft)    │ SoftReference<Object> ref = new ...  │
│                 │ 内存不足时才回收                       │
│                 │ 适合做内存缓存（如图片缓存）            │
├─────────────────┼─────────────────────────────────────┤
│ 弱引用 (Weak)    │ WeakReference<Object> ref = new ...  │
│                 │ 下次 GC 就回收，不管内存够不够          │
│                 │ LeakCanary 用来检测泄漏               │
├─────────────────┼─────────────────────────────────────┤
│ 虚引用 (Phantom) │ PhantomReference<Object> ref = ...   │
│                 │ 随时可能被回收，无法通过虚引用获取对象   │
│                 │ 配合 ReferenceQueue 跟踪回收时机       │
└─────────────────┴─────────────────────────────────────┘
```

## 六、垃圾回收算法

### 1. 标记-清除（Mark-Sweep）

```
标记阶段：从 GC Root 遍历，标记所有存活对象
清除阶段：清除所有未标记的对象

回收前：
┌──┬──┬──┬──┬──┬──┬──┬──┐
│A │  │B │  │C │  │  │D │   A、B、D 存活，其余回收
└──┴──┴──┴──┴──┴──┴──┴──┘

回收后：
┌──┬──┬──┬──┬──┬──┬──┬──┐
│A │空│B │空│C │空│空│D │   空闲空间不连续 → 内存碎片！
└──┴──┴──┴──┴──┴──┴──┴──┘

优点：简单
缺点：内存碎片，大对象可能找不到连续空间
```

### 2. 标记-整理（Mark-Compact）

```
标记阶段：同上
整理阶段：把存活对象向一端移动，清理边界外的空间

回收前：
┌──┬──┬──┬──┬──┬──┬──┬──┐
│A │  │B │  │  │  │  │D │
└──┴──┴──┴──┴──┴──┴──┴──┘

回收后（整理）：
┌──┬──┬──┬──┬──┬──┬──┬──┐
│A │B │D │              │   存活对象紧凑排列，无碎片
└──┴──┴──┴──┴──┴──┴──┴──┘

优点：无碎片
缺点：移动对象需要更新引用，耗时
适用：老年代（存活率高，不适合复制）
```

### 3. 复制算法（Copying）

```
把内存分成两半，每次只用一半
GC 时把存活对象复制到另一半，原来的一半整体清空

使用中的一半      空闲的一半
┌──┬──┬──┬──┐  ┌──┬──┬──┬──┐
│A │  │B │  │  │            │
└──┴──┴──┴──┘  └──┴──┴──┴──┘

        ↓ GC，复制存活对象

┌──┬──┬──┬──┐  ┌──┬──┬──┬──┐
│            │  │A │B │      │
└──┴──┴──┴──┘  └──┴──┴──┴──┘
  清空            现在用这半

优点：无碎片，速度快（只需移动存活对象）
缺点：内存利用率只有 50%
改进：Eden + S0 + S1 = 8:1:1，只浪费 10%
适用：新生代（大部分对象朝生夕死，存活率低，复制少）
```

### 分代收集策略

```
不同区域用不同算法，扬长避短：

新生代（对象存活率低，约 90% 会死）：
  → 复制算法（Eden + S0 + S1）
  → Minor GC，速度快，频繁触发

老年代（对象存活率高）：
  → 标记-清除 或 标记-整理
  → Major GC / Full GC，速度慢，尽量少触发
```

## 七、常见垃圾收集器

```
新生代收集器：           老年代收集器：
┌────────────────┐    ┌────────────────┐
│ Serial          │    │ Serial Old     │    ← 单线程，STW
│ ParNew          │    │ CMS            │    ← 并发标记清除
│ Parallel Scavenge│   │ Parallel Old   │    ← 多线程，吞吐量优先
└────────────────┘    └────────────────┘

整堆收集器：
┌────────────────┐
│ G1              │    ← 分 Region，可预测停顿
│ ZGC             │    ← 超低延迟（<10ms）
│ Shenandoah      │    ← 类似 ZGC
└────────────────┘
```

### CMS（Concurrent Mark Sweep）

```
目标：最短停顿时间

1. 初始标记 (STW)     ← 只标记 GC Root 直接引用的对象，很快
2. 并发标记            ← 和用户线程并发，遍历引用链
3. 重新标记 (STW)     ← 修正并发标记期间变动的对象
4. 并发清除            ← 和用户线程并发，清除垃圾

优点：停顿时间短
缺点：内存碎片（标记-清除）、CPU 敏感、浮动垃圾
```

### G1（Garbage First）

```
把堆划分为多个大小相等的 Region：

┌────┬────┬────┬────┬────┬────┐
│ E  │ S  │ O  │ E  │ O  │ H  │
├────┼────┼────┼────┼────┼────┤
│ O  │ E  │    │ O  │ E  │ O  │
├────┼────┼────┼────┼────┼────┤
│ E  │ O  │ E  │ S  │ O  │    │
└────┴────┴────┴────┴────┴────┘

E = Eden, S = Survivor, O = Old, H = Humongous（大对象）

每个 Region 可以动态切换角色
优先回收垃圾最多的 Region（Garbage First）
可以设置期望停顿时间（-XX:MaxGCPauseMillis=200）
```

## 八、类的回收

### 类回收的条件

**一个类可以被回收，必须同时满足三个条件：**

```
1. 该类的所有实例都已被回收
   堆中不存在该类及其子类的任何实例

2. 加载该类的 ClassLoader 已被回收
   如果是系统 ClassLoader（Bootstrap/App）→ 永远不会被回收
   只有自定义 ClassLoader 才可能被回收

3. 该类的 Class 对象没有被引用
   没有任何地方通过反射引用该 Class

三个条件缺一不可
```

### 哪些类能被回收

```
几乎不能回收：
  系统类（java.lang.*）         → ClassLoader 不会被回收
  App 自己的类                  → AppClassLoader 不会被回收
  框架类（Android SDK）         → 同上

可能被回收：
  动态加载的插件类               → 自定义 ClassLoader 可以被回收
  JSP 生成的类                  → 每次修改会生成新类
  动态代理生成的类               → 如果不再使用
  CGLIB 动态生成的类             → 同上
```

### 类回收的流程

```
自定义 ClassLoader 加载了 PluginClass
    │
    ▼
使用完毕后：

1. PluginClass 的所有实例 → 全部被 GC ✅
2. 自定义 ClassLoader → 没有被引用，可以被 GC ✅
3. PluginClass 的 Class 对象 → 没有被反射引用 ✅
    │
    ▼
下次 Full GC 时 → 方法区/元空间中的 PluginClass 元数据被回收
```

### 方法区 / 元空间的变迁

```
JDK 7 及之前：
┌──────────────┐
│  堆           │
├──────────────┤
│  永久代        │  ← 方法区的实现，在堆里面
│  (PermGen)    │     存放类信息、常量池、静态变量
│  固定大小      │     容易 OOM: PermGen space
└──────────────┘

JDK 8+：
┌──────────────┐     ┌──────────────┐
│  堆           │     │  元空间        │  ← 方法区的新实现
│              │     │ (Metaspace)   │     使用本地内存（Native Memory）
│              │     │  动态扩展      │     不再有固定大小限制
└──────────────┘     │  类信息        │     不容易 OOM
                     │  常量池        │
                     └──────────────┘

改变的原因：
  永久代大小难以设定，太小容易 OOM，太大浪费堆空间
  元空间使用本地内存，默认只受系统内存限制
  便于 HotSpot 和 JRockit 合并
```

### 元空间溢出的场景

```
虽然元空间用本地内存，但仍可能溢出：

1. 大量动态生成类
   CGLIB、动态代理、Groovy 等频繁生成新类 → 元空间持续增长

2. 类加载器泄漏
   自定义 ClassLoader 没有被回收 → 加载的类也不能回收

3. 反射大量不同的类
```

## 九、面试高频问题

### Q1: 堆和栈的区别？

| | 堆 | 栈 |
|--|---|---|
| 线程 | 共享 | 私有 |
| 存什么 | 对象实例 | 局部变量、方法调用 |
| 大小 | 大，可动态扩展 | 小，固定深度 |
| 速度 | 慢（需要 GC） | 快（自动出栈） |
| 异常 | OutOfMemoryError | StackOverflowError |

### Q2: 对象一定分配在堆上吗？

- 不一定。JIT 编译器优化后可能发生**逃逸分析**：
  - 如果对象不会逃出方法 → **栈上分配**（方法结束自动释放，不需要 GC）
  - 如果对象可以拆散 → **标量替换**（用局部变量代替对象）

### Q3: Minor GC 和 Full GC 的区别？

| | Minor GC | Full GC |
|--|---------|---------|
| 回收区域 | 新生代 | 整个堆 + 方法区 |
| 触发条件 | Eden 满 | 老年代满 / 方法区满 / System.gc() |
| 速度 | 快（毫秒级） | 慢（百毫秒~秒级） |
| STW | 短 | 长 |
| 频率 | 频繁 | 尽量避免 |

### Q4: 什么情况对象直接进入老年代？

- 大对象（超过 `-XX:PretenureSizeThreshold`）
- 长期存活（年龄达到 15，`-XX:MaxTenuringThreshold`）
- Survivor 空间不足，动态年龄判断（相同年龄的对象总大小 > Survivor 一半 → 该年龄及以上全部晋升）

### Q5: Android 上的 GC 和标准 JVM 有什么区别？

```
Android 用的不是 HotSpot JVM，而是 ART（Android Runtime）

ART 的 GC 特点：
  - 并发复制收集器（Concurrent Copying Collector）
  - 分代收集（和 JVM 类似）
  - 读屏障（Read Barrier）代替写屏障
  - 堆压缩（减少碎片）
  - 后台线程做 GC，尽量不暂停主线程

ART vs Dalvik：
  Dalvik（旧）→ 标记-清除，STW 时间长
  ART（新）  → 并发 GC，STW 时间大幅缩短
```

### Q6: 类的回收条件是什么？

- 该类所有实例已被回收
- 加载该类的 ClassLoader 已被回收
- 该类的 Class 对象没有被引用
- 三个条件同时满足才会回收，实际中系统类和 App 类几乎不会被回收
