# Java 线程池详解与最佳实践

## 一、为什么需要线程池

```
没有线程池：
  任务 1 → new Thread() → 执行 → 销毁
  任务 2 → new Thread() → 执行 → 销毁
  问题：频繁创建/销毁线程开销大，无法控制并发数量，可能 OOM

有线程池：
  任务 1 → 线程池中的线程 A 执行 → 执行完不销毁，等待下一个任务
  任务 2 → 线程 A 空闲，直接复用
  线程复用，数量可控，统一管理
```

## 二、ThreadPoolExecutor 7 个核心参数

```java
public ThreadPoolExecutor(
    int corePoolSize,         // 核心线程数（常驻线程）
    int maximumPoolSize,      // 最大线程数
    long keepAliveTime,       // 非核心线程的空闲存活时间
    TimeUnit unit,            // 存活时间单位
    BlockingQueue<Runnable> workQueue,   // 任务队列
    ThreadFactory threadFactory,         // 线程工厂（给线程命名）
    RejectedExecutionHandler handler     // 拒绝策略
)
```

```
参数关系图：

                    线程池
┌──────────────────────────────────────────┐
│                                          │
│  核心线程 (corePoolSize = 3)              │
│  ┌─────┐ ┌─────┐ ┌─────┐               │
│  │线程1 │ │线程2 │ │线程3 │  ← 不会被回收  │
│  └─────┘ └─────┘ └─────┘               │
│                                          │
│  非核心线程 (maximumPoolSize - core = 2)   │
│  ┌─────┐ ┌─────┐                        │
│  │线程4 │ │线程5 │  ← 空闲超过 keepAliveTime 就回收
│  └─────┘ └─────┘                        │
│                                          │
│  任务队列 (workQueue)                     │
│  ┌─────┬─────┬─────┬─────┬─────┐       │
│  │任务A │任务B │任务C │     │     │       │
│  └─────┴─────┴─────┴─────┴─────┘       │
│                                          │
│  队列也满了 + 线程也满了 → 拒绝策略 handler  │
└──────────────────────────────────────────┘
```

## 三、任务提交流程（核心）

```
提交一个新任务
    │
    ▼
① 当前线程数 < corePoolSize？
    │
    ├── 是 → 创建核心线程执行任务（即使有空闲核心线程也会创建新的，直到达到 core）
    │
    └── 否 ↓
    │
    ▼
② 任务队列 workQueue 未满？
    │
    ├── 是 → 任务放入队列，等待空闲线程来取
    │
    └── 否 ↓
    │
    ▼
③ 当前线程数 < maximumPoolSize？
    │
    ├── 是 → 创建非核心线程执行任务
    │
    └── 否 ↓
    │
    ▼
④ 触发拒绝策略（RejectedExecutionHandler）
```

**注意顺序：先填满核心线程 → 再填满队列 → 再创建非核心线程 → 最后拒绝**

```
举例：corePoolSize=2, maximumPoolSize=4, queueCapacity=3

任务 1 → 当前 0 个线程 < core(2) → 创建核心线程 1 执行
任务 2 → 当前 1 个线程 < core(2) → 创建核心线程 2 执行
任务 3 → 当前 2 个线程 = core(2) → 放入队列 [3]
任务 4 → 队列未满                → 放入队列 [3, 4]
任务 5 → 队列未满                → 放入队列 [3, 4, 5]
任务 6 → 队列满了！当前 2 < max(4)→ 创建非核心线程 3 执行
任务 7 → 队列满！当前 3 < max(4) → 创建非核心线程 4 执行
任务 8 → 队列满！当前 4 = max(4) → 拒绝策略！
```

## 四、四种拒绝策略

```
┌─────────────────────┬───────────────────────────────────┐
│ AbortPolicy（默认）   │ 直接抛 RejectedExecutionException │
│                     │ 调用方感知到，可以做兜底              │
├─────────────────────┼───────────────────────────────────┤
│ CallerRunsPolicy     │ 由提交任务的线程自己执行             │
│                     │ 不丢弃，但会阻塞提交线程             │
├─────────────────────┼───────────────────────────────────┤
│ DiscardPolicy        │ 直接丢弃，什么都不做                │
│                     │ 静默丢失，不推荐                    │
├─────────────────────┼───────────────────────────────────┤
│ DiscardOldestPolicy  │ 丢弃队列中最早的任务，重新提交当前任务│
│                     │ 可能丢失重要任务                    │
└─────────────────────┴───────────────────────────────────┘
```

## 五、常见队列类型

```
┌─────────────────────┬────────────────────────────────────┐
│ LinkedBlockingQueue  │ 无界队列（默认 Integer.MAX_VALUE）   │
│                     │ 任务堆积不会触发非核心线程创建        │
│                     │ 可能 OOM！                          │
├─────────────────────┼────────────────────────────────────┤
│ ArrayBlockingQueue   │ 有界队列（指定容量）                 │
│                     │ 队列满了才创建非核心线程              │
│                     │ 推荐使用                            │
├─────────────────────┼────────────────────────────────────┤
│ SynchronousQueue     │ 不存储任务（容量为 0）               │
│                     │ 直接交给线程，没有空闲线程就创建新的   │
│                     │ CachedThreadPool 用的               │
├─────────────────────┼────────────────────────────────────┤
│ PriorityBlockingQueue│ 优先级队列                          │
│                     │ 按优先级执行，不是 FIFO               │
└─────────────────────┴────────────────────────────────────┘
```

## 六、为什么不推荐 Executors 创建线程池

```java
// ❌ 这四种都有隐患
Executors.newFixedThreadPool(n)     // LinkedBlockingQueue 无界 → 任务堆积 OOM
Executors.newSingleThreadExecutor() // LinkedBlockingQueue 无界 → 同上
Executors.newCachedThreadPool()     // maximumPoolSize = MAX_VALUE → 线程暴涨 OOM
Executors.newScheduledThreadPool(n) // DelayedWorkQueue 无界 → 任务堆积 OOM
```

```
具体分析：

FixedThreadPool:
  new ThreadPoolExecutor(n, n, 0, LinkedBlockingQueue())
                                   ↑ 无界队列，容量 = Integer.MAX_VALUE
  核心线程忙不过来 → 任务全部堆积在队列里 → 内存不断增长 → OOM

CachedThreadPool:
  new ThreadPoolExecutor(0, Integer.MAX_VALUE, 60s, SynchronousQueue())
                            ↑ 最大线程数无限
  每来一个任务就创建线程 → 高并发时线程暴涨 → OOM

阿里巴巴 Java 开发手册明确规定：
  线程池不允许使用 Executors 创建，必须用 ThreadPoolExecutor
```

## 七、项目最佳实践

### 1. 手动创建，明确参数

```kotlin
val threadPool = ThreadPoolExecutor(
    4,                                          // 核心线程：CPU 密集型 = CPU 核心数
    8,                                          // 最大线程：IO 密集型可以多一些
    60L, TimeUnit.SECONDS,                      // 非核心线程空闲 60 秒回收
    ArrayBlockingQueue(100),                    // 有界队列！容量 100
    MyThreadFactory("business-pool"),           // 自定义线程名
    ThreadPoolExecutor.CallerRunsPolicy()       // 拒绝策略：调用方执行
)
```

### 2. 自定义 ThreadFactory（方便排查问题）

```kotlin
class MyThreadFactory(private val poolName: String) : ThreadFactory {
    private val counter = AtomicInteger(0)

    override fun newThread(r: Runnable): Thread {
        return Thread(r, "$poolName-thread-${counter.incrementAndGet()}").apply {
            isDaemon = false
            uncaughtExceptionHandler = Thread.UncaughtExceptionHandler { t, e ->
                Log.e("ThreadPool", "Thread ${t.name} crashed", e)
            }
        }
    }
}

// 出问题时日志：business-pool-thread-3 crashed: NullPointerException
// 而不是默认的 pool-1-thread-3（看不出是哪个业务的线程池）
```

### 3. 核心线程数怎么定

```
CPU 密集型任务（计算、排序、编解码）：
  核心线程数 = CPU 核心数
  线程太多反而频繁切换，降低效率

IO 密集型任务（网络请求、文件读写、数据库）：
  核心线程数 = CPU 核心数 * 2（或更多）
  线程大部分时间在等 IO，CPU 空闲，可以多开线程

混合型：
  拆成两个线程池，CPU 密集和 IO 密集分开管理

获取 CPU 核心数：
  Runtime.getRuntime().availableProcessors()
```

### 4. Android 项目中的线程池实践

```kotlin
object AppExecutors {

    // IO 密集型：网络、数据库、文件
    val io: ExecutorService = ThreadPoolExecutor(
        4,
        Runtime.getRuntime().availableProcessors() * 2,
        60L, TimeUnit.SECONDS,
        ArrayBlockingQueue(200),
        MyThreadFactory("io-pool"),
        ThreadPoolExecutor.CallerRunsPolicy()
    )

    // CPU 密集型：计算、JSON 解析、图片处理
    val computation: ExecutorService = ThreadPoolExecutor(
        Runtime.getRuntime().availableProcessors(),
        Runtime.getRuntime().availableProcessors(),
        0L, TimeUnit.SECONDS,
        ArrayBlockingQueue(50),
        MyThreadFactory("compute-pool"),
        ThreadPoolExecutor.AbortPolicy()
    )

    // 单线程：顺序执行，如日志写入
    val single: ExecutorService = ThreadPoolExecutor(
        1, 1,
        0L, TimeUnit.SECONDS,
        LinkedBlockingQueue(500),
        MyThreadFactory("single-pool"),
        ThreadPoolExecutor.DiscardOldestPolicy()
    )

    // 主线程 Handler（方便切回主线程）
    val main: Executor = Executor { command ->
        Handler(Looper.getMainLooper()).post(command)
    }
}

// 使用
AppExecutors.io.execute {
    val data = api.fetchData()
    AppExecutors.main.execute {
        textView.text = data
    }
}
```

### 5. 配合协程使用（现代推荐）

```kotlin
// 协程的 Dispatcher 底层就是线程池
// 通常不需要自己创建线程池

Dispatchers.IO          // 共享的 IO 线程池，最多 64 个线程
Dispatchers.Default     // CPU 密集型，线程数 = CPU 核心数
Dispatchers.Main        // 主线程

// 自定义协程 Dispatcher（基于自己的线程池）
val customDispatcher = AppExecutors.io.asCoroutineDispatcher()

viewModelScope.launch(customDispatcher) {
    val data = repository.fetchData()
    withContext(Dispatchers.Main) {
        textView.text = data
    }
}
```

### 6. 线程池监控

```kotlin
fun logPoolStatus(pool: ThreadPoolExecutor, name: String) {
    Log.d("ThreadPool", """
        [$name]
        核心线程数: ${pool.corePoolSize}
        当前线程数: ${pool.poolSize}
        活跃线程数: ${pool.activeCount}
        队列任务数: ${pool.queue.size}
        已完成任务: ${pool.completedTaskCount}
        最大线程数: ${pool.largestPoolSize}
    """.trimIndent())
}

// 队列任务数持续增长 → 线程数不够或任务太慢
// 活跃线程数始终等于最大线程数 → 可能需要扩容
```

### 7. 线程池关闭

```kotlin
// 优雅关闭：等待已提交的任务执行完
pool.shutdown()
// 等待最多 10 秒
if (!pool.awaitTermination(10, TimeUnit.SECONDS)) {
    // 超时，强制关闭
    pool.shutdownNow()
}

// shutdown()     → 不接受新任务，等已有任务执行完
// shutdownNow()  → 不接受新任务，中断正在执行的，返回队列中未执行的
```

## 八、面试高频问题

### Q1: 线程池的任务提交顺序？

- 核心线程未满 → 创建核心线程
- 核心线程满了 → 放入队列
- 队列满了 → 创建非核心线程
- 非核心也满了 → 拒绝策略
- **注意：不是先创建到最大线程再放队列，而是先放队列！**

### Q2: 核心线程会被回收吗？

- 默认不会，核心线程即使空闲也常驻
- 可以设置 `allowCoreThreadTimeOut(true)`，核心线程也按 keepAliveTime 回收

### Q3: 线程池中线程抛异常会怎样？

```
execute() 提交：
  线程直接抛出异常 → 线程死亡 → 线程池创建新线程补充
  异常会打印到 stderr（如果没有 UncaughtExceptionHandler）

submit() 提交：
  异常被封装在 Future 中 → 调用 future.get() 时才抛出
  线程不会死亡
  如果不调 get()，异常会被吞掉！

建议：
  用 execute() 时设置 UncaughtExceptionHandler
  用 submit() 时一定要 get() 或者在任务内 try-catch
```

### Q4: FixedThreadPool 和手动创建的区别？

```java
// FixedThreadPool
new ThreadPoolExecutor(n, n, 0, new LinkedBlockingQueue<>())
// 队列无界 → 任务无限堆积 → OOM

// 手动创建
new ThreadPoolExecutor(n, n, 0, new ArrayBlockingQueue<>(100))
// 队列有界 → 超出 100 触发拒绝策略 → 可控
```

### Q5: Android 中 AsyncTask 的线程池是什么？

```
AsyncTask 内部用的线程池（已废弃）：
  核心线程 = 1
  最大线程 = 20
  队列 = LinkedBlockingQueue(128)

  Android 3.0 之后 AsyncTask 默认串行执行（SerialExecutor）
  现在推荐用协程替代 AsyncTask
```

### Q6: 协程的 Dispatchers.IO 底层线程池是怎样的？

```
Dispatchers.IO：
  共享的线程池，最多 64 个线程（或 CPU 核心数，取大者）
  和 Dispatchers.Default 共享线程
  Default 用完的线程可以被 IO 复用

Dispatchers.Default：
  线程数 = CPU 核心数（最少 2 个）
  用于 CPU 密集型任务

两者共享同一个线程池，通过 LimitingDispatcher 控制并发数
```
