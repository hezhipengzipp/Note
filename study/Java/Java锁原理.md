# Java 锁原理

## 一、前置知识：对象头 Mark Word

Java 对象在内存中的布局，前 64 位（64 位 JVM）是 Mark Word，锁信息就存在这里：

```
┌──────────────────────────────────────────────────────┬────────┬───────┐
│                    Mark Word (64 bit)                 │ 锁标志位│ 状态  │
├──────────────────────────────────────────────────────┼────────┼───────┤
│ [unused:25 | hashCode:31 | age:4 | biased:0]        │   01   │ 无锁  │
│ [threadId:54 | epoch:2   | age:4 | biased:1]        │   01   │ 偏向锁│
│ [ptr to Lock Record in stack]                        │   00   │ 轻量级│
│ [ptr to Monitor (heavyweight)]                       │   10   │ 重量级│
│ [空]                                                  │   11   │ GC标记│
└──────────────────────────────────────────────────────┴────────┴───────┘
```

## 二、偏向锁

### 1. 设计动机

研究发现大多数情况下，锁不仅不存在多线程竞争，而且总是由**同一个线程**反复获取。每次都做 CAS 是浪费，所以偏向锁的思路是：**第一次 CAS 之后，后续直接免锁进入**。

### 2. 加锁流程

```
线程 A 第一次进入 synchronized 块
         │
         ▼
  ┌───────────────────────────┐
  │ 检查 Mark Word 锁标志位    │
  │ 是否为 01 且 biased = 1   │
  │ （可偏向状态）              │
  └───────────┬───────────────┘
              │
       ┌──────┴──────┐
       │             │
    无线程 ID       已有线程 ID
    （匿名偏向）
       │             │
       ▼             ▼
  CAS 将自己的      线程 ID 是自己？
  线程 ID 写入           │
  Mark Word        ┌────┴────┐
       │           │         │
    CAS 成功       是        否
       │           │         │
       ▼           ▼         ▼
    偏向成功     直接进入    发生竞争
    进入同步块   同步块      → 撤销偏向锁
               （零开销！）
```

### 3. 内存视角

```
首次偏向前（匿名可偏向）：
┌──────────────────────────────────────┬────────┐
│ [0x0000000000 | epoch:2 | age:4 | 1] │   01   │
│  threadId 为空        biased=1 ──┘   │        │
└──────────────────────────────────────┴────────┘

首次偏向后（线程 A 持有）：
┌──────────────────────────────────────┬────────┐
│ [threadA_id  | epoch:2 | age:4 | 1]  │   01   │
│  写入线程 A 的 ID                     │        │
└──────────────────────────────────────┴────────┘

线程 A 再次进入：
  只需比对 threadId == 当前线程？ → 是 → 直接进入，无 CAS
```

**关键区别**：偏向锁标志位仍然是 `01`，和无锁一样，靠 `biased` 位区分。

### 4. 偏向锁撤销

偏向锁没有主动释放，线程退出同步块后 Mark Word **不变**，ID 留着。撤销发生在**其他线程来竞争**时：

```
线程 B 尝试获取锁（Mark Word 中是线程 A 的 ID）
         │
         ▼
  ┌─────────────────────────┐
  │ 等待全局安全点            │
  │ （Safe Point, STW）      │
  └───────────┬─────────────┘
              │
              ▼
  ┌─────────────────────────┐
  │ 检查线程 A 是否还活着     │
  │ 且仍在同步块内？          │
  └───────────┬─────────────┘
              │
       ┌──────┴──────┐
       │             │
   A 已退出同步块   A 仍在同步块内
       │             │
       ▼             ▼
  撤销偏向锁       升级为轻量级锁
  恢复为无锁       （给 A 创建 Lock Record，
  或重新偏向 B      CAS 替换对象头为 00）
       │             │
       ▼             ▼
  线程 B 重新      线程 B 按轻量级锁
  竞争获取锁       流程 CAS 竞争
```

> **注意**：撤销需要 STW（Stop The World），这就是偏向锁的代价。

### 5. 批量重偏向 & 批量撤销

JVM 不会无限容忍频繁撤销：

```
同一个类的对象偏向锁撤销次数
         │
         ├── 达到 20 次 → 批量重偏向（Bulk Rebias）
         │                epoch +1，后续对象自动偏向新线程
         │
         └── 达到 40 次 → 批量撤销（Bulk Revoke）
                          该类所有对象永久禁用偏向锁
```

### 6. 面试常问

- **Q：为什么 JDK 15 默认禁用了偏向锁？**
  现代应用多线程竞争普遍，偏向锁撤销的 STW 开销大于它节省的 CAS 开销，得不偿失。可通过 `-XX:+UseBiasedLocking` 手动开启。

- **Q：偏向锁的撤销为什么需要 STW？**
  撤销时需要遍历持有线程的栈帧，确认是否还在同步块内，这必须在安全点暂停线程才能安全操作。

- **Q：调用了 hashCode() 的对象能用偏向锁吗？**
  不能。无锁状态 Mark Word 存 hashCode，偏向锁要用同样的位置存线程 ID，两者冲突。一旦计算了 hashCode，就直接跳过偏向锁。

## 三、轻量级锁

### 1. 加锁流程

### 第一步：创建 Lock Record，拷贝 Mark Word

每个线程有自己的**栈**，每调用一个方法就压入一个**栈帧**。当线程进入 `synchronized` 块时，JVM 在当前栈帧里开辟一块空间叫 **Lock Record**：

```
线程 A 的栈                         堆中的对象
┌─────────────────┐
│  当前栈帧        │
│ ┌─────────────┐ │          ┌──────────────────┐
│ │ Lock Record │ │          │   对象头 Mark Word │
│ │             │ │  拷贝     │                  │
│ │ Displaced   │◄──────────│ hashCode|age|01  │
│ │ Mark Word   │ │          │  (无锁状态)       │
│ │             │ │          ├──────────────────┤
│ └─────────────┘ │          │   对象实例数据     │
│                 │          └──────────────────┘
└─────────────────┘
```

- **做了什么**：把对象头原来的 Mark Word（包含 hashCode、GC 年龄等信息）**复制一份**存到 Lock Record 里，这份备份叫 **Displaced Mark Word**
- **为什么要备份**：因为下一步要把对象头覆盖掉，解锁时需要用这份备份**还原**回去

### 第二步：CAS 替换对象头

备份做好后，线程用 **CAS**（Compare And Swap）做一次原子操作 — 把对象头的 Mark Word 替换成一个指针，指向刚才的 Lock Record：

```
线程 A 的栈                         堆中的对象

┌─────────────────┐
│  当前栈帧        │
│ ┌─────────────┐ │          ┌──────────────────┐
│ │ Lock Record │◄──────────│ ptr to Lock Rec|00│
│ │             │ │  指回来   │  (轻量级锁状态)   │
│ │ Displaced   │ │          ├──────────────────┤
│ │ Mark Word   │ │          │   对象实例数据     │
│ │(原始MW备份) │ │          └──────────────────┘
│ └─────────────┘ │
└─────────────────┘

对象头变化：
  之前: [hashCode | age | 01]    ← 无锁
  之后: [ptr to Lock Record | 00] ← 轻量级锁
```

**CAS 的含义**：
- **Compare**：检查对象头现在是否还是刚才拷贝的那个值（没被别人改过）
- **Swap**：如果是，就原子地替换成指向 Lock Record 的指针，标志位改为 `00`

**CAS 成功** → 对象头指向了你的栈帧，等于"这个锁归你了"
**CAS 失败** → 说明别的线程抢先改了对象头，存在竞争，进入自旋或膨胀

### 完整加锁流程图

```
线程 A 进入 synchronized 块
         │
         ▼
  ┌─────────────────────────┐
  │ 在当前栈帧中创建          │
  │ Lock Record（锁记录）     │
  │ 拷贝对象头 Mark Word      │
  │ → 称为 Displaced Mark Word│
  └───────────┬─────────────┘
              │
              ▼
  ┌─────────────────────────┐
  │ CAS 尝试将对象头 Mark Word│
  │ 替换为指向 Lock Record   │
  │ 的指针，标志位改为 00     │
  └───────────┬─────────────┘
              │
       ┌──────┴──────┐
       │             │
    CAS 成功       CAS 失败
       │             │
       ▼             ▼
   加锁成功      检查 Mark Word 是否
   进入同步块    指向当前线程栈帧？
                     │
              ┌──────┴──────┐
              │             │
             是（锁重入）   否（存在竞争）
              │             │
              ▼             ▼
         计数器 +1      自旋等待
         进入同步块     （有限次 CAS 重试）
                            │
                     ┌──────┴──────┐
                     │             │
                  自旋成功       自旋失败
                     │             │
                     ▼             ▼
                  加锁成功     膨胀为重量级锁
                              标志位 → 10
                              阻塞在 Monitor
```

### 2. 解锁流程

解锁就是加锁的反过程 — 把备份的 Displaced Mark Word 通过 CAS 写回对象头：

```
线程 A 退出 synchronized 块
         │
         ▼
  ┌─────────────────────────────┐
  │ CAS 将 Displaced Mark Word  │
  │ 替换回对象头                 │
  └───────────┬─────────────────┘
              │
       ┌──────┴──────┐
       │             │
    CAS 成功       CAS 失败
       │             │
       ▼             ▼
    解锁成功      说明锁已膨胀为重量级
    恢复无锁      释放 Monitor
                  唤醒等待线程

对象头变化：
  之前: [ptr to Lock Record | 00] ← 轻量级锁
  之后: [hashCode | age | 01]     ← 无锁，恢复原样
```

**一句话总结**：Lock Record 是备份箱，对象头是锁标识。加锁时把原始信息备份到栈帧，对象头改成"指向我"；解锁时把备份还原回去。整个过程靠 CAS 保证原子性，不需要进入内核态。

### 3. 核心要点

| 项目 | 说明 |
|------|------|
| **适用场景** | 多线程交替执行同步块，竞争不激烈 |
| **实现手段** | CAS + 栈帧中的 Lock Record |
| **避免了什么** | 避免重量级锁的内核态切换（用户态自旋代替） |
| **代价** | 自旋消耗 CPU，竞争激烈时反而更慢 |
| **自旋优化** | JDK 6 引入**自适应自旋** — 根据上次自旋成功/失败动态调整次数 |

## 四、自旋锁

### 1. 原理

线程获取锁失败后**不立即阻塞**，而是在原地循环（自旋）不断尝试 CAS 获取锁，赌的是持有锁的线程很快就会释放：

```
线程 B 尝试获取锁（已被线程 A 持有）
         │
         ▼
    ┌─────────┐    CAS 失败     ┌─────────┐
    │  CAS    │───────────────→│  CAS    │──→ ... 循环
    │  尝试   │◄───────────────│  尝试   │◄── ... 重试
    └─────────┘                └─────────┘
         │                          │
      CAS 成功                   超过次数
         │                          │
         ▼                          ▼
      获取锁成功               放弃自旋
      进入同步块               膨胀为重量级锁

线程 A ──────[持有锁]──────→ 释放锁
              很短的时间
```

> 自旋锁不是一种独立的锁状态，而是**轻量级锁竞争失败后的一种策略**。

### 2. 自适应自旋（JDK 6+）

不再固定自旋次数，JVM 根据**历史数据**动态调整：

```
                    上次自旋结果
                         │
              ┌──────────┴──────────┐
              │                     │
           上次成功               上次失败
              │                     │
              ▼                     ▼
       增加自旋次数            减少自旋次数
       （这个锁值得等）        甚至直接跳过自旋
                               （别浪费 CPU 了）
```

### 3. 优缺点

| 优点 | 缺点 |
|------|------|
| 避免线程切换的开销（用户态完成） | 自旋期间一直占用 CPU |
| 锁持有时间短时效率高 | 锁持有时间长时纯浪费 |
| 无需内核态切换 | 多线程同时自旋放大 CPU 浪费 |

## 五、重量级锁

### 1. 原理

当竞争激烈、自旋无法解决时，锁膨胀为重量级锁。底层依赖操作系统的 **Mutex Lock**（互斥量），获取失败的线程会被**挂起（park）**，释放锁时再**唤醒（unpark）**。

### 2. 核心结构：ObjectMonitor

每个重量级锁对应一个 C++ 层的 `ObjectMonitor` 对象：

```
ObjectMonitor
┌─────────────────────────────────────┐
│  _owner        → 当前持有锁的线程    │
│  _count        → 重入计数器          │
│  _EntryList    → 阻塞等待锁的线程队列 │
│  _WaitSet      → 调用 wait() 的线程  │
│  _recursions   → 重入次数            │
└─────────────────────────────────────┘

对象头 Mark Word:
┌──────────────────────────────┬────────┐
│ ptr to ObjectMonitor         │   10   │
│ （指向 Monitor 的指针）       │        │
└──────────────────────────────┴────────┘
```

### 3. 三个队列/集合详解

Monitor 里核心就三个"位置"，线程在它们之间流转：

```
┌─────────────────────────────────────────────────────────┐
│                    ObjectMonitor                         │
│                                                         │
│  _owner  →  当前持有锁的线程（只有一个）                    │
│                                                         │
│  ┌───────────────┐       ┌───────────────┐              │
│  │  _EntryList   │       │   _WaitSet    │              │
│  │  (阻塞队列)    │       │   (等待集合)   │              │
│  │               │       │               │              │
│  │ 想抢锁但没抢到  │       │ 主动调 wait() │              │
│  │ 的线程在这排队  │       │ 放弃锁在这等   │              │
│  └───────────────┘       └───────────────┘              │
└─────────────────────────────────────────────────────────┘
```

| 位置 | 谁在这里 | 怎么进来的 | 怎么出去 |
|------|---------|-----------|---------|
| **_owner** | 正在执行同步块的线程（1个） | 从 _EntryList 竞争胜出 | 执行完退出 or 调 wait() |
| **_EntryList** | 想获取锁但锁被占了的线程 | `synchronized` 时锁已被占 | _owner 释放锁后被唤醒竞争 |
| **_WaitSet** | 主动调了 `wait()` 的线程 | 在同步块内调 `obj.wait()` | 被 `notify()/notifyAll()` 唤醒 |

**_EntryList vs _WaitSet 关键区别**：

```
_EntryList（锁池）                    _WaitSet（等待池）
┌────────────────────┐              ┌────────────────────┐
│ "我要锁，但锁被占了" │              │ "我有锁，但条件没满足│
│                    │              │  我主动放弃等通知"   │
│ 进入方式:           │              │ 进入方式:           │
│   synchronized 抢锁失败│           │   在同步块内调 wait()│
│                    │              │                    │
│ 出去方式:           │              │ 出去方式:           │
│   锁释放后被唤醒竞争  │              │   被 notify() 唤醒   │
│                    │              │   → 移到 _EntryList │
│ 线程状态:           │              │   → 重新竞争锁       │
│   BLOCKED          │              │                    │
│                    │              │ 线程状态:           │
│ 是否持有过锁: 从没有  │              │   WAITING           │
│                    │              │                    │
│                    │              │ 是否持有过锁: 持有过， │
│                    │              │   主动释放了          │
└────────────────────┘              └────────────────────┘
```

> **一句话**：`_EntryList` 是"门外排队的"，`_WaitSet` 是"进去了又自己出来等通知的"。`notify()` 只是把人从等待池搬到锁池，**不是直接给锁**，搬过去之后还要跟锁池里的线程一起竞争。

### 4. 线程流转实例

```
线程 A、B、C 同时要进入 synchronized(obj)
         │
         ▼
    A 抢到锁 ──────────────→  _owner = A
    B、C 没抢到 ───────────→  _EntryList: [B, C]
                                  │
                                  │ （B、C 被 park 挂起，不消耗 CPU）
                                  │
    ══════════════════════════════════════════
    场景 1: A 执行完正常退出
    ══════════════════════════════════════════
                                  │
    A 退出 synchronized ─→ _owner = null
                                  │
                                  ▼
                           唤醒 _EntryList 中的线程
                           B、C 竞争 → B 胜出
                                  │
                                  ▼
                           _owner = B
                           _EntryList: [C]

    ══════════════════════════════════════════
    场景 2: A 在同步块内调了 wait()
    ══════════════════════════════════════════
                                  │
    A 调 obj.wait() ──┬──→ 释放锁，_owner = null
                      └──→ A 进入 _WaitSet: [A]
                                  │
                                  ▼
                           唤醒 _EntryList → B 获取锁
                           _owner = B
                           _EntryList: [C]
                           _WaitSet: [A]
                                  │
    ══════════════════════════════════════════
    场景 3: B 在同步块内调了 notify()
    ══════════════════════════════════════════
                                  │
    B 调 obj.notify() ────→ 从 _WaitSet 取出 A
                              移到 _EntryList
                                  │
                                  ▼
                           _owner = B（B 还没释放锁！）
                           _EntryList: [C, A]
                           _WaitSet: []
                                  │
                           B 退出 synchronized
                                  │
                                  ▼
                           C、A 竞争 _owner
```

### 5. 加锁 / 解锁流程

```
线程 A 进入 synchronized（对象已膨胀为重量级锁）
         │
         ▼
  ┌─────────────────────────┐
  │ 通过对象头找到            │
  │ ObjectMonitor            │
  └───────────┬─────────────┘
              │
              ▼
  ┌─────────────────────────┐
  │ _owner 是否为空？        │
  └───────────┬─────────────┘
              │
       ┌──────┴──────┐
       │             │
      为空         不为空
       │             │
       ▼             ▼
   设置 _owner    _owner 是自己？
   为当前线程          │
   _count = 1    ┌────┴────┐
       │         │         │
       ▼         是        否
    获取成功      │         │
              重入         ▼
          _recursions++  进入 _EntryList
              │          线程挂起（park）
              ▼          等待被唤醒
           获取成功            │
                              │
                    ─ ─ ─ ─ ─ ┘
                    （持有线程释放后）
                              │
                              ▼
                    从 _EntryList 中
                    选一个线程唤醒（unpark）
                    竞争获取 _owner
```

### 6. wait / notify 与 Monitor 的关系

```
synchronized(obj) {
    obj.wait();     // 释放锁，进入 _WaitSet
    ...
    obj.notify();   // 从 _WaitSet 移到 _EntryList
}

                   ┌──────────────┐
        notify()   │              │  wait()
    ┌──────────────│  _WaitSet    │◄──────────┐
    │              │ (等待池)      │            │
    │              └──────────────┘            │
    ▼                                         │
┌──────────────┐   获取 _owner   ┌────────────┴───┐
│ _EntryList   │───────────────→│   同步块执行中   │
│ (锁池/阻塞队列)│◄───────────────│   _owner=当前   │
└──────────────┘   锁释放,竞争    └────────────────┘
```

### 7. wait / notify 使用示例（生产者-消费者）

```java
public class ProducerConsumer {
    private final List<Integer> buffer = new ArrayList<>();
    private final int MAX_SIZE = 5;
    private final Object lock = new Object();

    // 生产者
    public void produce() throws InterruptedException {
        synchronized (lock) {
            // 用 while 不用 if —— 防止虚假唤醒
            while (buffer.size() == MAX_SIZE) {
                System.out.println("缓冲区满，生产者等待...");
                lock.wait();  // 释放锁，进入 _WaitSet
            }
            buffer.add(1);
            System.out.println("生产了一个，当前数量：" + buffer.size());
            lock.notifyAll();  // 唤醒所有等待的线程（移到 _EntryList）
        }
    }

    // 消费者
    public void consume() throws InterruptedException {
        synchronized (lock) {
            while (buffer.isEmpty()) {
                System.out.println("缓冲区空，消费者等待...");
                lock.wait();  // 释放锁，进入 _WaitSet
            }
            buffer.remove(0);
            System.out.println("消费了一个，当前数量：" + buffer.size());
            lock.notifyAll();  // 唤醒所有等待的线程
        }
    }
}
```

执行流程对应 Monitor：

```
消费者调用 consume()
         │
         ▼
    synchronized(lock) 获取锁
    _owner = 消费者线程
         │
         ▼
    buffer 为空 → lock.wait()
         │
         ├── 1. 释放锁，_owner = null
         ├── 2. 消费者进入 _WaitSet
         │
         │   此时生产者可以获取锁了
         │          │
         │          ▼
         │   synchronized(lock)
         │   _owner = 生产者线程
         │          │
         │          ▼
         │   buffer.add(1)
         │   lock.notifyAll()
         │          │
         │          ├── 消费者从 _WaitSet → _EntryList
         │          │
         │          ▼
         │   生产者退出 synchronized，释放锁
         │
         ▼
    消费者从 _EntryList 竞争获取锁
    重新检查 while 条件
    buffer 不为空 → 消费
```

**三个关键点**：

| 要点 | 说明 |
|------|------|
| **必须在 synchronized 内调用** | wait/notify 操作的是 Monitor，没持有锁就调会抛 `IllegalMonitorStateException` |
| **用 while 不用 if** | 被唤醒后要**重新检查条件**，因为可能是虚假唤醒，或者被别的线程抢先消费了 |
| **优先用 notifyAll** | `notify` 只唤醒一个，如果唤醒的恰好不是该干活的线程，可能全部卡死 |

```java
// 错误写法 ❌
if (buffer.isEmpty()) {
    lock.wait();  // 唤醒后不再检查，可能 buffer 又被别人消费空了
}

// 正确写法 ✅
while (buffer.isEmpty()) {
    lock.wait();  // 唤醒后重新检查条件
}
```

### 8. 性能代价

```
用户态                          内核态
┌──────────┐   系统调用    ┌──────────┐
│ 线程运行  │─────────────→│ mutex_lock│
│          │  上下文切换    │ 挂起线程  │
│          │◄─────────────│ 唤醒线程  │
└──────────┘              └──────────┘
     约 0.5-1 μs 的切换开销（偏向锁/轻量级锁为 0）
```

| 开销来源 | 说明 |
|---------|------|
| **用户态 → 内核态切换** | 每次加锁/解锁都可能涉及系统调用 |
| **线程调度** | 挂起和唤醒需要操作系统介入 |
| **上下文切换** | 保存/恢复寄存器、TLB 刷新等 |
| **缓存失效** | 线程切换后 CPU 缓存命中率下降 |

## 六、ReentrantLock

### 1. 底层核心：AQS

ReentrantLock 不直接操作 OS 互斥量，而是基于 **AQS（AbstractQueuedSynchronizer）** 实现。AQS 是 Java 并发包的基石，`ReentrantLock`、`Semaphore`、`CountDownLatch` 都建立在它之上。

```
AQS 核心结构
┌─────────────────────────────────────────────────┐
│  state (volatile int)                            │
│  = 0 → 锁空闲                                    │
│  = 1 → 锁被持有                                  │
│  > 1 → 重入次数                                   │
│                                                  │
│  exclusiveOwnerThread → 当前持有锁的线程           │
│                                                  │
│  CLH 变体双向队列（等待队列）：                      │
│  head ←→ Node(B) ←→ Node(C) ←→ tail             │
│           WAITING     WAITING                    │
└─────────────────────────────────────────────────┘
```

### 2. CLH 队列结构

获取锁失败的线程会被包装成 **Node** 加入队列尾部：

```
     head                                    tail
       │                                       │
       ▼                                       ▼
  ┌─────────┐    prev    ┌─────────┐   prev  ┌─────────┐
  │  哨兵    │◄─────────│ Node(B) │◄────────│ Node(C) │
  │ (空节点) │─────────→│ WAITING │────────→│ WAITING │
  └─────────┘    next    └─────────┘   next  └─────────┘
                              │                   │
                         线程 B 被 park       线程 C 被 park
                         等待前驱唤醒          等待前驱唤醒
```

每个 Node 只关注自己的**前驱节点**状态，前驱释放锁时负责唤醒后继 — 这就是 CLH 的核心思想。

### 3. 加锁流程（非公平锁，默认）

```
线程 A 调用 lock.lock()
         │
         ▼
  ┌─────────────────────────┐
  │ CAS 尝试 state: 0 → 1   │   ← 非公平：不管队列，先抢一次
  └───────────┬─────────────┘
              │
       ┌──────┴──────┐
       │             │
    CAS 成功       CAS 失败
       │             │
       ▼             ▼
  设置 owner      调用 acquire(1)
  为当前线程           │
       │              ▼
       ▼       ┌─────────────────┐
    获取成功    │ tryAcquire(1)   │
               │ 再尝试一次 CAS   │
               └────────┬────────┘
                        │
                 ┌──────┴──────┐
                 │             │
              成功           失败
                 │             │
                 ▼             ▼
              获取成功    ┌─────────────────┐
                        │ state != 0       │
                        │ owner 是自己？    │
                        └────────┬────────┘
                                 │
                          ┌──────┴──────┐
                          │             │
                         是            否
                          │             │
                          ▼             ▼
                     state++       包装为 Node
                     重入成功      加入 CLH 队列尾部
                                  park 挂起
                                       │
                                  （被前驱唤醒后）
                                       │
                                       ▼
                                  自旋尝试获取锁
                                  （只有队列头部才能抢）
```

### 4. 解锁流程

```
线程 A 调用 lock.unlock()
         │
         ▼
  ┌─────────────────────────┐
  │ state - 1                │
  └───────────┬─────────────┘
              │
       ┌──────┴──────┐
       │             │
   state == 0     state > 0
   （完全释放）    （还有重入）
       │             │
       ▼             ▼
  owner = null    返回，锁还持有
  唤醒 CLH 队列
  head 的后继节点
  （unpark）
       │
       ▼
  后继线程被唤醒
  自旋 CAS 获取锁
```

### 5. 公平锁 vs 非公平锁

```java
new ReentrantLock(true);   // 公平锁
new ReentrantLock(false);  // 非公平锁（默认）
```

```
                非公平锁                          公平锁
            ┌─────────────┐               ┌─────────────┐
新线程来了 → │ 先 CAS 插队  │    新线程来了 → │ 先检查队列    │
            │ 抢到就用     │               │ 有人排队？    │
            │ 没抢到再排队  │               │ 有 → 乖乖排队 │
            └─────────────┘               │ 没有 → CAS   │
                                          └─────────────┘
```

| 对比项 | 非公平锁 | 公平锁 |
|--------|---------|--------|
| **吞吐量** | 高（减少线程切换） | 低 |
| **饥饿风险** | 有（新线程可能一直插队） | 无 |
| **实现** | lock 时直接 CAS | lock 前先查队列 |
| **适用场景** | 大多数场景 | 严格要求先来先得 |

### 6. 对比 synchronized

| 对比项 | synchronized | ReentrantLock |
|--------|-------------|---------------|
| **实现层面** | JVM 内置，monitorenter/exit | Java API，基于 AQS |
| **锁类型** | 非公平 | 可选公平/非公平 |
| **可中断** | 不可中断 | `lockInterruptibly()` |
| **超时获取** | 不支持 | `tryLock(timeout)` |
| **条件变量** | 只有一个 wait/notify | 多个 `Condition` |
| **释放方式** | 自动（退出同步块） | 手动 `unlock()`，必须在 finally 中 |
| **锁绑定** | 绑定对象 | 绑定 Lock 实例 |

### 7. Condition 机制

`synchronized` 只有一个等待队列（`_WaitSet`），而 ReentrantLock 可以创建**多个 Condition**，实现精确唤醒：

```java
ReentrantLock lock = new ReentrantLock();
Condition notFull  = lock.newCondition();  // 生产者等待
Condition notEmpty = lock.newCondition();  // 消费者等待
```

```
                    AQS
┌──────────────────────────────────────────┐
│            CLH 等待队列（抢锁）            │
│  head ←→ Node(B) ←→ Node(C) ←→ tail    │
└──────────────────────────────────────────┘

  Condition: notFull              Condition: notEmpty
┌──────────────────┐           ┌──────────────────┐
│ 条件队列（单向）    │           │ 条件队列（单向）    │
│ Node(生产者1)     │           │ Node(消费者1)     │
│   ↓              │           │   ↓              │
│ Node(生产者2)     │           │ Node(消费者2)     │
└──────────────────┘           └──────────────────┘

await()  → 释放锁，从 CLH 队列移到对应 Condition 队列
signal() → 从 Condition 队列移回 CLH 队列，重新竞争锁
```

对比 synchronized 的 wait/notify 只能**随机或全部唤醒**，Condition 可以**精确唤醒指定类型**的线程。

## 七、偏向锁 vs 轻量级锁对比

| 对比项 | 偏向锁 | 轻量级锁 |
|--------|--------|---------|
| **场景** | 始终单线程访问 | 多线程交替访问 |
| **加锁方式** | 首次 CAS 写线程 ID，后续只比对 | 每次都 CAS |
| **解锁方式** | 不主动释放，ID 留在对象头 | CAS 还原 Displaced Mark Word |
| **开销** | 首次一次 CAS，后续零开销 | 每次进出都一次 CAS |
| **撤销代价** | 需要 STW | 无 |
| **标志位** | `01`（biased=1） | `00` |

## 八、锁升级全景对比

```
无锁 ──→ 偏向锁 ──→ 轻量级锁 ──→ 重量级锁
         │          │             │
         │          │             │
      无竞争      交替竞争      激烈竞争
      单线程      CAS 自旋      OS Mutex
      零开销      用户态         内核态
```

> **注意**：锁只能升级，不能降级（偏向锁撤销除外）。

## 九、面试常问

- **Q：轻量级锁和偏向锁的区别？**
  偏向锁是无竞争时连 CAS 都省掉，直接比对线程 ID；轻量级锁是有交替竞争时用 CAS 自旋避免阻塞。

- **Q：轻量级锁一定比重量级锁快吗？**
  不一定。竞争激烈时自旋空转浪费 CPU，反而不如直接阻塞让出 CPU。

- **Q：什么是自适应自旋？**
  JVM 根据同一个锁上次自旋的结果动态调整自旋次数 — 上次成功则多转几次，上次失败则少转或直接膨胀。

- **Q：为什么 JDK 15 默认禁用了偏向锁？**
  现代应用多线程竞争普遍，偏向锁撤销的 STW 开销大于它节省的 CAS 开销，得不偿失。

- **Q：调用了 hashCode() 的对象能用偏向锁吗？**
  不能。无锁状态 Mark Word 存 hashCode，偏向锁要用同样的位置存线程 ID，两者冲突。

- **Q：重量级锁为什么慢？**
  需要用户态 → 内核态切换，涉及系统调用 mutex，线程挂起/唤醒都由 OS 调度，一次切换约 0.5-1 微秒。

- **Q：wait() 和 sleep() 的区别？**
  `wait()` 释放锁进入 `_WaitSet`，必须在 synchronized 中调用；`sleep()` 不释放锁，只是让出 CPU 时间片。

- **Q：notify() 和 notifyAll() 的区别？**
  `notify()` 随机唤醒 `_WaitSet` 中一个线程移到 `_EntryList`；`notifyAll()` 唤醒全部，但它们仍需竞争 `_owner`。

- **Q：ReentrantLock 怎么实现可重入的？**
  AQS 的 `state` 记录重入次数，`exclusiveOwnerThread` 记录持有线程。加锁时发现 owner 是自己就 `state++`，解锁时 `state--`，减到 0 才真正释放。

- **Q：为什么默认非公平？**
  非公平锁允许新线程直接 CAS 插队，如果恰好锁刚释放就能立即获取，减少了线程切换开销，吞吐量更高。

- **Q：AQS 为什么用 CLH 队列而不是普通队列？**
  CLH 队列每个节点只自旋检查前驱状态，竞争分散到各节点，避免所有线程同时 CAS 同一变量（惊群效应）。

- **Q：什么时候用 ReentrantLock 而不是 synchronized？**
  需要公平锁、可中断、超时获取、多条件变量时用 ReentrantLock；简单同步场景用 synchronized 更简洁。
