# HTTPS 证书验证与密钥交换

## 一、TLS 是什么

**TLS（Transport Layer Security，传输层安全协议）** 就是 HTTPS 中那个 "S" 的实现。

```
HTTP  = 明文传输
HTTPS = HTTP + TLS = 加密传输
```

它的前身叫 **SSL**（Secure Sockets Layer），现在 SSL 已经废弃，但很多人还是混着叫"SSL/TLS"。版本演进：

```
SSL 1.0 → SSL 2.0 → SSL 3.0 → TLS 1.0 → TLS 1.1 → TLS 1.2 → TLS 1.3（当前主流）
         （已废弃）  （已废弃）  （已废弃）  （已废弃）   （广泛使用）  （推荐）
```

TLS 工作在**传输层和应用层之间**，对上层应用透明：

```
┌──────────────┐
│  HTTP / 应用层 │
├──────────────┤
│  TLS          │  ← 在这里加密/解密
├──────────────┤
│  TCP / 传输层  │
├──────────────┤
│  IP / 网络层   │
└──────────────┘
```

TLS 做三件事：**加密**（防窃听）、**认证**（防冒充）、**完整性**（防篡改）。

## 二、非对称加密交换密钥过程

非对称加密本身不直接用于加密大量数据，而是用来**安全地交换对称密钥**。

### 核心概念

- **公钥**：公开的，任何人可获取
- **私钥**：保密的，仅持有者拥有
- 公钥加密的数据，只有对应私钥能解密

### RSA 密钥交换流程

```
客户端 (Alice)                              服务端 (Bob)
    |                                           |
    |  1. 请求连接（ClientHello）                 |
    |  ----------------------------------------> |
    |                                           |
    |  2. 返回证书（含 Bob 的公钥）                 |
    |  <---------------------------------------- |
    |                                           |
    |  3. 验证证书合法性（CA 签名链）                |
    |                                           |
    |  4. 生成随机对称密钥（Pre-Master Secret）      |
    |     用 Bob 的公钥加密该密钥                    |
    |  ----------------------------------------> |
    |                                           |
    |          5. Bob 用自己的私钥解密，得到对称密钥   |
    |                                           |
    |  6. 双方用该对称密钥进行后续加密通信             |
    |  <=======================================> |
```

### 关键步骤说明

1. **Bob 提供公钥**：通过数字证书发送，证书由 CA（证书颁发机构）签名，防止中间人伪造
2. **Alice 验证证书**：用 CA 的公钥验证证书签名，确认公钥确实属于 Bob
3. **Alice 加密对称密钥**：生成一个随机的对称密钥，用 Bob 的公钥加密后发送
4. **Bob 解密**：只有 Bob 的私钥能解密，中间人截获密文也无法解开
5. **切换对称加密**：后续通信使用协商好的对称密钥（如 AES），因为对称加密速度远快于非对称

### 更现代的方式：ECDHE 密钥交换

现代 TLS 1.3 更常用 **ECDHE**（椭圆曲线 Diffie-Hellman 临时密钥交换）：

```
Alice                                       Bob
  |  各自生成临时的 ECDH 密钥对                  |
  |  交换各自的 ECDH 公钥                       |
  |  <---------------------------------------->|
  |                                            |
  |  双方各自用「自己的私钥 + 对方的公钥」计算出     |
  |  相同的共享密钥（数学特性保证结果一致）          |
```

ECDHE 的优势是**前向保密**（Forward Secrecy）：即使服务器的长期私钥将来泄露，之前的通信也无法被解密，因为每次会话使用的临时密钥已经销毁。

## 三、RSA 握手 vs DH 握手

两者的核心区别在于：**对称密钥是怎么产生的**。

### RSA 握手流程

```
客户端                                     服务端
  |                                          |
  |  ---- ClientHello ---------------------->|
  |  <--- ServerHello + 证书(含公钥) ---------|
  |                                          |
  |  客户端生成 Pre-Master Secret              |
  |  用服务端公钥加密                           |
  |  ---- 加密后的 Pre-Master Secret -------->|
  |                                          |
  |           服务端用私钥解密，得到 Pre-Master   |
  |                                          |
  |  双方用 Pre-Master 推导出对称密钥            |
  |  <========= 对称加密通信 =================>|
```

**特点：密钥由客户端单方面生成，用服务端公钥加密传过去。**

### DH（Diffie-Hellman）握手流程

```
客户端                                     服务端
  |                                          |
  |  ---- ClientHello ---------------------->|
  |  <--- ServerHello + 证书 + DH 参数 -------|
  |                                          |
  |  客户端生成 DH 临时密钥对                    |
  |  服务端生成 DH 临时密钥对                    |
  |                                          |
  |  ---- 客户端 DH 公钥 ------------------->|
  |  <--- 服务端 DH 公钥 --------------------|
  |                                          |
  |  双方各自计算：                             |
  |  自己的 DH 私钥 × 对方的 DH 公钥 = 共享密钥  |
  |  （数学特性保证两边算出来一样）                |
  |                                          |
  |  <========= 对称加密通信 =================>|
```

**特点：密钥是双方各自计算出来的，Pre-Master Secret 从未在网络上传输过。**

### 关键区别对比

| | RSA 握手 | DH 握手（ECDHE） |
|--|---------|-----------------|
| **密钥怎么来的** | 客户端生成，用服务端公钥加密后传输 | 双方各自计算，密钥本身不传输 |
| **前向保密** | **没有** | **有** |
| **服务端私钥泄露后** | 之前所有通信都能被解密 | 之前的通信仍然安全 |
| **性能** | 略快（少一步交换） | 略慢（多一步 DH 计算） |
| **TLS 1.3 支持** | 已移除 | 唯一支持的方式 |

### 前向保密 — DH 取代 RSA 的根本原因

**RSA 没有前向保密：**

```
攻击者录制了过去 3 年所有的加密流量
        ↓
某天服务端私钥泄露
        ↓
攻击者用私钥解密 Pre-Master Secret
        ↓
推导出所有会话的对称密钥
        ↓
过去 3 年的通信全部被解密 💀
```

**DH 有前向保密：**

```
攻击者录制了过去 3 年所有的加密流量
        ↓
某天服务端私钥泄露
        ↓
但每次会话用的 DH 临时密钥早已销毁
        ↓
没有临时私钥，无法计算共享密钥
        ↓
过去的通信仍然安全 ✅
```

RSA 握手中，服务端私钥既用于**身份认证**又用于**密钥交换**，一旦泄露全盘皆输。DH 握手中，服务端私钥**只用于身份认证**（签名 DH 参数），密钥交换靠的是临时密钥，用完即毁。TLS 1.3 已经只保留 DH，彻底淘汰了 RSA 密钥交换。

## 四、证书链验证的具体过程

### 证书里有什么

```
┌─────────────────────────────────┐
│  证书内容（明文部分）              │
│  - 持有者域名（如 example.com）   │
│  - 持有者的公钥                   │
│  - 颁发者（Issuer）名称           │
│  - 有效期                        │
│  - 序列号                        │
├─────────────────────────────────┤
│  数字签名                        │
│  = 颁发者用自己的私钥对            │
│    上述内容的哈希值进行加密的结果    │
└─────────────────────────────────┘
```

### 签名验证的核心逻辑

用颁发者的公钥解密签名，得到哈希值 A；再对证书明文内容自己算一遍哈希，得到哈希值 B。**A == B 则证书未被篡改，且确实是该颁发者签发的。**

```
颁发者私钥签名：Hash(证书内容) → 加密 → 签名
验证过程：
  签名 → 用颁发者公钥解密 → 哈希值 A
  证书内容 → 重新计算哈希 → 哈希值 B
  A == B ? ✅ 合法 : ❌ 伪造/被篡改
```

### 证书链的逐级验证

```
┌──────────────────┐
│   根证书 (Root CA) │  ← 预装在操作系统/浏览器中，自签名，无条件信任
│   公钥: PK_root    │
└────────┬─────────┘
         │ 用 Root 私钥签名
         ▼
┌──────────────────────┐
│  中间证书 (Intermediate CA) │  ← 由根 CA 签发
│  公钥: PK_mid               │
└────────┬─────────────────┘
         │ 用 Intermediate 私钥签名
         ▼
┌──────────────────────────┐
│  服务器证书 (example.com)   │  ← 由中间 CA 签发
│  公钥: PK_server            │
└──────────────────────────┘
```

验证过程**从下往上**逐级进行：

- **第 1 步：验证服务器证书** — 取 Intermediate CA 的公钥解密签名，与证书内容哈希对比
- **第 2 步：验证中间证书** — 取 Root CA 的公钥解密签名，与中间证书内容哈希对比
- **第 3 步：根证书** — 在本地信任库中预装，直接信任，链条终止

### 除了签名验证，还检查什么

| 检查项 | 说明 |
|--------|------|
| **域名匹配** | 证书中的域名是否与访问的域名一致 |
| **有效期** | 当前时间是否在证书的 Not Before ~ Not After 之间 |
| **吊销状态** | 通过 CRL 或 OCSP 查询证书是否已被吊销 |
| **密钥用途** | 证书的 Key Usage 扩展是否允许用于服务器认证 |

### 为什么需要中间 CA

根 CA 的私钥极其珍贵（通常离线保存在硬件安全模块中）。加一层中间 CA：中间 CA 的私钥泄露了，只需吊销该中间证书，根 CA 重新签发一个新的中间 CA 即可，**影响范围可控**。

## 五、Android 中的证书验证

### 各层职责

```
┌─────────────────────────────────────┐
│  业务代码                            │  ← 通常什么都不用写
├─────────────────────────────────────┤
│  Retrofit                           │  ← 只负责接口定义、序列化，不管证书
├─────────────────────────────────────┤
│  OkHttp                             │  ← 负责 TLS 握手的发起和配置
│  内部使用 SSLSocketFactory +          │
│  X509TrustManager                   │
├─────────────────────────────────────┤
│  Android 系统 (Java Security API)    │  ← 真正执行证书链验证的地方
│  TrustManagerImpl → CertPathValidator │
├─────────────────────────────────────┤
│  系统信任库                           │  ← /system/etc/security/cacerts/
│  预装了 ~150 个根证书                  │
└─────────────────────────────────────┘
```

- **Retrofit** 完全不参与证书验证，只是 OkHttp 上面的一层封装
- **OkHttp** 默认使用系统提供的 `TrustManager`，不需要额外配置
- **系统的 `TrustManagerImpl`** 完成整个链式验证过程

### 需要手动介入的场景

#### 1. 证书固定（Certificate Pinning）

防止中间人用合法但非预期的证书冒充服务器，在系统验证通过之后**额外检查**证书公钥哈希：

```kotlin
val client = OkHttpClient.Builder()
    .certificatePinner(
        CertificatePinner.Builder()
            .add("api.example.com", "sha256/AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA=")
            .build()
    )
    .build()
```

#### 2. 自签名证书（测试环境）

测试服务器的自签名证书不在系统信任库中，需要自定义 `TrustManager`：

```kotlin
val trustManager = object : X509TrustManager {
    override fun checkServerTrusted(chain: Array<X509Certificate>, authType: String) {
        // 自定义验证逻辑
    }
    override fun checkClientTrusted(chain: Array<X509Certificate>, authType: String) {}
    override fun getAcceptedIssuers(): Array<X509Certificate> = arrayOf()
}

val sslContext = SSLContext.getInstance("TLS").apply {
    init(null, arrayOf(trustManager), null)
}

val client = OkHttpClient.Builder()
    .sslSocketFactory(sslContext.socketFactory, trustManager)
    .build()
```

#### 3. 双向认证（mTLS）

服务端也要验证客户端身份，需要配置客户端证书：

```kotlin
val keyStore = KeyStore.getInstance("PKCS12").apply {
    load(clientCertInputStream, "password".toCharArray())
}
val kmf = KeyManagerFactory.getInstance("X509").apply {
    init(keyStore, "password".toCharArray())
}
val sslContext = SSLContext.getInstance("TLS").apply {
    init(kmf.keyManagers, null, null)
}
```

### 常见错误做法

信任所有证书 — **绝对不要在生产环境使用**：

```kotlin
// ❌ 等于关闭了整个 HTTPS 的保护，中间人可以随意拦截和篡改数据
override fun checkServerTrusted(chain: Array<X509Certificate>, authType: String) {
    // 什么都不做 = 信任任何证书
}
```

### 场景总结

| 场景 | 需要做什么 |
|------|-----------|
| 正常 HTTPS 请求 | 什么都不用做，系统全权处理 |
| 增强安全（防中间人） | 配置 `CertificatePinner` |
| 测试环境自签名证书 | 自定义 `TrustManager` |
| 双向认证 mTLS | 配置 `KeyManager` + `TrustManager` |
