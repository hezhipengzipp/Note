# 网络分层模型

## 一、两套主流模型

有两套主流模型：**OSI 七层**（理论参考）和 **TCP/IP 四层**（实际使用）。

### 对照关系

```
    OSI 七层模型              TCP/IP 四层模型          常见协议/设备
┌─────────────────┐
│  7. 应用层        │
├─────────────────┤
│  6. 表示层        │  ──→   应用层              HTTP, HTTPS, FTP, DNS,
├─────────────────┤                            SMTP, WebSocket, SSH
│  5. 会话层        │
├─────────────────┤
│  4. 传输层        │  ──→   传输层              TCP, UDP  │ 端口号
├─────────────────┤
│  3. 网络层        │  ──→   网络层              IP, ICMP, ARP  │ 路由器
├─────────────────┤
│  2. 数据链路层     │  ──→   网络接口层           以太网, Wi-Fi  │ 交换机
├─────────────────┤                            MAC 地址
│  1. 物理层        │
└─────────────────┘                            光纤, 网线, 无线电波
```

## 二、每层的核心职责

| 层 | 核心职责 | 类比 |
|----|---------|------|
| **应用层** | 提供具体的网络服务（网页、邮件、文件传输） | 你写的信的内容 |
| **表示层** | 数据格式转换、加密解密、压缩 | 把中文翻译成英文 |
| **会话层** | 建立、管理、终止会话 | 打电话时的"喂"和"挂了" |
| **传输层** | 端到端的可靠传输，端口区分进程 | 快递选顺丰(TCP)还是扔过去(UDP) |
| **网络层** | 寻址和路由，把数据从 A 送到 B | 快递单上的地址(IP) |
| **数据链路层** | 相邻节点之间的可靠传输 | 同一条街上门牌号之间的投递(MAC) |
| **物理层** | 比特流的物理传输 | 公路本身 |

## 三、数据封装过程

发送数据时，每一层都会给数据加上自己的头部信息，层层包装：

```
应用层       [HTTP 数据]
              ↓ 加 TCP 头
传输层       [TCP头 | HTTP 数据]          ← 段 (Segment)
              ↓ 加 IP 头
网络层       [IP头 | TCP头 | HTTP 数据]    ← 包 (Packet)
              ↓ 加 MAC 头 + 尾
数据链路层   [MAC头 | IP头 | TCP头 | HTTP 数据 | MAC尾]  ← 帧 (Frame)
              ↓
物理层        010110100111...             ← 比特流 (Bits)
```

接收方则反过来，逐层拆包。

## 四、实际开发中怎么理解

作为 Android 开发者，日常接触的主要是：

```
你写的代码（Retrofit/OkHttp）  →  应用层（HTTP）
                                    ↓
TLS 加密                       →  介于应用层和传输层之间
                                    ↓
系统内核自动处理                  →  传输层（TCP）→ 网络层（IP）→ 往下全自动
```

## 五、面试常见追问

### 为什么实际用 TCP/IP 四层而不是 OSI 七层？

OSI 是先有模型再有实现，偏理论；TCP/IP 是先有实现再总结模型，更贴合实际。OSI 的表示层和会话层在实践中很难独立出来，功能都被合并到了应用层。

### TCP 和 UDP 的区别

| | TCP | UDP |
|--|-----|-----|
| 连接 | 需要三次握手建立连接 | 无连接，直接发 |
| 可靠性 | 可靠（重传、排序、流控） | 不可靠，可能丢包 |
| 速度 | 较慢 | 快 |
| 场景 | HTTP、文件传输 | 视频直播、DNS、游戏 |
