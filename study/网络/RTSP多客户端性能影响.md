# RTSP 多客户端连接性能影响

## 一、先区分：RTSP 只管信令，不管数据

```
  RTSP 服务器（如 IP 摄像头）

  ┌─────────────────────────────────────────────────────┐
  │                    RTSP Server                       │
  │                                                     │
  │  RTSP (TCP 554) ─── 控制面：DESCRIBE/SETUP/PLAY/TEARDOWN │
  │                     只传信令，数据量极小              │
  │                                                     │
  │  RTP  (UDP) ─── 数据面：音视频裸流                   │
  │                  这才是占带宽的大头                   │
  │                                                     │
  │  RTCP (UDP) ─── 控制面补充：丢包反馈/QoS 统计        │
  └─────────────────────────────────────────────────────┘
```

**关键认知：性能瓶颈 99% 在 RTP 数据面上，RTSP 信令面本身开销很低。**

## 二、多客户端的性能影响逐层分析

```
  一个 RTSP 流，N 个客户端同时拉流:

                       ┌─────────────┐
                       │ RTSP Server │  (IP Camera / Media Server)
                       └──────┬──────┘
                              │
              ┌───────────────┼───────────────┐
              │               │               │
              ▼               ▼               ▼
         客户端1           客户端2          客户端3

  每个客户端产生:
    1 个 RTSP TCP 连接  ── 开销很小（信令）
    1 对 RTP 端口 (UDP) ── 数据推流
    1 对 RTCP 端口(UDP) ── QoS 反馈
```

```
┌────────────────────────┬────────────────────────────────────┐
│       影响维度          │              分析                   │
├────────────────────────┼────────────────────────────────────┤
│ 带宽（最关键）          │ 单路 × N = 总带宽                   │
│                        │ 一路 1080p H.264 ≈ 4-8 Mbps        │
│                        │ 10 路 = 40-80 Mbps 上传带宽         │
│                        │ 这是最常见的瓶颈                    │
├────────────────────────┼────────────────────────────────────┤
│ CPU                    │ - 不解码不重新编码: 仅包转发，        │
│                        │   CPU 开销很低                      │
│                        │ - 需要转码: N 倍编码开销             │
│                        │ - 需要复用/解复用: 开销小            │
├────────────────────────┼────────────────────────────────────┤
│ 内存                   │ 每客户端 ≈ 几百KB-几MB 缓冲区        │
│                        │ 10 客户端 ≈ 数十MB，可控            │
├────────────────────────┼────────────────────────────────────┤
│ 端口/文件描述符         │ 每客户端 1 TCP + 2 UDP ≈ 3 fd      │
│                        │ 100 客户端 = 300 fd，远低于限制      │
├────────────────────────┼────────────────────────────────────┤
│ 连接数                  │ RTSP 用 TCP 长连接                  │
│                        │ 系统 fd 上限 > 65535，不是瓶颈       │
└────────────────────────┴────────────────────────────────────┘
```

## 三、带宽 —— 核心瓶颈

```
  单路 RTSP 流的典型码率:

  ┌────────────────┬──────────────────────────────┐
  │     视频规格     │       典型码率 (H.264)        │
  ├────────────────┼──────────────────────────────┤
  │ 720p / 25fps   │ 1.5 - 3 Mbps                 │
  │ 1080p / 25fps  │ 4 - 8 Mbps                   │
  │ 4K / 25fps     │ 15 - 30 Mbps                 │
  └────────────────┴──────────────────────────────┘

  带宽消耗 = 码率 × 客户端数

  1路 1080p = 8Mbps
  10路      = 80Mbps  ← 百兆网卡 100Mbps 基本打满
  20路      = 160Mbps ← 必须千兆网卡

  而且注意:
  - 是服务器上行带宽（多数服务器上行远小于下行）
  - 云服务器带宽很贵（阿里云 100Mbps ≈ 几千元/月）
```

```
  IP Camera 的带宽限制更严重：

  ┌────────────────────────────────────────────────────┐
  │  普通 IPC 同时只能推少数几路                          │
  │                                                    │
  │  海康/大华常见 IPC:                                  │
  │  - 主码流 + 子码流 = 2路同时输出                     │
  │  - 主码流: 1080p 供本地 NVR 录像                     │
  │  - 子码流: 640×480 供远程预览                        │
  │                                                    │
  │  RTSP 直接拉 IPC:                                    │
  │  - 大部分 IPC 限制 ≤ 3-5 路 RTSP 同时连接            │
  │  - 超过后直接拒绝或踢掉旧连接                         │
  │  - 这是设备固件的硬限制，不是协议限制                  │
  └────────────────────────────────────────────────────┘
```

## 四、CPU 影响 —— 取决于是否转码

```
                       场景 A: 纯转发（不解码不编码）

     IPC ─── H.264 码流 ─── 服务器 ─── H.264 码流 ─── 客户端1
                      ──────┼─────── H.264 码流 ─── 客户端2
                            └─────── H.264 码流 ─── 客户端3

    CPU 仅做: 收包 → 复制 → 发给多个目标
    消耗: 极低，纯内存操作
    10 客户端的 CPU 几乎可以忽略


                       场景 B: 转码后分发

     IPC ─── H.264 码流 ─── 服务器 ── 解码 ── 重新编码 ── 客户端1
                                          └── 重新编码 ── 客户端2
                                          └── 重新编码 ── 客户端3

    每多一个客户端 = 多一次完整的 解码+编码
    CPU × N，这是线性甚至超线性增长
```

**如果客户端需要不同分辨率/码率（如手机端要低码率），必须转码。** 此时常用方案：

```
  转码方案 —— 一次转，多次用:

  H.264 源流
     │
     ▼
  解码为 YUV 原始帧
     │
     ├── 编码 1080p H.264 (高清端)
     ├── 编码 720p H.264  (中清端)
     └── 编码 360p H.264  (低清端)
          │
          └── 每个档位编码一次，同一档位的客户端共享编码结果
              不需要每客户端单独编码
```

## 五、网络模型 —— Unicast vs Multicast

```
  Unicast（一对一）:                    Multicast（一对多）:

    Server                              Server
    ╱  │  ╲                                │
   ╱   │   ╲                               ▼  (一份数据)
  ▼    ▼    ▼                         ┌──────────┐
 C1    C2   C3                        │  路由器    │
                                       └────┬─────┘
  带宽 = 单路 × N                     ╱    │    ╲
  每增加一个客户端，                     ▼    ▼    ▼
  服务器上行 + 单路带宽                  C1    C2   C3

  适用: 互联网/广域网                   带宽 = 单路 × 1
                                      服务器只发一份
                                      路由器负责复制

                                      适用: 局域网/VPN
                                      需要路由器和客户端支持IGMP
                                      RTSP 支持 Multicast (Transport: RTP/AVP;multicast)
```

```
  RTSP SETUP 请求中指定传输方式:

  客户端 → 服务器:
    SETUP rtsp://192.168.1.100/stream/track1 RTSP/1.0
    Transport: RTP/AVP;unicast;client_port=8000-8001     ← 单播
    Transport: RTP/AVP;multicast;destination=224.2.3.4   ← 组播 (局域网内)

  实际落地：
  - 互联网上几乎全是 Unicast（公网路由器不支持组播）
  - 局域网内（如监控系统）可以用 Multicast 节省带宽
```

## 六、综合场景分析

### 场景1：10 个客户端直接拉 IPC（直连）

```
  10 Client ── RTSP ──▶ IPC (海康摄像头)

  结果：
  - 大多数 IPC 固件限制 ≤ 5 路 RTSP
  - 到第 6 路就 503 Service Unavailable
  - 即使能连，IPC 的百兆网口上行也扛不住 10×8Mbps=80Mbps
```

### 场景2：10 个客户端拉 Media Server（转发）

```
  10 Client ── RTSP ──▶ Media Server ── RTSP ──▶ IPC
                        (SRS/ZLMediaKit/Janus)

  带宽:
    IPC → Server: 8Mbps (只拉一路)
    Server → 10 Client: 80Mbps (server 要有足够上行带宽)

  CPU:
    纯转发: 低，几乎无感
    转码: 看转多少个档位
```

### 场景3：100 个客户端 + CDN 分发

```
                     IPC
                      │ (一路 RTSP)
                      ▼
              ┌──────────────┐
              │  Media Server │  (源站，只拉一路)
              └──────┬───────┘
                     │ (RTMP/HLS/WebRTC 推流到 CDN)
                     ▼
              ┌──────────────┐
              │   CDN 边缘节点  │
              └──────┬───────┘
          ┌──────────┼──────────┐
          ▼          ▼          ▼
       客户端1     客户端2    客户端100

  源站压力: 仅 1 路
  边缘节点: 分区域分担
  100 客户端的带宽由 CDN 承担
```

## 七、常见性能优化手段

```
┌─────────────────────────────────────────────────────────────┐
│  优化手段                  │  效果                           │
├─────────────────────────────────────────────────────────────┤
│ 子码流 (Sub Stream)       │ IPC 提供低码流给多客户端预览     │
│                           │ 主码流 8Mbps → 子码流 512Kbps    │
│                           │ 带宽降低 16 倍                   │
├───────────────────────────┼─────────────────────────────────┤
│ Media Server 转发         │ 解耦 IPC 和客户端                │
│ (ZLMediaKit / SRS)       │ IPC 只承受 1 路拉流               │
│                           │ Server 扛多路分发                │
├───────────────────────────┼─────────────────────────────────┤
│ 按需转码                  │ 转一次、多路复用                  │
│                           │ 只在有需要时转码                  │
│                           │ 同档位客户端共享编码结果          │
├───────────────────────────┼───────────────────────────────┤
│ Multicast (局域网)         │ 服务器上行 = 单路带宽            │
│                           │ 10 路变 1 路                    │
├───────────────────────────┼───────────────────────────────┤
│ CDN 分发                  │ 大规模场景的终极方案              │
│                           │ 边缘节点就近服务                  │
├───────────────────────────┼───────────────────────────────┤
│ 连接超时 + 无数据踢出      │ 僵尸连接占资源                   │
│                           │ RTSP 默认无心跳，需主动管理       │
├───────────────────────────┼───────────────────────────────┤
│ 限制单 IP 最大连接数       │ 防止单一客户端滥用               │
└───────────────────────────┴─────────────────────────────────┘
```

## 八、面试题

**Q1: RTSP 多客户端连接，主要性能瓶颈在哪？**

> **带宽**是第一瓶颈。RTSP 本身只是信令协议（TCP 控制通道），开销极小。真正的压力在 RTP 数据面：每个客户端都需要独立的 RTP 音视频流，N 个客户端 = N × 码率的服务器上行带宽。其次是 IPC 设备本身的连接数限制（多数 IPC 限制 3-5 路）。CPU 和内存通常不是瓶颈，除非需要转码。

**Q2: RTSP 能支持多少个同时连接？**

> 协议本身没有限制。实际受限于：① IPC 固件限制（通常 3-5 路）② 服务器上行带宽（一路 1080p ≈ 8Mbps，10路 = 80Mbps）③ 若经过 Media Server 转发 + 纯转发，可支撑数百路（瓶颈在网卡带宽）。大规模用 CDN 分发几乎无上限。

**Q3: Multicast 为什么在互联网上不普及？**

> 组播要求网络中所有路由器都支持 IGMP 协议并正确配置组播路由。公网路由器出于安全和计费考虑，基本不开启组播转发。只在局域网（企业内网、监控专网）内可用。互联网上 RTSP 实际上 100% 走 Unicast。

**Q4: Media Server 转发和直接拉 IPC 有什么区别？**

> 直接拉 IPC：每个客户端独立向 IPC 发起 RTSP 连接，IPC 承受全部压力（带宽 × N + 连接数 × N）。经过 Media Server：Server 从 IPC 只拉 1 路，然后分发给 N 个客户端。IPC 的压力降为常量，Server 承担分发负载。Server 可以部署在高带宽服务器上，还可以做转码、录制、转协议（RTSP→HLS/WebRTC）。
