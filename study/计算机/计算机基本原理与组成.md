## 一、CPU 架构

### Q1: CPU 内部由哪几部分组成？

```
┌─ CPU ──────────────────────────────────────────┐
│                                                │
│  ┌─ 控制单元 (CU) ──────────────────────────┐  │
│  │  取指令、译码、控制执行流程                  │  │
│  │  相当于"指挥官"                            │  │
│  └──────────────────────────────────────────┘  │
│                                                │
│  ┌─ 算术逻辑单元 (ALU) ─────────────────────┐  │
│  │  执行加减乘除、与或非、移位等运算            │  │
│  │  相当于"计算器"                            │  │
│  └──────────────────────────────────────────┘  │
│                                                │
│  ┌─ 寄存器组 (Registers) ───────────────────┐  │
│  │  CPU 内部的超高速存储（几十个，每个 64 位）  │  │
│  │  相当于"手边的草稿纸"                      │  │
│  │                                          │  │
│  │  通用寄存器：R0~R30（ARM）/ RAX~R15（x86）│  │
│  │  PC（程序计数器）：下一条指令的地址          │  │
│  │  SP（栈指针）：当前栈顶位置                 │  │
│  │  FP（帧指针）：当前函数栈帧的基地址          │  │
│  │  状态寄存器：零标志、进位标志、溢出标志       │  │
│  └──────────────────────────────────────────┘  │
│                                                │
│  ┌─ Cache（高速缓存）──────────────────────┐  │
│  │  L1 Cache（32~64KB，1~3 周期）           │  │
│  │  L2 Cache（256KB~1MB，10 周期左右）       │  │
│  │  L3 Cache（几 MB~几十 MB，共享，30 周期）  │  │
│  └──────────────────────────────────────────┘  │
│                                                │
└────────────────────────────────────────────────┘
```

### Q2: 一条指令是怎么执行的？（指令流水线）

```
经典五级流水线：

  取指(IF)   译码(ID)   执行(EX)   访存(MEM)  写回(WB)
  ────────  ────────  ────────  ─────────  ────────
  从内存取    解析操作码   ALU 运算   读写内存    结果写回
  出指令      和操作数                           寄存器

流水线并行（5 条指令同时在不同阶段）：

时钟周期  1    2    3    4    5    6    7    8
指令 1   IF → ID → EX → MEM → WB
指令 2        IF → ID → EX → MEM → WB
指令 3             IF → ID → EX → MEM → WB
指令 4                  IF → ID → EX → MEM → WB
指令 5                       IF → ID → EX → MEM → WB

没有流水线：5 条指令要 25 个周期
有流水线：5 条指令只要 9 个周期

类比：洗衣服流水线
  洗衣机洗 → 烘干机烘 → 叠衣服
  不用等第一批全做完才开始第二批
```

### Q3: 什么是分支预测？预测错了怎么办？

```
if (x > 0) {
    doA();  // 分支 A
} else {
    doB();  // 分支 B
}

CPU 流水线执行到 if 时，x > 0 的结果还没算出来
但流水线不能停下来等（一停就浪费好几个周期）

解决方案：分支预测器猜一个方向，提前执行

  预测正确 ✅：
    流水线继续跑，没有浪费

  预测错误 ❌：
    已经提前执行的指令全部作废（Pipeline Flush）
    回到分支点重新执行正确的分支
    浪费 10~20 个周期

和 Android 的关系：
  排序数组上的循环比未排序的快
  因为排序后 if 条件结果更有规律
  分支预测器猜得更准
```

## 二、存储层次

### Q4: 从寄存器到硬盘，速度差多少？

```
┌──────────────────────────────────────────────────────────┐
│                    存储金字塔                              │
│                                                          │
│              ┌───────┐                                   │
│              │寄存器  │  ← 1 周期（~0.3ns）               │
│              │64个×8B │    最快，最贵，最小                  │
│             ─┴───────┴─                                  │
│            ┌───────────┐                                 │
│            │ L1 Cache  │  ← 3 周期（~1ns）                │
│            │  64KB     │    CPU 核心独享                   │
│           ─┴───────────┴─                                │
│          ┌───────────────┐                               │
│          │  L2 Cache     │  ← 10 周期（~3ns）             │
│          │  256KB~1MB    │    CPU 核心独享                 │
│         ─┴───────────────┴─                              │
│        ┌───────────────────┐                             │
│        │   L3 Cache        │  ← 30 周期（~10ns）          │
│        │   几MB~几十MB      │    多核共享                   │
│       ─┴───────────────────┴─                            │
│      ┌───────────────────────┐                           │
│      │      内存 (RAM)        │  ← 100 周期（~100ns）      │
│      │      几GB~十几GB       │    主存                     │
│     ─┴───────────────────────┴─                          │
│    ┌───────────────────────────┐                         │
│    │      SSD / Flash          │  ← ~100,000ns（0.1ms）   │
│    │      几十GB~几百GB         │                          │
│   ─┴───────────────────────────┴─                        │
│  ┌───────────────────────────────┐                       │
│  │      HDD（机械硬盘）           │  ← ~10,000,000ns（10ms）│
│  │      几TB                     │                        │
│  └───────────────────────────────┘                       │
│                                                          │
│  速度差距：                                                │
│  寄存器 vs 内存 ≈ 100 倍                                   │
│  内存 vs SSD ≈ 1000 倍                                    │
│  内存 vs HDD ≈ 100,000 倍                                 │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**和 Android 的关系**：
- SharedPreferences 读写 = 访问 Flash，比内存慢 1000 倍
- MMKV 比 SP 快的原因：mmap 把文件映射到内存，减少系统调用
- Bitmap 占内存大的原因：像素数据直接在 RAM 中，1920×1080×4 = 8MB

### Q5: CPU Cache 是怎么工作的？什么是缓存行（Cache Line）？

```
CPU 不是一个字节一个字节从内存读的
而是一次读一个 Cache Line（通常 64 字节）

内存地址空间：
┌────┬────┬────┬────┬────┬────┬────┬────┬────┐
│ 0  │ 8  │ 16 │ 24 │ 32 │ 40 │ 48 │ 56 │ 64 │ ...
└────┴────┴────┴────┴────┴────┴────┴────┴────┘
│←─── Cache Line 0 (64B) ────→│←── Cache Line 1 ──→│

你访问地址 8 的数据：
  CPU 发现 Cache 里没有（Cache Miss）
  → 从内存读取整个 Cache Line 0（地址 0~63 的 64 字节）
  → 下次访问地址 16 时就在 Cache 里了（Cache Hit）
```

**这就是为什么遍历数组比遍历链表快**：

```
数组（内存连续）：
┌──┬──┬──┬──┬──┬──┬──┬──┐
│a0│a1│a2│a3│a4│a5│a6│a7│  ← 一个 Cache Line 装了 8 个 int
└──┴──┴──┴──┴──┴──┴──┴──┘
  读 a0 时把 a1~a7 也加载了 → 后 7 次都是 Cache Hit ✅

链表（内存分散）：
┌──┐    ┌──┐        ┌──┐          ┌──┐
│n0│──→ │n1│───→    │n2│────→     │n3│
└──┘    └──┘        └──┘          └──┘
  地址 0x100  0x8800  0x2200  0x9900
  每个节点在不同 Cache Line → 每次都 Miss ❌
```

### Q6: 什么是伪共享（False Sharing）？

```
多核 CPU，每个核有自己的 L1 Cache：

Core 0 的 L1 Cache         Core 1 的 L1 Cache
┌──────────────────┐      ┌──────────────────┐
│  Cache Line 拷贝  │      │  Cache Line 拷贝  │
│  [varA] [varB]   │      │  [varA] [varB]   │
└──────────────────┘      └──────────────────┘

如果 varA 和 varB 恰好在同一个 Cache Line 里：

  Core 0 修改 varA → Core 0 的 Cache Line 标记为 Modified
                   → Core 1 的同一 Cache Line 被 Invalidate
                   → Core 1 访问 varB 时被迫从内存重新加载整个 Line

  明明两个核改的是不同变量
  但因为在同一个 Cache Line 里
  导致互相"踩脚" → 性能暴跌

解决方案：Cache Line Padding（填充到 64 字节对齐）
  Java 中 @Contended 注解就是干这个的
```

## 三、内存管理

### Q7: 虚拟内存是怎么回事？为什么每个进程都觉得自己有 4GB（32位）？

```
每个进程看到的（虚拟地址空间）：

进程 A 视角              进程 B 视角
┌──────────────┐       ┌──────────────┐
│ 0xFFFFFFFF   │       │ 0xFFFFFFFF   │
│ 内核空间      │       │ 内核空间      │
├──────────────┤       ├──────────────┤
│              │       │              │
│ 栈           │       │ 栈           │
│   ↓          │       │   ↓          │
│              │       │              │
│   ↑          │       │   ↑          │
│ 堆           │       │ 堆           │
│              │       │              │
│ 数据段       │       │ 数据段       │
│ 代码段       │       │ 代码段       │
│ 0x00000000   │       │ 0x00000000   │
└──────────────┘       └──────────────┘
  各自以为有完整 4GB      互不干扰

实际物理内存只有 8GB，靠 MMU 翻译：

虚拟地址 → MMU（页表翻译）→ 物理地址

进程 A 的 0x1000 ──→ 物理 0x5000
进程 B 的 0x1000 ──→ 物理 0x9000
同一个虚拟地址映射到不同物理地址 → 进程隔离
```

### Q8: 页表和 TLB 是什么？

```
虚拟地址翻译过程：

虚拟地址 = [页号 | 页内偏移]
           例：[0x0010 | 0x3A8]

    虚拟页号 0x0010
        │
        ▼
    先查 TLB（Translation Lookaside Buffer）
    ┌─────────────────────────┐
    │  TLB：页表的缓存          │
    │  容量很小（几十~几百条）    │
    │  速度很快（1~2 周期）      │
    │                         │
    │  0x0010 → 物理页 0x5    │  ← Hit！直接拿到物理页号
    │  0x0020 → 物理页 0x8    │
    └─────────────────────────┘
        │
        ├─ TLB Hit（~98% 的情况）→ 直接拿到物理地址 → 1 周期
        │
        └─ TLB Miss → 查页表（在内存中）→ 可能上百周期
                │
                ▼
            ┌─ 页表（Page Table）──────────┐
            │  虚拟页号 → 物理页号          │
            │  0x0010 → 0x5  (present=1)  │
            │  0x0011 → 0x9  (present=1)  │
            │  0x0012 → ???  (present=0)  │ ← Page Fault!
            └─────────────────────────────┘
                │
                ├─ present=1 → 拿到物理页号，更新 TLB
                └─ present=0 → Page Fault（缺页中断）
                        │
                        ▼
                    OS 从磁盘/Flash 加载数据到物理内存
                    更新页表
                    重新执行这条指令
```

**和 Android 的关系**：
- App 启动时大量 Page Fault（代码和资源要从 Flash 加载到内存）
- 这就是冷启动慢的硬件层面原因
- LMK（Low Memory Killer）杀进程就是为了腾出物理页给前台 App

### Q9: 进程的内存布局长什么样？

```
高地址
┌──────────────────────────┐ 0xFFFFFFFF
│       内核空间             │ ← 用户代码不可访问
├──────────────────────────┤ 0xC0000000 (32位) / 更高 (64位)
│                          │
│       栈 (Stack)          │ ← 向下增长 ↓
│       局部变量、函数参数    │    每个线程有自己的栈
│       返回地址             │    Android 默认线程栈 1MB
│          ↓                │
│                          │
│       （空闲区域）          │
│                          │
│          ↑                │
│       堆 (Heap)           │ ← 向上增长 ↑
│       malloc / new 分配   │    Android 中是 ART 管理的 Java 堆
│       GC 管理              │
│                          │
├──────────────────────────┤
│       内存映射区 (mmap)    │ ← 共享库、文件映射
│       .so 文件            │    Android 的 .dex .oat .so 在这里
│       匿名映射             │
├──────────────────────────┤
│       BSS 段              │ ← 未初始化的全局/静态变量（零填充）
├──────────────────────────┤
│       数据段 (Data)        │ ← 已初始化的全局/静态变量
├──────────────────────────┤
│       代码段 (Text)        │ ← 机器指令，只读 + 可执行
├──────────────────────────┤
│       保留区              │ ← 地址 0 附近，空指针访问这里会 SIGSEGV
└──────────────────────────┘ 0x00000000
低地址
```

## 四、栈与堆

### Q10: 栈和堆的区别？为什么栈比堆快？

```
栈 (Stack)                           堆 (Heap)
──────────                          ──────────
分配方式：移动 SP 指针（1 条指令）      分配方式：malloc 搜索空闲块（复杂算法）
释放方式：函数返回自动弹栈              释放方式：free / GC 回收
速度：极快（1 个周期）                 速度：慢（可能上百周期）
大小：固定（Android 线程栈 ~1MB）      大小：动态增长（受物理内存限制）
碎片：无（后进先出，连续分配）          碎片：有（反复分配释放产生空洞）
线程：每个线程独享一个栈                线程：所有线程共享堆（需要锁）
```

```
函数调用时栈的变化：

void foo() {
    int a = 1;      // 栈上分配
    int b = 2;      // 栈上分配
    bar(a, b);       // 调用 bar
}

void bar(int x, int y) {
    int c = x + y;   // 栈上分配
}

    高地址
    ┌──────────────┐
    │  foo 的栈帧    │
    │  返回地址      │
    │  旧 FP        │
    │  a = 1        │
    │  b = 2        │
    ├──────────────┤ ← foo 的 FP
    │  bar 的栈帧    │
    │  返回地址      │  ← 回到 foo 的下一条指令
    │  旧 FP        │  ← 指向 foo 的 FP
    │  x = 1        │  ← 参数
    │  y = 2        │  ← 参数
    │  c = 3        │  ← 局部变量
    └──────────────┘ ← SP（栈顶）
    低地址

    bar 返回时：SP 直接跳回 foo 的栈帧，c/x/y 瞬间"释放"
    不需要 GC，不需要 free，就是移个指针
```

### Q11: 栈溢出（StackOverflow）是怎么回事？

```
无限递归：
void infinite() {
    infinite();  // 每次调用都在栈上分配一个栈帧
}

栈空间（1MB）
┌──────────────┐
│  栈帧 1       │
│  栈帧 2       │
│  栈帧 3       │
│  ...          │
│  栈帧 N       │
│  栈帧 N+1     │ ← 超出栈空间边界！
├──────────────┤
│  Guard Page   │ ← OS 设置的保护页，不可写
└──────────────┘
  写入 Guard Page → 触发 SIGSEGV → StackOverflowError

Android 中：
  主线程栈 ~8MB
  子线程栈默认 ~1MB
  递归深度太大或局部变量太大 → 栈溢出
```

## 五、中断机制

### Q12: 什么是中断？为什么需要中断？

```
没有中断（轮询方式 ❌）：

  CPU 不停地问："键盘有输入吗？网卡有数据吗？触屏有事件吗？"
  99.99% 的时间答案是"没有"
  CPU 白白浪费在轮询上

有中断（✅）：

  CPU 正在执行你的 App 代码
      │
      │  ← 触屏硬件产生中断信号
      │
      ▼
  CPU 暂停当前代码
  保存现场（寄存器状态压栈）
      │
      ▼
  跳转到中断处理程序
  处理触摸事件
      │
      ▼
  恢复现场（寄存器出栈）
  继续执行 App 代码

类比：
  轮询 = 你每 10 秒看一次手机有没有消息
  中断 = 手机来消息时震动通知你
```

### 中断的种类

```
硬件中断（外部设备触发）：
  ├─ 触屏控制器 → 触摸事件
  ├─ 网卡 → 数据到达（epoll 的底层就是这个）
  ├─ 定时器 → 时钟中断（OS 进程调度的基础）
  └─ 磁盘控制器 → IO 完成

软件中断 / 系统调用（程序主动触发）：
  ├─ syscall 指令 → 进入内核态
  ├─ read() / write() / open() / mmap()
  └─ Android 的 Binder 通信底层也是 ioctl 系统调用

异常（CPU 执行指令时发现的错误）：
  ├─ 缺页异常（Page Fault）→ OS 加载页面
  ├─ 除零异常 → SIGFPE
  ├─ 非法地址访问 → SIGSEGV（NullPointerException 的底层）
  └─ 非法指令 → SIGILL
```

## 六、进程与线程的硬件视角

### Q13: 进程切换时硬件层面发生了什么？

```
进程 A 正在运行
    │
    │  时钟中断到来（每 ~10ms 一次）
    │
    ▼
保存进程 A 的硬件上下文：
  ┌───────────────────────────────┐
  │  所有通用寄存器的值             │
  │  PC（程序计数器）               │
  │  SP（栈指针）                   │
  │  状态寄存器                     │
  │  页表基地址寄存器（CR3 / TTBR） │
  │  → 保存到 A 的 task_struct 中  │
  └───────────────────────────────┘
    │
    ▼
切换页表：
  加载进程 B 的页表基地址
  → TLB 全部失效！（TLB Flush）
  → 这就是进程切换比线程切换贵的主要原因
    │
    ▼
恢复进程 B 的硬件上下文：
  从 B 的 task_struct 中恢复所有寄存器
    │
    ▼
进程 B 从上次暂停的位置继续执行
```

### Q14: 线程切换为什么比进程切换快？

```
进程切换：
  保存/恢复寄存器 + 切换页表 + TLB Flush + Cache 大量 Miss
  代价：~几千~几万 CPU 周期

线程切换（同一进程内）：
  保存/恢复寄存器（每个线程有自己的栈和寄存器组）
  不需要切换页表！（同一进程共享地址空间）
  TLB 不失效！Cache 大部分还有效！
  代价：~几百 CPU 周期

这就是为什么 Android 用多线程而不是多进程做并发：
  线程切换代价 ≈ 进程切换的 1/10 ~ 1/100
```

## 七、字节序与对齐

### Q15: 大端序和小端序是什么？

```
存储 int 值 0x12345678：

大端序（Big-Endian）：高字节在低地址
  地址:  0x00  0x01  0x02  0x03
  数据:  0x12  0x34  0x56  0x78
  网络传输用大端（"网络字节序"）

小端序（Little-Endian）：低字节在低地址
  地址:  0x00  0x01  0x02  0x03
  数据:  0x78  0x56  0x34  0x12
  ARM / x86 / Android 都是小端

和 Android 的关系：
  网络数据用大端 → 收到后要转成小端
  Java 的 ByteBuffer 默认大端
  Native 层操作二进制数据要注意字节序
```

### Q16: 什么是内存对齐？为什么需要？

```
未对齐访问（❌）：
  地址:  0  1  2  3  4  5  6  7
  数据:  [  int A  ][  int B  ]
         ↑ 起始地址 0，对齐 ✅

  地址:  0  1  2  3  4  5  6  7
  数据:     [  int C  ]
            ↑ 起始地址 1，未对齐 ❌

  CPU 一次读 4 字节（按 4 字节边界）
  读 C 需要两次内存访问 + 拼接：
    第一次读 [0~3]，取出字节 1~3
    第二次读 [4~7]，取出字节 4
    拼接成完整的 int → 慢一倍

对齐访问（✅）：
  编译器自动在结构体中插入 padding：

  struct Example {
      char a;     // 1 字节
      // padding  // 3 字节（凑到 4 字节边界）
      int b;      // 4 字节（从地址 4 开始，对齐）
      char c;     // 1 字节
      // padding  // 3 字节（结构体总大小对齐到 4 的倍数）
  };
  // sizeof = 12，不是 6！

和 Android 的关系：
  JVM 自动处理对齐，Java 开发者不用操心
  但 NDK / JNI 写 C/C++ 时要注意结构体对齐
  Parcel 序列化也是 4 字节对齐的
```

## 八、和 Android 开发的关联总结

| 底层原理 | Android 中的体现 |
|---------|-----------------|
| Cache Line | 遍历数组比链表快；数据结构选择影响性能 |
| 虚拟内存 / 页表 | 进程隔离；mmap 文件映射（MMKV 的基础） |
| Page Fault | 冷启动慢（代码页从 Flash 加载到内存） |
| TLB Flush | 进程切换开销大，多线程优于多进程 |
| 栈 vs 堆 | 局部变量在栈上（快），对象在堆上（GC 管理） |
| 中断 | 触摸事件、网络数据到达、定时器、系统调用 |
| 分支预测 | 热路径代码排列影响性能 |
| 字节序 | 网络编程、NDK 二进制操作 |
| 内存对齐 | Parcel 4 字节对齐、NDK 结构体 |
| 存储层次 | SP 慢（Flash）、内存缓存快（RAM）、寄存器最快 |

## 九、面试回答模板

**Q: 为什么遍历数组比链表快？**

> 因为 CPU Cache 的工作方式。CPU 从内存读数据是以 Cache Line（通常 64 字节）为单位的。数组在内存中连续存储，读取第一个元素时会把相邻元素也加载到 Cache 里，后续访问都是 Cache Hit。链表的节点分散在内存各处，每次访问几乎都是 Cache Miss，要去内存读取，比 Cache Hit 慢几十到上百倍。

**Q: 进程切换和线程切换的开销差在哪？**

> 核心差在页表切换和 TLB。进程切换要保存/恢复所有寄存器，还要切换页表基地址寄存器，导致 TLB 全部失效，后续内存访问都要重新查页表，Cache 也大量失效。线程切换只需要保存/恢复寄存器和栈指针，因为同一进程的线程共享地址空间，页表不用换，TLB 和 Cache 大部分仍然有效。所以线程切换的开销大约只有进程切换的 1/10 到 1/100。

**Q: 虚拟内存有什么好处？**

> 三个好处：一是进程隔离，每个进程有独立的虚拟地址空间，A 进程无法访问 B 进程的内存；二是内存超卖，虚拟地址空间可以比物理内存大，不用的页暂时不分配物理内存，按需加载（Page Fault）；三是简化编程，每个进程都从地址 0 开始，链接器和加载器不用关心物理地址。
