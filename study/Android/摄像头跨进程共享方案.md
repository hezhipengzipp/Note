# 摄像头跨进程共享方案（Service + Binder/AIDL）

本文档整理 Android 上一路摄像头同时被多个应用消费的工程方案。方案核心是一个独占 Camera 的 Service + AIDL 暴露订阅接口，车机场景下常用。用于团队知识沉淀和面试答辩参考。

---

## 一、问题场景

Android 的 `CameraDevice` 同一时间**只能被一个 client 持有**（Camera2 的独占语义）。当多个 App / 模块需要同一路摄像头数据时：

- 预览 App 要看画面
- DVR 要录制
- AI 模块要识别
- WebRTC 要远程推流

直接各自打开相机会报错。必须做一层"代理"——一个进程持相机，其他进程问它要。

---

## 二、整体架构

│  CameraHolderService (独立进程或 system 进程)

│  ┌───────────────────────────────────┐

## 三、AIDL 接口设计

### 3.1 主接口 `ICameraProxyService.aidl`

```aidl
package com.example.cameraproxy;

import android.view.Surface;
import com.example.cameraproxy.ICameraCallback;

interface ICameraProxyService {
    // 打开摄像头（幂等，已打开则直接返回 true）
    boolean openCamera(String cameraId, int width, int height, int format);

    // 订阅帧流：传入自己的 Surface + 回调
    // 返回 subscriberId 作为后续操作的句柄
    int subscribe(in Surface surface, ICameraCallback callback);

    // 取消订阅
    void unsubscribe(int subscriberId);

    // 获取支持的参数（分辨率组合、格式等）
    CameraCapabilities getCapabilities(String cameraId);

    // 控制参数（曝光、白平衡等，某个 client 独占）
    boolean requestControl(int subscriberId);
    void setExposure(int subscriberId, int value);
    void releaseControl(int subscriberId);
}
```

### 3.2 回调接口 `ICameraCallback.aidl`

```aidl
interface ICameraCallback {
    void onFrameStart();                  // 首帧到达
    void onError(int code, String msg);   // 相机异常
    void onCameraClosed();                // 被其他订阅者踢掉/系统回收
    void onConfigChanged(int w, int h);   // 分辨率变了
}
```

### 3.3 关键设计取舍

| 字段 | 传什么 | 为什么 |
|------|--------|--------|
| `Surface` | 直接传 | Android 已经帮你实现了跨进程的 Surface（底层 BufferQueue + GraphicBuffer token），这是**零拷贝**的关键。传递帧数据一定不要走 byte[] |
| `ICameraCallback` | 传 Binder 代理 | Service 调用 Client 方法，也是 Binder 反向调用 |
| `subscriberId` | int 句柄 | 不要直接用 Surface 对象做 key，Surface 对象跨进程每次创建新代理，equals 不可靠 |

## 四、Service 核心实现

### 4.1 订阅者数据结构

```kotlin
data class Subscriber(
    val id: Int,
    val callback: ICameraCallback,
    val surface: Surface,
    val pid: Int,
    val deathRecipient: IBinder.DeathRecipient
)

private val subscribers = ConcurrentHashMap<Int, Subscriber>()
private val idGenerator = AtomicInteger(1)
```

### 4.2 订阅实现

```kotlin
override fun subscribe(surface: Surface, callback: ICameraCallback): Int {
    val id = idGenerator.getAndIncrement()
    val pid = Binder.getCallingPid()

    // ★ 监听 client 死亡，自动清理
    val deathRecipient = IBinder.DeathRecipient {
        Log.w(TAG, "Subscriber $id (pid=$pid) died, auto-unsubscribe")
        unsubscribeInternal(id)
    }
    callback.asBinder().linkToDeath(deathRecipient, 0)

    subscribers[id] = Subscriber(id, callback, surface, pid, deathRecipient)

    // ★ 关键：重新配置 CaptureSession
    reconfigureSession()

    return id
}
```

### 4.3 动态重配 CaptureSession（难点）

Camera2 的 `CameraCaptureSession` **不支持动态加减 Surface**，必须销毁重建：

```kotlin
private fun reconfigureSession() {
    val outputs = subscribers.values.map { it.surface }
    if (outputs.isEmpty()) {
        currentSession?.close()
        currentSession = null
        return
    }

    // 1. 关闭旧 session
    currentSession?.close()

    // 2. 创建新 session
    cameraDevice.createCaptureSession(
        outputs,
        object : CameraCaptureSession.StateCallback() {
            override fun onConfigured(session: CameraCaptureSession) {
                currentSession = session
                val builder = cameraDevice.createCaptureRequest(
                    CameraDevice.TEMPLATE_PREVIEW
                )
                outputs.forEach { builder.addTarget(it) }
                session.setRepeatingRequest(builder.build(), null, handler)
                subscribers.values.forEach { it.callback.onFrameStart() }
            }

            override fun onConfigureFailed(session: CameraCaptureSession) {
                subscribers.values.forEach {
                    it.callback.onError(ERROR_CONFIG_FAILED, "...")
                }
            }
        },
        handler
    )
}
```

**问题**：每次有新订阅者都重建 session，会造成**几百毫秒的黑屏**。

### 4.4 优化：SurfaceTexture 中转（关键优化）

```
CameraDevice → CaptureSession → 固定的 [中转 Surface (SurfaceTexture)]
                                         │
                                 GL 渲染线程
                                         │
                        ┌────────────────┼────────────────┐
                        ▼                ▼                ▼
                   Subscriber1 Surface  Subscriber2  Subscriber3
                   (EGLSurface 合成)   (EGLSurface)  (EGLSurface)
```

**原理：**

1. Camera 永远只输出到 **1 个** SurfaceTexture（固定不变）
2. 起一个 GL 线程，把 SurfaceTexture 的纹理 `glDraw` 到每个 Subscriber 的 Surface
3. **新增/删除订阅者只改 GL 线程的输出列表，不动 CaptureSession**

**好处：**

- 没有黑屏
- 订阅者数量无上限（只受 GPU 性能限制）
- 每路订阅者可以独立做 crop / scale / 滤镜

**代价：**

- 多一次 GPU copy（但 GPU fan-out 很便宜）
- 增加 ~2ms 延迟
- 需要管理 GL context / EGL surface

实际车机项目基本都这么做，比直接硬件 fan-out 灵活很多。

## 五、Surface 跨进程的原理

为什么 AIDL 传 `Surface` 不丢帧、不序列化图像数据？

```
App Client                      Camera Service
┌────────────┐                  ┌────────────┐
│  Surface   │                  │  Surface   │
│  ┌──────┐  │                  │  ┌──────┐  │
│  │ Prod │  │                  │  │ Cons │  │
│  └──────┘  │                  │  └──────┘  │
└──────┬─────┘                  └──────┬─────┘
       │                               │
       └──────── BufferQueue ──────────┘
         (SurfaceFlinger 管理)
         [GraphicBuffer × N，ashmem 共享]
```

- `Surface` 是 `BufferQueue` 的生产端/消费端引用
- BufferQueue 由 **SurfaceFlinger 持有**，两端通过 Binder 句柄共享同一个 queue
- Buffer 本体（`GraphicBuffer`）分配在 **ashmem**，fd 通过 Binder 传递
- 生产者（Camera Service）写完一帧，queue 通知消费者（App）GPU 渲染
- **整个路径零拷贝，只有 fd 传递**

所以 AIDL 传 `Surface` = 传一个对 BufferQueue 的共享引用，不是在传像素数据。

## 六、高级问题

### 6.1 多订阅者分辨率 / 格式不一样

**场景**：预览要 1920x1080，AI 识别要 640x480 NV21，录制要 1280x720 H264。

| 方案 | 原理 | 适用 |
|------|------|------|
| A. 最大公约数 | 用 1920x1080 输出，小尺寸由硬件 ImageProcessor 缩放 | 订阅者差异小 |
| B. 多 ImageReader | Camera 同时输出到两个 ImageReader（硬件支持时），分别分发 | 中等灵活性 |
| C. GL 中转缩放 | 所有订阅者经 GL 中转，GL 层为每路做独立 viewport 缩放 | **最灵活**，车机推荐 |

### 6.2 参数冲突处理

曝光、白平衡、对焦等参数**全局唯一**，但多订阅者可能都想设置。

**requestControl 模式：**

```kotlin
private val controlOwner = AtomicReference<Int?>(null)

override fun requestControl(subscriberId: Int): Boolean {
    return controlOwner.compareAndSet(null, subscriberId)
}

override fun setExposure(subscriberId: Int, value: Int) {
    if (controlOwner.get() != subscriberId) {
        throw SecurityException("Not the control owner")
    }
    // 真正设置
}

override fun releaseControl(subscriberId: Int) {
    controlOwner.compareAndSet(subscriberId, null)
}
```

或**优先级模式**：预览 < 录制 < AI < 通话，高优先级抢占低优先级的控制权。

### 6.3 Binder 死亡监听（必备）

Client App 崩溃不会主动 unsubscribe，Service 不清理会：

- SurfaceBufferQueue 一直被 hold，内存泄漏
- CaptureSession 里挂着失效的 Surface，送帧会报错

所以 `linkToDeath` **不是可选项**。

### 6.4 线程模型

Service 实现涉及多个线程：

```
- Binder 线程池（AIDL 调用进来）
- Camera Handler 线程（Camera2 回调）
- GL 渲染线程（SurfaceTexture → 多 Surface 分发）
- 业务处理线程（参数设置、日志等）
```

`subscribers` 跨这几个线程用，必须 `ConcurrentHashMap` 或加锁保护。

### 6.5 权限问题

- Service 自身清单声明 `android.permission.CAMERA`，并 runtime 请求过
- Client App 按业务自己声明（用户感知）
- 实际使用是 Service 打开的，Client 理论上不需要 CAMERA 权限（系统 Camera HAL 的常见做法）
- 注意 Android 12+ 的 **privacy indicator**（绿点）会亮，绕不开
- 系统签名 App 可以额外拿 `android.permission.CAMERA_HEADLESS` 或类似权限免指示器

## 七、常见坑

### 坑 1：CaptureSession 重建时的黑屏

已讲过，方案是 SurfaceTexture 中转。

### 坑 2：Surface 被销毁后继续用

Client 的 SurfaceView 销毁，Service 继续往里写 → `BufferQueue has been abandoned`。

**解决**：`onSurfaceDestroyed` 立刻调 `unsubscribe`，或 Service 端监听 Surface 的 `isValid`。

### 坑 3：前后台切换 Camera 被系统回收

Android 10+ 后台 App 默认不能用相机。Service 要：

- 设 `foreground service type = camera`
- 监听 `CameraDevice.StateCallback.onDisconnected`（被抢占时）
- 失败时走 `onCameraClosed` 通知所有订阅者

### 坑 4：多路高码率编码压 CPU

订阅者里有 2~3 个跑 MediaCodec 编码时：

- 直接把 MediaCodec 的 **InputSurface** 作为订阅 Surface 传过来
- 让 **GPU 直接喂给编码器** —— 不要先拉成 YUV byte[] 再编码，CPU 会爆

### 坑 5：帧率同步问题

多订阅者处理速度不一（如 AI 识别只要 15fps，预览要 30fps）：

- GL 中转层按每个 Subscriber 的 FPS 要求**降采样**
- 不能无脑全转发

## 八、面试答题模板

### 整体架构

> 独占 Camera 的 Service 进程 + AIDL 暴露订阅接口，每个业务方通过 AIDL 注册自己的 Surface 作为消费端。

### 数据流

> 我们没有直接把 N 个 Surface 都加进 CaptureSession，而是做了 **SurfaceTexture 中转**——Camera 固定输出到一个 SurfaceTexture，GL 线程再把纹理分发到所有订阅者的 Surface。这样新增/删除订阅者不用重建 CaptureSession，避免了黑屏。

### 跨进程零拷贝

> 帧数据本身不走 Binder，Surface 跨进程传递的是 BufferQueue 的引用，底层 GraphicBuffer 通过 ashmem 共享 fd，整条链路零拷贝。

### 生命周期

> Binder DeathRecipient 监听 Client 进程死亡自动清理订阅，参数控制用 requestControl 抢占模式避免冲突。

### 性能关键点

> 编码 / 推流类订阅者直接用 MediaCodec 的 InputSurface，让 GPU 数据直通编码器不经过 CPU。

## 九、相关扩展方向

如果后续要深入，可考虑：

- **虚拟相机 HAL**（Android 13+）：让消费方用标准 Camera2 API 打开"假相机"，完全透明
- **v4l2loopback**（Linux 车机）：内核级虚拟 v4l2 设备，兼容第三方 App
- **Ashmem / HardwareBuffer 裸协议**：不走 Surface，自己管理共享内存帧同步，延迟最低但实现复杂
