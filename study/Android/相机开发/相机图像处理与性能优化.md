# 相机图像处理与性能优化

## 一、YUV — 相机的"母语"

Camera 输出的原始格式是 **YUV**，不是 RGB。GPU 渲染需要 RGB，所以必须理解 YUV。

### 1.1 YUV 格式

```
Y = 亮度（灰度图）
U = 蓝色色差 (Cb)
V = 红色色差 (Cr)

人眼对亮度敏感、对色彩不敏感 → UV 可以降采样
```

**常见格式**：

| 格式 | 排列 | 常见场景 |
|------|------|---------|
| **NV21** | YYYY... VUVUVU | Android Camera1 默认 |
| **NV12** | YYYY... UVUVUV | Camera2/MediaCodec/通用 |
| **YUV420P (I420)** | YYYY... UUUU... VVVV | 编码输入标准格式 |
| **YV12** | YYYY... VVVV... UUUU | 老旧格式 |

```
NV21 (YUV420sp) 内存布局：
YYYY YYYY YYYY YYYY     ← W×H 个 Y
VU   VU   VU   VU       ← (W/2 × H/2) 个 VU 交错

I420 (YUV420p) 内存布局：
YYYY YYYY YYYY YYYY     ← W×H 个 Y
UUUU UUUU               ← (W/2 × H/2) 个 U
VVVV VVVV               ← (W/2 × H/2) 个 V

Y 的总大小 = W × H
U/V 的总大小 = W/2 × H/2 × 2 = W × H / 2
一张 YUV420 帧的总大小 = W × H × 1.5 字节
```

### 1.2 YUV → RGB 转换

```glsl
// GPU 端 YUV→RGB 转换片元着色器
// 输入：NV21 格式的两张纹理（Y 平面 + UV 平面）
precision mediump float;
varying vec2 vTexCoord;
uniform sampler2D uTextureY;      // Y 平面纹理
uniform sampler2D uTextureUV;     // UV 平面纹理

void main() {
    float y = texture2D(uTextureY, vTexCoord).r;
    vec2 uv = texture2D(uTextureUV, vTexCoord).rg;

    // YUV → RGB 标准转换矩阵 (BT.601)
    float r = y + 1.402 * (uv.g - 0.5);
    float g = y - 0.344 * (uv.r - 0.5) - 0.714 * (uv.g - 0.5);
    float b = y + 1.772 * (uv.r - 0.5);

    gl_FragColor = vec4(r, g, b, 1.0);
}
```

```java
// CPU 端 YUV→RGB（慢，只适合单帧分析）
public static int[] yuvToRgb(byte[] nv21, int width, int height) {
    int size = width * height;
    int[] rgb = new int[size];

    for (int y = 0; y < height; y++) {
        for (int x = 0; x < width; x++) {
            int yIndex = y * width + x;
            int uvIndex = size + (y / 2) * width + (x / 2) * 2;

            int Y = nv21[yIndex] & 0xFF;
            int U = nv21[uvIndex + 1] & 0xFF;   // V before U in NV21
            int V = nv21[uvIndex] & 0xFF;

            // 转换（BT.601 full range）
            int r = (int)(Y + 1.402 * (V - 128));
            int g = (int)(Y - 0.344 * (U - 128) - 0.714 * (V - 128));
            int b = (int)(Y + 1.772 * (U - 128));

            rgb[yIndex] = 0xFF000000 | clamp(r) << 16 | clamp(g) << 8 | clamp(b);
        }
    }
    return rgb;
}
```

### 1.3 ImageReader — Android 获取 YUV 的正规方式

```kotlin
// Camera2 获取 YUV 帧
val imageReader = ImageReader.newInstance(
    width, height, ImageFormat.YUV_420_888, maxImages
)
imageReader.setOnImageAvailableListener({ reader ->
    val image = reader.acquireLatestImage()
    image?.use {
        // YUV_420_888 是 multi-plane 格式
        val yBuffer = it.planes[0].buffer  // Y plane
        val uBuffer = it.planes[1].buffer  // U plane
        val vBuffer = it.planes[2].buffer  // V plane

        // pixelStride 可能是 1（紧凑）或 2（半字节对齐）
        val yPixelStride = it.planes[0].pixelStride
        val yRowStride = it.planes[0].rowStride

        // ★ 注意：rowStride 可能大于 width（对齐原因）
        // 所以要逐行 copy，不能直接 bulk read
    }
}, handler)
```

**面试坑**：`YUV_420_888` 是 **generic** 格式，不一定是 NV21 或 I420。每个 plane 的 pixelStride 和 rowStride 都要检查。

## 二、CPU vs GPU 分工

```
Camera 原始帧 (YUV)
       │
       ├──→ CPU: AI 识别（人脸检测、二维码、场景识别）
       │         需要逐像素分析，CPU 更灵活
       │
       └──→ GPU: 渲染 + 滤镜 + 特效（OpenGL）
                 需要并行处理大量像素，GPU 更快

       ──→ 编码器: MediaCodec（硬件编码 H264/H265）
```

### 什么场景用 CPU 处理？

| 场景 | 原因 | 方案 |
|------|------|------|
| 人脸关键点 | 需要像素级分析 | CPU + ML Kit |
| 二维码识别 | 非图形计算 | CPU + MLKit / ZXing |
| 直方图统计 | 需要回读数据 | CPU 或 GPU 回读 |
| 保存 JPEG | 需要编码 | JPEG 编码器（CPU/硬件） |

### 什么场景用 GPU？

| 场景 | 原因 |
|------|------|
| 滤镜渲染 | 每像素独立计算，天然并行 |
| 缩放/旋转 | GPU 硬件插值 |
| 颜色空间转换 | 每像素矩阵乘 |
| 特效叠加 | 纹理混合 (Blend) |

**黄金法则**：别把 GPU 里的数据读回 CPU（glReadPixels 非常贵），全程在 GPU 管线内完成。

## 三、Camera2 + OpenGL 完整管线

```
Camera2
  │ request
  ▼
CameraCaptureSession
  │ preview
  ▼
SurfaceTexture  ──→  GL Texture (GPU)
  │ updateTexImage()
  ▼
片元着色器 (滤镜)   ←── Uniform 参数
  │
  ├──→ FBO → EGLSurface → SurfaceView/TextureView（屏幕预览）
  │
  └──→ MediaCodec InputSurface（录制/推流）
                      │
                      ▼
                  H264/H265 编码器
```

### 完整代码骨架

```kotlin
class CameraGLRenderer : GLThread() {

    private var camera: CameraDevice? = null
    private var surfaceTexture: SurfaceTexture? = null
    private var textureId = 0
    private var program = 0

    fun startCamera() {
        // 1. 在 GL 线程创建纹理
        handler.post {
            textureId = createTexture()
            surfaceTexture = SurfaceTexture(textureId)
            surfaceTexture?.setDefaultBufferSize(previewW, previewH)

            // 2. 打开 Camera
            val manager = context.getSystemService(CAMERA_SERVICE)
                as CameraManager
            manager.openCamera(cameraId, object : CameraDevice.StateCallback() {
                override fun onOpened(device: CameraDevice) {
                    camera = device
                    val session = device.createCaptureSession(
                        listOf(Surface(surfaceTexture)),
                        callback, null
                    )
                    val request = device.createCaptureRequest(
                        CameraDevice.TEMPLATE_PREVIEW
                    )
                    request.addTarget(Surface(surfaceTexture))
                    session.setRepeatingRequest(request.build(), null, null)
                }
            }, null)
        }
    }

    // 每帧渲染循环
    fun onDrawFrame() {
        // 更新纹理（取最新帧）
        surfaceTexture?.updateTexImage()

        // 获取变换矩阵
        val matrix = FloatArray(16)
        surfaceTexture?.getTransformMatrix(matrix)

        // 使用着色器渲染
        GLES20.glUseProgram(program)

        // 绑定纹理
        GLES20.glActiveTexture(GLES20.GL_TEXTURE0)
        GLES20.glBindTexture(GLES20.GL_TEXTURE_2D, textureId)

        // 设置矩阵 uniform
        GLES20.glUniformMatrix4fv(uTexMatrixLoc, 1, false, matrix, 0)

        // 画全屏矩形
        GLES20.glDrawArrays(GLES20.GL_TRIANGLE_STRIP, 0, 4)

        // 交换缓冲区
        eglSwapBuffers(display, surface)
    }
}
```

## 四、性能优化关键技术

### 4.1 GPU 回读（ReadPixels）有多贵

```java
// ★ 极其昂贵！不要每帧调用！
ByteBuffer buffer = ByteBuffer.allocateDirect(width * height * 4);
GLES20.glReadPixels(0, 0, width, height, GL_RGBA, GL_UNSIGNED_BYTE, buffer);
```

**glReadPixels 导致**：
1. GPU 等待当前帧全部渲染完成（**CPU-GPU 同步**）
2. 像素数据从 GPU VRAM → 系统内存（通过总线）
3. 整帧管线**流水线被冲刷 (Pipeline Flush)**

**替代方案**：
- 需要保存：用 **MediaCodec** 编码，不走 CPU
- 需要分析：用 **Compute Shader / RenderScript** 在 GPU 上做完回读结果（一个 int）
- 真需要回读 → 设置 **PBO (Pixel Buffer Object)** 异步读取

### 4.2 纹理上传优化

| 方式 | 性能 | 说明 |
|------|------|------|
| `glTexImage2D` | 慢 | 重新分配 + 上传全部像素 |
| `glTexSubImage2D` | 中等 | 不重新分配，只更新区域 |
| `glEGLImageTargetTexture2DOES` | **最快** | EGL 直接绑定外部图像，零拷贝 |
| `SurfaceTexture.updateTexImage` | 最快 | 底层就是 EGLImage + BufferQueue |

**Camera 预览已经在用最快的路径**（SurfaceTexture），不需要额外优化。

### 4.3 降低 GPU 负载的技巧

```
1. 降分辨率渲染
   滤镜在 1/2 或 1/4 分辨率 FBO 上跑 → upscale 到全屏

2. 减少纹理采样次数
   双边滤波 7x7 = 49 次 → 分离成水平+垂直 = 14 次

3. 减少 FBO 切换
   glBindFramebuffer 会触发渲染 flush

4. 避免 glFinish / glReadPixels
   流水线等待 kill 性能

5. Shader 精度优化
   能 mediump 就别 highp
   避免分支（if/else 在 GPU 上很慢）

6. 控制帧率
   预览 30fps 够了，不需要跑 60fps 浪费 GPU
```

### 4.4 内存关键指标

```
1080p 一帧：
  NV21     = 1920 × 1080 × 1.5 = 3.1 MB
  RGBA8888 = 1920 × 1080 × 4   = 8.3 MB

ImageReader 的 maxImages = 3 → ~9.3 MB YUV buffer
GPU 纹理 = 8.3 MB
FBO 纹理 = 8.3 MB × N 个滤镜

实测：3 级滤镜链 ≈ 8.3 × 4 = 33 MB GPU 内存
```

## 五、OpenGL 与 MediaCodec 联动

**录制 / 推流的正确做法**：把滤镜后的画面直接传给编码器。

```kotlin
// 1. 创建 MediaCodec 编码器
val codec = MediaCodec.createEncoderByType(MediaFormat.MIME_TYPE_AVC)
val format = MediaFormat.createVideoFormat(
    MIME_TYPE_AVC, width, height
).apply {
    setInteger(MediaFormat.KEY_BIT_RATE, 2_000_000)
    setInteger(MediaFormat.KEY_FRAME_RATE, 30)
    setInteger(MediaFormat.KEY_COLOR_FORMAT,
        MediaCodecInfo.CodecCapabilities.COLOR_TColorFormat)
    setInteger(MediaFormat.KEY_I_FRAME_INTERVAL, 1)
}
codec.configure(format, null, null, MediaCodec.CONFIGURE_FLAG_ENCODE)

// 2. 获取 InputSurface（编码器的输入）
val inputSurface = codec.createInputSurface()

// ★ 编码器的 InputSurface 可以直接作为 EGLSurface 渲染目标！

// 3. 渲染时，同时画到两个 Surface
eglSwapBuffers(display, displaySurface)    // 屏幕显示
eglSwapBuffers(display, recordSurface)     // 编码器输入
// 或者用 FBO 一次渲染 + 分别 blit 到两个目标
```

**面试重点**：MediaCodec 的 InputSurface 就是一个 Surface，可以和 OpenGL 打通。**关键是不经过 CPU**。

## 六、Android 相机相关高阶话题

### 6.1 Camera1 vs Camera2

| | Camera1 | Camera2 |
|---|---|---|
| 架构 | 单一自旋锁 | 状态机 + 管道 |
| 帧回调 | `onPreviewFrame` byte[] | `ImageReader` Surface |
| 手动控制 | 有限 | 完全手动（曝光、对焦、ISO） |
| 输出格式 | NV21 默认 | YUV_420_888 + Surface |
| 多路输出 | ❌ | ✅ 多个 Surface |

### 6.2 前置镜像翻转

Camera 前置镜头默认是**镜像**的。用户看到的自己应该是镜像的。

```glsl
// 顶点着色器中翻转纹理坐标 x
vTexCoord = vec2(1.0 - aTexCoord.x, aTexCoord.y);
```

### 6.3 追焦 (Focus Tracking)

```
Motion Photo / 实时追焦方案：
1. Camera2 输出 YUV 帧
2. CPU 跑人脸检测（ML Kit）
3. 检测到人脸区域 → 调用 Camera2 的 AF 区域
4. 用户点击屏幕 → 转换坐标 → AE/AF 区域

坐标转换：
    触摸点 (viewX, viewY)
    → 转 NDC = (viewX / viewW * 2 - 1, 1 - viewY / viewH * 2)
    → 转 Camera Sensor 坐标（矩阵变换）
    → 设置 MeteringRectangle
```

### 6.4 HDR / 多帧合成

```
算法流程：
1. Camera2 输出多帧（不同曝光：欠曝 → 正常 → 过曝）
2. 对齐（光流法 / 特征匹配）
3. 融合（每像素选最佳曝光区域）
4. 色调映射（Tone Mapping）

OpenGL 方案：每帧以纹理传入，片元着色器做融合
性能：主要卡在对齐步骤，一般用 HW JPEG 或 NPU 加速
```

## 七、面试高频题

### Q1：Camera + OpenGL 数据流中，怎么做到不卡顿？

> 关键在于**不阻塞管线**：
> 1. Camera 帧通过 SurfaceTexture 直接到 GPU，不经过 CPU
> 2. OpenGL 渲染在独立 GL 线程，不与 UI 线程争抢
> 3. 用 `acquireLatestImage()` 而不是 `acquireNextImage()`，丢帧也不堵
> 4. 滤镜链不要太长（2~3 级 FBO 足够了）
> 5. 低端机做降采样渲染 + 降低帧率

### Q2：YUV420 为什么比 RGB 省内存？

> RGB888 每个像素 3 字节，YUV420 的 UV 降采样到 1/4，
> 相当于 Y 占 1 字节/像素，UV 共占 0.5 字节/像素，
> 总共 **1.5 字节/像素**，比 RGB 省一半。

### Q3：怎么在 GPU 上做 YUV→RGB，效率最高？

> 两种方案：
> 1. 把 Y 和 UV 分别作为两个纹理传到 GPU，shader 里做转换 → 每次多一次纹理采样
> 2. 用 **EGLImage** 直接绑定 Camera 的 GraphicBuffer，通过 **external texture** 采样
> 方案 2 零拷贝但需要 `GLES11Ext.GL_TEXTURE_EXTERNAL_OES`

### Q4：SurfaceTexture 的 updateTexImage() 是怎么工作的？

> 每次调用 updateTexImage() 会把**老纹理解绑**，从 BufferQueue 的队首 dequeue 一个新 buffer，绑定为当前纹理的 EGLImage。所以：
> - 每次 updateTexImage 后纹理内容是新帧
> - 不调用则纹理保持上一帧（但 buffer 会积压）
> - 丢帧：如果 BufferQueue 满了（比如渲染太慢），Consumer 会跳过旧 buffer 取最新的

### Q5：OpenGL 线程模型怎么设计最好？

```
推荐模型：
┌─────────────────┐
│  Camera 回调线程  │   ImageReader / CameraCallback
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  命令队列         │   ConcurrentLinkedQueue<Runnable>
│  (无锁/轻量)      │   主线程 post 参数更新
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  GL 渲染线程      │   EGL Context 绑定在此
│  (唯一)          │   - updateTexImage
│                 │   - 滤镜渲染
│                 │   - eglSwapBuffers
└────────┬────────┘
         │
    ┌────┴────┐
    ▼         ▼
 屏幕显示    编码录制
```

**核心原则**：EGL Context 始终绑定在同一个线程，不切不共享。

### Q6：怎么排查相机预览卡顿？

> 1. **SurfaceTexture 丢帧**：用 `Choreographer` 检查回调间隔
> 2. **GPU 渲染超时**：用 GPU profiler / systrace 看 GL 命令耗时
> 3. **Camera 帧率不足**：检查光线是否够、帧率是否设太低
> 4. **BufferQueue 满了**：消费者处理太慢，检查 FBO 链长度
> 5. **GC**：每帧 new 对象会 GC，所有 Buffer 要复用

### Q7：说说你对 OpenGL 纹理的理解

> 纹理是 GPU 上的图像对象，通过纹理 ID 引用。
> 纹理可以绑定到纹理单元（最多 32 个），片元着色器通过 sampler2D 采样。
> SurfaceTexture 是 Android 对 OpenGL 纹理的扩展（GL_TEXTURE_EXTERNAL_OES），
> 允许从 BufferQueue 直接更新纹理内容，是 Camera→GPU 零拷贝的关键。
