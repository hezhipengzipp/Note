# Android 卡顿自动检测方案详解

## 一、卡顿定义

```
   理想 60 FPS = 每帧 16.67ms
                     ↓
   ┌──────────────────────────────────┐
   │  实际帧时间                       │
   ├──────────────────────────────────┤
   │  < 16.67ms  → 流畅               │
   │  > 16.67ms  → 单帧丢帧（jank）    │
   │  > 32ms     → 用户能感知卡顿     │
   │  > 100ms    → 明显卡顿（lag）    │
   │  > 5s       → ANR                │
   └──────────────────────────────────┘
```

**卡顿本质**：主线程没能在一个 Vsync 周期内完成处理。

---

## 二、五大检测方案对比

```
     卡顿检测方案
          │
   ┌──────┼──────┬─────────┬──────────┐
   │      │      │         │          │
Looper  Vsync  WatchDog  字节码      ANR
Printer Frame  喂狗      插桩         FileObserver
   │      │      │         │          │
轻量    简单   ANR级     生产级      系统级
精度低  无栈   主线程死   方法栈      ANR
```

| 方案                | 精度 | 性能 | 能拿堆栈 | 代表实现        |
| ------------------- | ---- | ---- | -------- | --------------- |
| Looper Printer      | 中   | 高   | ✅        | BlockCanary     |
| Choreographer Frame | 中   | 极高 | ❌        | JankStats       |
| WatchDog            | 低   | 高   | ✅        | ANR-WatchDog    |
| 字节码插桩 + Vsync  | 高   | 中   | ✅ 完整栈 | Matrix Trace    |
| ANR FileObserver    | 系统级 | 低 | ✅ trace  | xCrash/Probe    |

## 三、方案 1：Looper Printer（BlockCanary）

### 原理

`Looper.loop()` 在每条消息分发前后调用 `Printer.println`，注入自定义 Printer 即可拦截：

```java
// Looper 源码
public static void loop() {
    for (;;) {
        Message msg = queue.next();
        if (logging != null) {
            logging.println(">>>>> Dispatching to " + msg);  // ① 开始
        }
        msg.target.dispatchMessage(msg);
        if (logging != null) {
            logging.println("<<<<< Finished to " + msg);     // ② 结束
        }
    }
}
```

### 实现

```kotlin
class BlockMonitor(private val threshold: Long = 500) {

    fun start() {
        Looper.getMainLooper().setMessageLogging(object : Printer {
            private var startTime = 0L
            private var startStack = ""

            override fun println(x: String) {
                if (x.startsWith(">>>>> Dispatching")) {
                    startTime = SystemClock.elapsedRealtime()
                    // 子线程定时 dump 主线程堆栈
                    HandlerThread.scheduleStackDump()
                } else if (x.startsWith("<<<<< Finished")) {
                    val cost = SystemClock.elapsedRealtime() - startTime
                    if (cost > threshold) {
                        reportBlock(cost, getMainThreadStack())
                    }
                }
            }
        })
    }
}
```

**配套**：子线程用固定频率（如每 50ms）dump 一次主线程栈，卡顿发生时取最近一次堆栈。

### 优点
- 实现简单，零侵入
- 无需字节码插桩

### 缺点
- 只能监控**Looper 消息**，异步消息（postAtFrontOfQueue/同步屏障）漏检
- 堆栈是采样的，**不一定是真正卡的方法**
- println 字符串拼接本身有微小开销

## 四、方案 2：Choreographer FrameCallback（JankStats）

### 原理

`Choreographer` 在每个 Vsync 信号回调 `FrameCallback.doFrame(long frameTimeNanos)`，可以测两次回调的时间差：

```
Vsync   Vsync   Vsync   Vsync
  │       │       │       │
  ▼       ▼       ▼       ▼
doFrame doFrame doFrame doFrame
  │← 16ms →│← 16ms →│← 50ms (丢2帧!)
```

### 实现

```kotlin
class FrameMonitor : Choreographer.FrameCallback {
    private var lastFrameTime = 0L

    fun start() = Choreographer.getInstance().postFrameCallback(this)

    override fun doFrame(frameTimeNanos: Long) {
        if (lastFrameTime != 0L) {
            val gap = (frameTimeNanos - lastFrameTime) / 1_000_000  // ms
            if (gap > 16.67 * 2) {
                val drops = (gap / 16.67).toInt() - 1
                Log.w("Frame", "丢 $drops 帧")
            }
        }
        lastFrameTime = frameTimeNanos
        Choreographer.getInstance().postFrameCallback(this)  // 持续监听
    }
}
```

### 进阶：JankStats（Jetpack）

Google 官方方案，封装了帧时间统计 + Activity 生命周期感知：

```kotlin
val jankStats = JankStats.createAndTrack(window) { frameData ->
    if (frameData.isJank) {
        report(frameData)   // 包含 jank duration、UI 状态
    }
}
```

### 优点
- 极轻量，零拷贝（Vsync 系统级回调）
- 准确反映用户感知

### 缺点
- **拿不到具体哪个方法卡的**，需配合堆栈采样
- 异步绘制（如 SurfaceView）覆盖不全

## 五、方案 3：WatchDog 喂狗（ANR-WatchDog）

### 原理

子线程定时 post 一个 Runnable 到主线程，主线程执行就是"喂狗"，超时未响应说明卡死。

```
   监控线程              主线程
       │                   │
       │── post(食物) ───▶ │
       │                   │ 队列里执行
       │ ◀── 食物消失 ───  │  喂到
       │                   │
       │ 等待 N 秒          │
       │     │             │
       │     ▼             │
       │  没响应？           │
       │     │             │
       │     ▼             │
       │  dump 主线程堆栈    │
       │  上报"假死"        │
```

### 实现

```kotlin
class WatchDog(private val timeout: Long = 5000) : Thread() {
    private val mainHandler = Handler(Looper.getMainLooper())
    @Volatile private var tick = 0

    override fun run() {
        while (!isInterrupted) {
            val expected = tick + 1
            mainHandler.post { tick = expected }
            try { sleep(timeout) } catch (e: InterruptedException) { return }
            if (tick != expected) {
                // 主线程没响应
                val stack = Looper.getMainLooper().thread.stackTrace
                report("ANR risk", stack)
            }
        }
    }
}
```

### 优点
- 适合检测**真正的死锁/ANR**（5s+ 卡死）
- 实现极简

### 缺点
- 时间粒度大，普通卡顿（< 1s）漏检
- 主线程已卡死时 dump 的堆栈才是真正问题

## 六、方案 4：字节码插桩 + Vsync（Matrix 方案）

**生产级最佳方案**，腾讯、字节都在用。

### 原理

```
编译期 ASM 插桩：
  每个方法 enter/exit 写入 long[]
  long 编码：[methodId(20bit)][isStart(1bit)][timestamp(43bit)]

运行时：
  环形缓冲区记录最近 N 条方法调用
       ↓
  Choreographer 监听 doFrame 间隔
       ↓
  超过阈值 → 从缓冲区还原方法调用栈
       ↓
  上报：完整执行链路 + 各方法耗时
```

### 编译期插桩（ASM）

```java
// 每个方法插入
class MyActivity {
    void onCreate() {
        MethodTracer.i(METHOD_ID_1);   // 入口
        try {
            // 原方法体
        } finally {
            MethodTracer.o(METHOD_ID_1); // 出口
        }
    }
}
```

### 运行时还原

```kotlin
class MethodTracer {
    private val buffer = LongArray(BUFFER_SIZE)  // 环形缓冲
    private var index = 0

    @JvmStatic
    fun i(id: Int) {
        buffer[index] = encode(id, true, SystemClock.elapsedRealtime())
        index = (index + 1) % BUFFER_SIZE
    }

    @JvmStatic
    fun o(id: Int) {
        buffer[index] = encode(id, false, SystemClock.elapsedRealtime())
        index = (index + 1) % BUFFER_SIZE
    }

    fun dumpOnJank() {
        val stack = restoreStack(buffer)  // 还原调用栈
        upload(stack)
    }
}
```

### 优点
- **拿到完整方法栈 + 每个方法耗时**
- 性能开销极小（一次 long 写入）
- 线上可用

### 缺点
- 接入成本高（要写 Gradle 插件）
- 可能产生大量数据，需要后端聚合

## 七、方案 5：ANR 监听（FileObserver）

### 原理

ANR 发生时系统会写 `/data/anr/traces.txt`，监听这个文件就能在 ANR 时拿到完整 trace。

```kotlin
val observer = object : FileObserver("/data/anr/", CLOSE_WRITE) {
    override fun onEvent(event: Int, path: String?) {
        if (path?.contains("trace") == true) {
            val anrFile = File("/data/anr/$path")
            uploadAnrTrace(anrFile)
        }
    }
}
observer.startWatching()
```

**Android 7.0+ 限制**：普通应用没有 `/data/anr/` 读权限。

### 替代方案

- **Signal 拦截**（xCrash/KSCrash 思路）：监听 SIGQUIT 信号，主动 dump trace
- **ApplicationExitInfo（API 30+）**：系统提供死亡原因 + trace

```kotlin
val ams = getSystemService(ActivityManager::class.java)
val exitInfos = ams.getHistoricalProcessExitReasons(packageName, 0, 5)
exitInfos.forEach { info ->
    if (info.reason == REASON_ANR) {
        val trace = info.traceInputStream?.bufferedReader()?.readText()
        upload(trace)
    }
}
```

## 八、组合方案（生产推荐）

```
     启动监控
        │
        ├─ Choreographer FrameCallback     → 实时丢帧统计
        │       ↓ 超阈值
        │   触发字节码栈 dump
        │
        ├─ WatchDog (5s 超时)             → ANR 预警
        │       ↓
        │   主线程 dump
        │
        ├─ ApplicationExitInfo            → 上次启动 ANR 详情
        │
        └─ 关键路径 Trace.beginSection    → 业务耗时归因
```

**采样上报**：
- 不全量上报每次卡顿（数据爆炸）
- 按堆栈相似度聚合 → 同一个堆栈一天上报一次
- 区分 P50/P90/P99 卡顿

## 九、关键指标

| 指标          | 含义                              | 阈值              |
| ------------- | --------------------------------- | ----------------- |
| FPS           | 每秒帧数                          | < 50 → 卡         |
| Janky frames  | 单帧 > 16.67ms 的占比             | > 5% 警惕         |
| Frozen frames | 单帧 > 700ms                      | 任何一帧都要查    |
| ANR rate      | ANR 次数 / 启动次数                | < 0.1%            |
| Slow render   | 渲染慢的帧                        | Play Vitals 指标  |

## 十、面试题

### Q1：为什么 BlockCanary 在生产不够用？

1. **只监控同步消息**：异步消息（postAtFrontOfQueue、同步屏障 SyncBarrier）漏检
2. **堆栈是采样的**：可能采到的是空闲方法，不是真正卡的
3. **println 字符串拼接有开销**：高频消息时累积可观
4. **看不到方法间嵌套调用**：只知道整个 dispatchMessage 慢，定位不到具体方法

生产推荐 Matrix 方案（字节码插桩 + 环形缓冲）。

### Q2：Choreographer 怎么判断丢帧？

`postFrameCallback` 在每个 Vsync 触发，记录两次 `doFrame` 的时间差：
- 间隔 ≤ 16.67ms：正常
- 间隔 > 16.67ms：丢了 `(gap / 16.67) - 1` 帧

可以连续记录帧间隔，统计丢帧分布。

### Q3：Matrix Trace Canary 怎么拿到完整方法栈？

1. **编译期 ASM 插桩**：每个方法入口出口写入一个 long（包含 method id + 时间戳 + start/end 标记）
2. **环形缓冲**：限制大小，避免 OOM
3. **卡顿触发时**：从缓冲区按时间倒序还原调用栈
4. **method id → 方法名映射表**：编译期生成，运行时本地或上报后查表

整个过程零反射、零字符串操作，性能开销极小。

### Q4：WatchDog 方案怎么避免误报？

1. **超时阈值合理**（一般 5s，对应 ANR）
2. **多次确认**：连续 N 次未喂狗才上报
3. **过滤系统场景**：刚启动、切换前后台、CPU 满载时不报
4. **dump 时再做一次检查**：dump 前再 post 一次确认

### Q5：ANR 和卡顿的边界？

- **卡顿**：单帧或几帧延迟，用户能感知不流畅
- **ANR**：主线程持续无响应**超过 5 秒**（Activity）/ 10 秒（Broadcast），系统弹出"应用无响应"对话框

ANR 是卡顿的极端情况，监控方案不同：卡顿用帧/Looper，ANR 用 WatchDog 或 ApplicationExitInfo。

### Q6：怎么检测主线程的"内部死锁"？

WatchDog 是最有效的方法。如果主线程持有锁 A 等待锁 B，子线程持有 B 等 A，主线程永远响应不了，喂狗超时即可发现。然后 dump 所有线程栈分析锁持有情况。

### Q7：ApplicationExitInfo 比 FileObserver 好在哪？

- **不需要 root 权限**
- **拿到的不只是 ANR**：还有 OOM、被杀、用户主动退出等所有死亡原因
- **trace 是裁剪过的**，体积小
- **Android 11+ 官方 API**，稳定性高

老方案 FileObserver 在 Android 7+ 已经基本不可用。

### Q8：怎么定位用户场景下的偶发卡顿？

1. 字节码插桩 + 环形缓冲（Matrix）
2. 用户操作埋点（点击、滑动）
3. 卡顿发生时上报：
   - 操作上下文（哪个页面、什么操作）
   - 设备信息（机型、内存、CPU 占用）
   - 主线程栈
   - 业务自定义 Trace.beginSection
4. 后端按堆栈聚合，找高频问题

### Q9：JankStats 比手写 Choreographer 监听强在哪？

- 自动感知 Activity 生命周期，切后台暂停监听
- 提供 `FrameData`，含 jank 时长、UI 状态、自定义状态（如当前页面）
- 默认过滤动画干扰
- 支持采样模式
- Google 官方维护，兼容性好

### Q10：卡顿优化的经典套路？

1. **检测**：上面的方案，定位高频卡顿点
2. **归因**：是 IO？锁？布局？GC？
3. **优化**：
   - 主线程 IO → 切子线程
   - 锁 → 减小临界区
   - 布局 → 扁平化、ViewStub
   - GC → 减少对象分配
4. **回归**：上线后看指标曲线，确认改善
