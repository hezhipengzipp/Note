# Android 启动耗时统计详解

## 一、启动类型

Android 应用启动分为三种类型，统计口径不同：

| 类型       | 英文         | 触发场景                                    | 耗时 | 关键路径                          |
| ---------- | ------------ | ------------------------------------------- | ---- | --------------------------------- |
| 冷启动     | Cold Start   | 进程不存在，系统从 0 创建                   | 最长 | 进程创建 → Application → Activity |
| 温启动     | Warm Start   | 进程存在，Activity 被销毁                   | 中等 | 重建 Activity                     |
| 热启动     | Hot Start    | 进程、Activity 都存在，从后台切回前台       | 最短 | onResume → 重绘                   |

**面试重点是冷启动**，因为它包含完整的启动流程。

## 二、冷启动全流程与耗时分解

```
用户点击桌面图标
       │
       ▼
┌──────────────────────┐  ← Launcher(SystemServer) 发起启动请求
│ Zygote.fork          │  [系统耗时，不可控]
└──────────────────────┘
       │
       ▼
┌──────────────────────┐  ← 应用进程创建
│ ActivityThread.main  │
└──────────────────────┘
       │
       ▼
┌──────────────────────┐
│ bindApplication      │  ← attachBaseContext
│ Application.onCreate │  [可控：SDK 初始化、全局对象]
└──────────────────────┘
       │
       ▼
┌──────────────────────┐
│ Activity.onCreate    │  [可控：setContentView、数据加载]
│ Activity.onStart     │
│ Activity.onResume    │
└──────────────────────┘
       │
       ▼
┌──────────────────────┐
│ ViewRootImpl         │  ← 测量/布局/绘制
│ measure/layout/draw  │  [可控：布局层级、过度绘制]
└──────────────────────┘
       │
       ▼
┌──────────────────────┐
│ 首帧上屏 (首次显示)  │  ← reportFullyDrawn 或首帧 doFrame
└──────────────────────┘
```

**关键耗时点**：
1. **T1 = 点击图标 → Application.attachBaseContext**：系统层，难优化
2. **T2 = Application 全生命周期**：SDK 初始化、DEX 加载
3. **T3 = Activity.onCreate → onResume**：布局加载、数据准备
4. **T4 = onResume → 首帧绘制**：View 测量布局绘制

## 三、统计方案

### 方案一：adb 命令（面试常问）

#### 1. `adb shell am start -W`

最常用，直接输出三个时间：

```bash
adb shell am start -W com.example.app/.MainActivity
```

输出：

```
Starting: Intent { cmp=com.example.app/.MainActivity }
Status: ok
Activity: com.example.app/.MainActivity
ThisTime: 820        ← 最后一个 Activity 启动耗时
TotalTime: 820       ← 从启动开始到目标 Activity 首帧耗时(推荐使用)
WaitTime: 845        ← AMS 等到的总耗时(包含系统切换)
```

**三者区别**：
- `ThisTime`：最后一个 Activity（通常是入口 Activity）的启动耗时
- `TotalTime`：整个应用启动耗时，**自测启动速度优先看这个**
- `WaitTime`：AMS 角度的耗时，包含前一个 Activity 的 onPause

#### 2. Logcat 过滤 `Displayed`

Android 4.4+ 系统自动打印：

```bash
adb logcat | grep -i displayed
```

输出：

```
ActivityTaskManager: Displayed com.example.app/.MainActivity: +823ms
```

含义：**从启动意图到首帧完全绘制**，是系统官方口径。

### 方案二：代码埋点（线上监控必用）

#### 1. Application attachBaseContext 打点

在 **最早能 Hook 的位置** 记录起点：

```kotlin
class App : Application() {
    companion object {
        var startTime: Long = 0
    }

    override fun attachBaseContext(base: Context?) {
        startTime = SystemClock.elapsedRealtime()
        super.attachBaseContext(base)
    }
}
```

> 为什么用 `elapsedRealtime` 而不是 `System.currentTimeMillis`？
> 前者单调递增，不受系统时间修改影响，适合计算耗时。

#### 2. 首帧回调：`onWindowFocusChanged`（简易版）

```kotlin
class MainActivity : AppCompatActivity() {
    override fun onWindowFocusChanged(hasFocus: Boolean) {
        super.onWindowFocusChanged(hasFocus)
        if (hasFocus) {
            val cost = SystemClock.elapsedRealtime() - App.startTime
            Log.d("Launch", "启动耗时 = ${cost}ms")
        }
    }
}
```

**缺点**：`onWindowFocusChanged` 只代表 Window 获得焦点，不等于首帧真正可见。

#### 3. 首帧回调：`ViewTreeObserver.OnDrawListener`（推荐）

在 draw 阶段回调，拿到**真实首帧绘制时机**，比 `Choreographer.postFrameCallback` 更精确（后者要等下一次 Vsync，多一个周期 ~16ms）：

```kotlin
override fun onCreate(savedInstanceState: Bundle?) {
    super.onCreate(savedInstanceState)
    setContentView(R.layout.activity_main)

    val decorView = window.decorView
    decorView.viewTreeObserver.addOnDrawListener(
        object : ViewTreeObserver.OnDrawListener {
            override fun onDraw() {
                val cost = SystemClock.elapsedRealtime() - App.startTime
                Log.d("Launch", "首帧绘制耗时 = ${cost}ms")
                // 不能在 onDraw 里直接 remove，需 post 出去
                decorView.post {
                    decorView.viewTreeObserver.removeOnDrawListener(this)
                }
            }
        }
    )
}
```

**各方案时机对比**（由早到晚）：

| 回调                              | 时机                     | 精度      |
| --------------------------------- | ------------------------ | --------- |
| `OnGlobalLayoutListener`          | layout 完成              | ❌ 太早   |
| `OnPreDrawListener`               | draw 前                  | ✅ 备选   |
| `OnDrawListener`                  | draw 中                  | ✅ 推荐   |
| `Choreographer.postFrameCallback` | **下一帧 Vsync**，偏晚   | ⚠️ 有误差 |
| `onWindowFocusChanged`            | Window 获焦              | ⚠️ 最晚   |

#### 4. 完全可见：`reportFullyDrawn`（Google 推荐）

用于统计**数据加载完、用户真正能看到内容**的时间（含网络/异步）：

```kotlin
// 数据加载完成后调用
lifecycleScope.launch {
    val data = repository.load()
    adapter.setData(data)
    reportFullyDrawn()  // 系统会打印 "Fully drawn" 日志
}
```

Logcat：

```
ActivityTaskManager: Fully drawn com.example.app/.MainActivity: +1523ms
```

### 方案三：细粒度阶段埋点（定位瓶颈）

把启动拆成多个阶段，分别打点：

```kotlin
object LaunchTracer {
    private val events = mutableMapOf<String, Long>()
    private var startTime = 0L

    fun start() {
        startTime = SystemClock.elapsedRealtime()
    }

    fun mark(tag: String) {
        events[tag] = SystemClock.elapsedRealtime() - startTime
    }

    fun dump() {
        events.toSortedMap(compareBy { events[it] }).forEach { (tag, cost) ->
            Log.d("Launch", "$tag = ${cost}ms")
        }
    }
}

// 使用
class App : Application() {
    override fun attachBaseContext(base: Context?) {
        LaunchTracer.start()
        super.attachBaseContext(base)
        LaunchTracer.mark("attachBaseContext")
    }

    override fun onCreate() {
        super.onCreate()
        initSdk()
        LaunchTracer.mark("Application.onCreate")
    }
}

class MainActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_main)
        LaunchTracer.mark("Activity.onCreate")

        val decorView = window.decorView
        decorView.viewTreeObserver.addOnDrawListener(
            object : ViewTreeObserver.OnDrawListener {
                override fun onDraw() {
                    LaunchTracer.mark("首帧")
                    LaunchTracer.dump()
                    decorView.post {
                        decorView.viewTreeObserver.removeOnDrawListener(this)
                    }
                }
            }
        )
    }
}
```

## 四、起点选择的坑

| 起点候选                     | 是否推荐 | 说明                                          |
| ---------------------------- | -------- | --------------------------------------------- |
| `Application.attachBaseContext` | ✅ 推荐   | 目前用户代码能插入的最早时机                  |
| `Application.onCreate`       | ⚠️       | 错过了 attachBaseContext 到 onCreate 间的耗时 |
| `ContentProvider` 的 onCreate | ✅       | 比 Application.onCreate 更早，多进程可用      |
| 真正的 0 点（点击图标时刻） | ❌ 无法获取 | 只有系统知道，业务代码拿不到                |

**面试加分点**：Google 在 **Android 10+** 通过 `onAppStartup` / App Startup 库和 JankStats 提供更准确的统计入口。

## 五、线上监控架构

```
┌──────────┐   打点    ┌──────────┐  聚合 ┌──────────┐
│  App 端  │ ────────▶ │  本地缓存 │ ────▶ │  上报服务 │
└──────────┘           └──────────┘        └──────────┘
                                                 │
                                                 ▼
                                            ┌──────────┐
                                            │  监控平台 │ (按版本/机型分组)
                                            └──────────┘
```

关键指标：
- **启动耗时 P50/P90/P99**（只看平均值会被长尾掩盖）
- **冷启动占比**（热/温启动单独统计）
- **首帧 vs Fully Drawn** 两个指标都要

## 六、优化方向（面试延伸）

1. **Application 瘦身**：SDK 延迟初始化、按需初始化（App Startup 库）
2. **异步化**：子线程初始化（注意线程安全）
3. **闪屏优化**：用 `windowBackground` 替代代码绘制闪屏
4. **布局优化**：减少层级、使用 `ViewStub`、`include`、`merge`
5. **类加载优化**：MultiDex 优化、ReDex
6. **I/O 优化**：SharedPreferences 异步加载、DB 懒加载
7. **资源预加载**：利用 IdleHandler 在主线程空闲时加载

## 七、面试题

### Q1：adb am start -W 输出的 ThisTime、TotalTime、WaitTime 区别？

- **ThisTime**：最后一个 Activity 启动耗时（有闪屏时是闪屏 Activity）
- **TotalTime**：整个应用启动耗时，从启动开始到目标 Activity 首帧
- **WaitTime**：AMS 角度耗时，包含上一个 Activity onPause 的时间

自测应用启动速度看 **TotalTime**；排查 Activity 切换延迟看 **WaitTime**。

### Q2：为什么不能用 `onCreate` 结束作为启动完成？

`onCreate` 返回后还要经历 `onStart` → `onResume` → ViewRootImpl 的 measure/layout/draw，**首帧还没上屏**。用户感知的启动完成是首帧绘制完成，所以要用 `Choreographer.postFrameCallback` 或 `onWindowFocusChanged`。

### Q3：`onWindowFocusChanged` 和 `Choreographer.postFrameCallback` 哪个更准？

- `onWindowFocusChanged`：Window 获得焦点回调，早于首帧绘制，误差偏小。
- `Choreographer.postFrameCallback`：下一次 Vsync 回调，能拿到**真实首帧时间**，更准。

线上一般两个都打，对比取 `postFrameCallback`。

### Q4：如何统计"真正可交互"而不是"首帧显示"的耗时？

调用 `Activity.reportFullyDrawn()`。这个方法用于标记数据加载完毕、用户真正可以看到完整内容的时刻，Google Play Vitals 也以此作为启动指标。

### Q5：Application.attachBaseContext 一定是最早的打点吗？

不是。**ContentProvider 的 onCreate 早于 Application.onCreate**（在 attachBaseContext 之后）。一些 SDK（如 LeakCanary、FaceBook 初始化器）就是利用这一点做无侵入初始化。但 `attachBaseContext` 依然是业务代码能插入的最早位置之一。

### Q6：为什么启动耗时统计要用 `SystemClock.elapsedRealtime()` 而不是 `System.currentTimeMillis()`？

- `currentTimeMillis`：墙上时钟，受用户修改系统时间、NTP 同步影响，可能出现负数或跳跃。
- `elapsedRealtime`：自开机后单调递增，包含深度睡眠，适合测量时间间隔。

### Q7：线上启动耗时为什么要看 P90/P99 而不是平均值？

平均值会被大量快速启动"稀释"，掩盖长尾用户（低端机、冷启动）的真实体验。P90 表示 90% 用户启动耗时低于该值，能反映**尾部用户体验**，是优化的重点指标。

### Q8：首次安装冷启动为什么特别慢？

- **Dex 优化**：首次安装时系统 odex/oat 尚未完成，运行时解释执行
- **资源索引冷**：Resources.arsc 未进入 pagecache
- **多进程初始化**：ContentProvider、Service 同步启动
- **网络请求**：首次冷启往往伴随登录、配置拉取

面试答题模板可以用"首次启动 = 冷启动 + 冷 I/O + 冷网络"。
