# Android 内存优化详解

## 一、优化目标

```
           内存优化
               │
   ┌───────────┼───────────┐
   │           │           │
 降低常驻内存  防止 OOM    减少 GC 频率
   │           │           │
降低系统压力   不闪退      不卡顿
   │
提升后台存活率
```

**三个指标**：Java Heap / Native Heap / PSS 物理内存。

---

## 二、Android 内存分区

```
┌────────────────────────────────────┐
│  Java Heap（受 dalvik.vm.heapsize  │  ← Profiler Java 看这里
│  限制，通常 192~512MB）            │
├────────────────────────────────────┤
│  Native Heap（C/C++，Bitmap 8.0+   │  ← Profiler Native
│  在这里）                          │
├────────────────────────────────────┤
│  Stack（每线程 ~1MB）              │
├────────────────────────────────────┤
│  Code（.so / .dex）                │
├────────────────────────────────────┤
│  Graphics（GL、EGL 纹理）          │
└────────────────────────────────────┘
        ↑
   加起来就是 PSS
```

超过 heapsize 阈值就 OOM；PSS 过高容易被 LMK（Low Memory Killer）杀。

---

## 三、五大优化方向

### 方向 1：修内存泄漏（最优先）

**本质**：不该活的对象被 GC Root 引用无法回收。

**常见场景**：

| 场景                          | 根因                              | 解决                         |
| ----------------------------- | --------------------------------- | ---------------------------- |
| 静态变量持有 Activity         | static field → Activity           | 用 ApplicationContext        |
| 非静态内部类持有外部类        | 隐式 `this$0`                     | 改 static + WeakReference    |
| Handler 泄漏                  | MessageQueue 持有 Message 持有 Handler | static Handler + WeakRef |
| 单例持有 Activity Context     | 全局引用                          | 传 ApplicationContext        |
| 未反注册 BroadcastReceiver    | 系统持有                          | onDestroy 反注册             |
| Bitmap/Cursor 未 recycle/close | 资源未释放                        | try-with-resources           |
| 属性动画循环未 cancel         | Animator 持有 View                | onDestroy 里 cancel          |
| WebView 没销毁                | 多进程引用                         | WebView 独立进程 + removeAllViews |

```kotlin
// ❌ Handler 泄漏
class MainActivity : Activity() {
    private val handler = Handler {  // 内部类隐式持有 Activity
        updateUI(); true
    }
}

// ✅ static + WeakReference
class SafeHandler(activity: MainActivity) : Handler(Looper.getMainLooper()) {
    private val ref = WeakReference(activity)
    override fun handleMessage(msg: Message) {
        ref.get()?.updateUI()
    }
}
```

**工具**：
- **本地**：LeakCanary 自动检测、Matrix ResourceCanary（测试阶段）
- **线上**：ResourceCanary 官方不建议全量启用（Dump Hprof 会卡 5-15s），可"只监测不 Dump"模式用于发现测试期难复现的泄漏；OOM 堆栈收集推荐快手 **KOOM**（裁剪 hprof + 异步上报，线上可用）

---

### 方向 2：消除内存抖动

**表现**：Profiler 内存曲线锯齿状，短时间大量分配 + GC。

**优化**：

```kotlin
// ❌ 循环里拼字符串
var s = ""
for (i in list) s += i.name      // 每次 new 一个 String

// ✅ StringBuilder
val sb = StringBuilder()
for (i in list) sb.append(i.name)

// ❌ onDraw 创建对象
override fun onDraw(canvas: Canvas) {
    val paint = Paint()
}

// ✅ 成员变量复用
private val paint = Paint()
```

**规则**：**热路径（onDraw、onMeasure、帧回调、RecyclerView.onBindViewHolder）不 new 对象**。

---

### 方向 3：Bitmap 优化（内存大户）

Bitmap 占用计算：`宽 × 高 × 每像素字节`

```
ARGB_8888  每像素 4 字节  ← 默认
RGB_565    每像素 2 字节  ← 无透明度，省一半
ALPHA_8    每像素 1 字节  ← 仅掩膜
```

**6 招优化**：

```kotlin
// 1. 采样加载（inSampleSize）
val opts = BitmapFactory.Options().apply {
    inJustDecodeBounds = true
}
BitmapFactory.decodeFile(path, opts)
opts.inSampleSize = calcInSampleSize(opts, reqW, reqH)
opts.inJustDecodeBounds = false
val bitmap = BitmapFactory.decodeFile(path, opts)

// 2. 选择更省的格式
opts.inPreferredConfig = Bitmap.Config.RGB_565

// 3. 复用（inBitmap）
opts.inMutable = true
opts.inBitmap = oldBitmap     // 复用老 Bitmap 的内存

// 4. 及时 recycle（老版本，8.0+ 由 GC 管）
bitmap.recycle()

// 5. LRU 缓存
val cache = LruCache<String, Bitmap>(maxSize)

// 6. 用框架：Glide / Coil（自带 BitmapPool + 生命周期管理）
Glide.with(this).load(url).into(imageView)
```

**硬件加速的图**：Android 10+ 可用 `ImageDecoder` + `Bitmap.Config.HARDWARE`，像素存在 GPU，Java 堆占用极小。

### 方向 4：数据结构与集合

| 场景                             | 反模式            | 优化                             |
| -------------------------------- | ----------------- | -------------------------------- |
| key 是 int                       | `HashMap<Integer,X>` | `SparseArray<X>`、`SparseIntArray` |
| key 是 long                      | `HashMap<Long,X>`    | `LongSparseArray<X>`             |
| value 是基本类型                 | `ArrayList<Integer>` | `IntArray`                       |
| 两个对象 map                     | `HashMap`         | `ArrayMap`（< 1000 项更省）      |
| 已知大小                         | `ArrayList()`     | `ArrayList(knownSize)` 避免扩容  |
| Enum 多                           | enum class        | 用 `@IntDef` / 常量               |

### 方向 5：进程与资源管理

#### 1. 按内存压力释放资源

```kotlin
override fun onTrimMemory(level: Int) {
    when (level) {
        TRIM_MEMORY_UI_HIDDEN       -> clearUICache()        // 切后台
        TRIM_MEMORY_RUNNING_MODERATE,
        TRIM_MEMORY_RUNNING_LOW,
        TRIM_MEMORY_RUNNING_CRITICAL -> clearMemoryCache()   // 前台内存紧张
        TRIM_MEMORY_BACKGROUND,
        TRIM_MEMORY_MODERATE,
        TRIM_MEMORY_COMPLETE         -> clearAllCaches()     // 即将被杀
    }
}
```

#### 2. 大图/独立功能用独立进程

```xml
<activity android:process=":web" />   <!-- WebView 独立进程 -->
```

进程被杀时不影响主进程内存。

#### 3. LargeHeap 慎用

```xml
<application android:largeHeap="true"/>   <!-- 堆上限扩大，但别依赖 -->
```

只是治标，**治本还是减少占用**。

#### 4. 启用 R8 / ProGuard

```gradle
buildTypes {
    release {
        minifyEnabled true
        shrinkResources true
    }
}
```

减少 dex 大小、移除无用资源，间接降低运行时内存。

## 四、Native 内存优化

- 避免 JNI 里忘记 `ReleaseStringUTFChars` / `DeleteLocalRef`
- 大数组优先 `DirectByteBuffer`，减少 GC
- 第三方 so 做 `System.loadLibrary` 按需加载
- Android 10+ 的 `GWP-ASan` 可排查 Native 内存问题

```
日常开发：
  Android Profiler Memory    → 看曲线、抓 dump
  LeakCanary                 → 自动发现泄漏
  Layout Inspector           → 查过深 View

深度分析：

  Matrix TraceCanary/IOCanary → 卡顿、主线程 IO（线上可用）
  Matrix ResourceCanary      → 仅测试阶段，官方不建议线上全量（Dump Hprof 卡 5-15s）

  Probe (阿里)               → OOM 监控方案
```

## 六、MAT 排查 OOM 四步法

1. `adb shell am dumpheap <pkg> /data/local/tmp/heap.hprof`
2. `hprof-conv heap.hprof converted.hprof`（老版本需要）
3. MAT 打开 → Dominator Tree → 按 Retained Heap 排序
4. 可疑对象右键 `Merge Shortest Paths to GC Roots → exclude weak/soft` 找引用链

## 七、面试题

### Q1：Android 内存优化的整体思路？

按优先级：
1. **修泄漏**（LeakCanary 找，一个泄漏会持续累积）
2. **降峰值**（Bitmap 采样、避免大对象一次性加载）
3. **减抖动**（热路径不分配）
4. **合理释放**（onTrimMemory、Lifecycle）
5. **结构优化**（SparseArray、ArrayMap）

### Q2：Bitmap 内存到底在哪？

- **Android 2.3.3 以下**：Native 堆
- **Android 3.0 ~ 7.x**：Java 堆
- **Android 8.0+**：**Native 堆**（像素），Java 堆只有引用头 —— 由 NativeAllocationRegistry 管理，GC 时顺带释放

所以 8.0+ 单张大图不会直接撑爆 Java 堆。

### Q3：为什么 SparseArray 比 HashMap 省内存？

- HashMap 的 key 要装箱（`int → Integer`），每个 Integer 16 字节
- HashMap 内部是数组 + 链表/红黑树，有 `Node<K,V>` 额外对象
- SparseArray 是两个数组（`int[] keys` + `Object[] values`），无装箱、无 Entry

1000 项以下优势明显，超过后查找性能退化（二分查找 O(log n)）。

### Q4：onLowMemory 和 onTrimMemory 区别？

- `onLowMemory()`：整个系统内存不足，所有 app 都会收到，API 14 前常用
- `onTrimMemory(level)`：**更细粒度**，包含 app 自身切后台等情况，是现在推荐的

新代码只实现 `onTrimMemory`。

### Q5：静态集合如何防止泄漏？

- 用 `WeakHashMap` 让 key 弱引用
- 用 `LruCache` 有上限
- 生命周期结束 `clear()`
- 禁止用 `static List<Activity>` 这种结构

### Q6：线程/线程池也会引起内存泄漏？

会。

- 子线程持有 Activity 引用，Activity 销毁但线程还在跑 → 泄漏
- 线程池没 shutdown，Runnable 闭包持有外部 → 泄漏
- 典型坑：`lifecycleScope.launch` 能自动取消，而 `GlobalScope.launch` 不会

### Q7：Application 里能持有 Activity 引用吗？

不能。Application 的生命周期是进程级，持有的 Activity 引用会导致该 Activity 永远无法回收。若非要用，用 `WeakReference`，并在合适时机清理。

### Q8：MAT 的 Shallow Heap 和 Retained Heap 区别？

- **Shallow Heap**：对象自身占用（不含引用对象）
- **Retained Heap**：对象被 GC 后能释放的总内存（含仅被它引用的对象）

排查 OOM 看 **Retained Heap** —— 找"杀了它能省最多"的对象。

### Q9：频繁 GC 为什么会卡顿？

GC 某些阶段必须 STW（Stop The World），ART 的 Concurrent Copying GC 仍有初始标记和重标记两次暂停。单次几毫秒看似小，频繁 GC 叠加后就爆掉 16.67ms 一帧预算，表现为丢帧。详见 `GC频繁导致卡顿.md`（同目录）。

### Q10：System.gc() 应该调吗？

生产代码**禁止**调用。它触发 Full GC，STW 最长，反而加剧卡顿。真正优化应减少对象分配频率。
