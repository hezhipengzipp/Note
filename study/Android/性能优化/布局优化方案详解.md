# Android 布局优化方案详解

## 一、为什么优化布局

布局问题直接影响：
- **启动速度**（inflate 慢）
- **滑动帧率**（measure/layout 耗时）
- **内存占用**（View 对象多）
- **过度绘制**（GPU 多次填充同一像素）

```
一帧预算 16.67ms
┌─────────┬─────────┬─────────┬──────┐
│ measure │ layout  │  draw   │ sync │
│  1~5ms  │  1~3ms  │  5~10ms │      │
└─────────┴─────────┴─────────┴──────┘
       ↑         ↑         ↑
   层级越深    嵌套越深   Overdraw
   越慢        越慢       越慢
```

## 二、10 大优化方向

### 1. 减少层级（扁平化）

#### ConstraintLayout 替代多层嵌套

```xml
<!-- ❌ 三层嵌套 -->
<LinearLayout orientation="vertical">
    <RelativeLayout>
        <LinearLayout orientation="horizontal">
            <TextView/><TextView/>
        </LinearLayout>
    </RelativeLayout>
</LinearLayout>

<!-- ✅ ConstraintLayout 扁平 -->
<androidx.constraintlayout.widget.ConstraintLayout>
    <TextView app:layout_constraintTop_toTopOf="parent"/>
    <TextView app:layout_constraintEnd_toEndOf="parent"/>
</androidx.constraintlayout.widget.ConstraintLayout>
```

**原则**：层级一般不超过 **10 层**，复杂页面用 ConstraintLayout、MotionLayout。

---

### 2. `<merge>` 去掉根节点

当自定义 View 的 xml 根节点和父容器类型一致时，用 `<merge>` 省一层：

```xml
<!-- my_title_bar.xml，不要写 <LinearLayout> 再套内容 -->
<merge xmlns:android="...">
    <ImageView .../>
    <TextView .../>
</merge>
```

```kotlin
// 自定义 View
class TitleBar(...) : LinearLayout(...) {
    init {
        LayoutInflater.from(context).inflate(R.layout.my_title_bar, this, true)
    }
}
```

**使用条件**：父容器已确定布局方式（如 `LinearLayout`、`FrameLayout`）。

---

### 3. `<include>` 复用布局

```xml
<include
    layout="@layout/common_toolbar"
    android:layout_width="match_parent"
    android:layout_height="wrap_content"/>
```

**注意**：`<include>` 根标签要覆盖 width/height 才生效。

---

### 4. `<ViewStub>` 延迟加载

**作用**：占位不 inflate，需要时才加载，省启动时间。

```xml
<ViewStub
    android:id="@+id/stub_error"
    android:layout="@layout/error_view"
    android:layout_width="match_parent"
    android:layout_height="wrap_content"/>
```

```kotlin
// 需要时 inflate（只能 inflate 一次）
val errorView = findViewById<ViewStub>(R.id.stub_error).inflate()

// 再次可见用 visibility 控制
errorView.visibility = View.VISIBLE
```

**典型场景**：
- 空状态、错误页
- 只有 VIP 才显示的功能
- 首次引导浮层

---

### 5. 异步 Inflate（AsyncLayoutInflater）

耗时大的布局用子线程 inflate：

```kotlin
AsyncLayoutInflater(this).inflate(R.layout.complex, parent) { view, _, _ ->
    parent.addView(view)
}
```

**限制**：
- 布局不能含有 `<fragment>` 或主线程才能创建的 View
- 不能在布局里做 `AsyncTask` 相关操作
- 生产更推荐 **AsyncInflater**（Jetpack）或自己基于协程

---

### 6. 避免 Overdraw（过度绘制）

```
原色 → 蓝 1x → 绿 2x → 粉 3x → 红 4x+
                              ↑ 需优化
```

#### 三招去掉无用背景

```xml
<!-- 主题里移除默认 windowBackground -->
<style name="AppTheme">
    <item name="android:windowBackground">@null</item>
</style>
```

```xml
<!-- Fragment 容器有背景时，内层子 View 就不要再加背景 -->
<FrameLayout android:background="@color/white">
    <LinearLayout android:background="@color/white"/>  <!-- ❌ 多余 -->
</FrameLayout>
```

#### clipRect 裁剪自绘 View

```kotlin
override fun onDraw(canvas: Canvas) {
    canvas.save()
    canvas.clipRect(visibleRect)    // 只画可见区域
    super.onDraw(canvas)
    canvas.restore()
}
```

---

### 7. 减少 measure 次数

#### 避免权重 + wrap_content

```xml
<!-- ❌ LinearLayout 的 weight + wrap_content 会 measure 两次 -->
<LinearLayout android:orientation="horizontal">
    <TextView android:layout_width="wrap_content"
              android:layout_weight="1"/>
</LinearLayout>
```

`LinearLayout` 遇到 weight 需要先测一次拿剩余空间，再根据比例分配,是常见的双重 measure 来源。

#### RelativeLayout 的问题

RelativeLayout 要根据相对关系做 **两次 onMeasure**（横向纵向各一次）。布局复杂时比 ConstraintLayout 慢。

#### ConstraintLayout 的警示

ConstraintLayout 也有双重测量场景：
- 链（chain）且使用 `packed`、`spread_inside`
- `match_constraint` + `percent` 混用

用 Layout Inspector 看 **measure/layout count**。

---

### 8. RecyclerView 优化

这是列表布局的重点。

```kotlin
recyclerView.apply {
    setHasFixedSize(true)                    // 已知 item 大小不变
    itemAnimator = null                       // 关动画减少重绘
    recycledViewPool.setMaxRecycledViews(0, 20)
    setItemViewCacheSize(4)
}
```

**ViewHolder 优化**：
- 不在 `onBindViewHolder` 里 `findViewById`（编译期已缓存）
- 避免在 bind 里创建对象（monitor GC）
- 用 DiffUtil 局部刷新
- 复杂 item 用 **预加载**（`prefetchItemCount`）

**其他**：
- item 嵌套 RecyclerView 设 `setRecycledViewPool` 共享
- 预取下一屏用 `PrefetchItemViewCacheExtension`
- 首屏高度大量 item 可考虑 **AsyncInflate + ViewCompat.setId**

---

### 9. 自定义 View 优化

| 问题                 | 优化                          |
| -------------------- | ----------------------------- |
| onDraw new 对象      | 成员变量复用 Paint/Path/Rect  |
| 频繁 invalidate      | 只刷需要的 `invalidate(rect)` |
| 全量重绘             | `clipRect` + 脏区             |
| 复杂绘制             | `setLayerType(HARDWARE, null)` |
| onMeasure 多次       | 避免在 measure 里做耗时操作    |

---

### 10. 上层架构方向

- **Compose**：声明式，无 xml 层级，**平均比 View 体系更扁平**，但冷启动首帧慢一点
- **DataBinding/ViewBinding**：编译期生成代码，省掉 findViewById 开销（微小）
- **LayoutInflater Hook**：字节的 X2C 把 xml 转成 Java 代码，inflate 快一倍
- **SurfaceView / TextureView**：独立渲染线程，游戏/视频场景

| 工具                       | 作用                              |
| -------------------------- | --------------------------------- |
| **Layout Inspector**       | 看层级、measure count、命中过深   |
| **开发者选项 - GPU 呈现条** | 看哪段慢                          |
| **开发者选项 - GPU 过度绘制** | 看红色区域                       |
| **Profile GPU Rendering**  | 每帧耗时条形图                    |

## 五、面试题

### Q1：ConstraintLayout 一定比 RelativeLayout 快吗？

**不一定**。简单布局（2-3 个 View）ConstraintLayout 反而有额外约束求解开销。**复杂布局（多层嵌套）** ConstraintLayout 明显更快，因为扁平化减少了 measure/layout 递归。

### Q2：`<include>` 和 `<merge>` 配合使用的规则？

- `<include>` 里的布局根节点是 `<merge>` 时，必须保证 include 的父容器已经是对应布局类型（`LinearLayout`、`FrameLayout` 等）
- `<merge>` 不能作为顶层布局直接 setContentView
- `<merge>` 里的属性（layout_xxx）会被忽略，必须在 `<include>` 标签上覆盖

### Q3：ViewStub 的原理？

`ViewStub` 是 0 宽高的**轻量 View**，本身不参与 measure/layout（不可见）。调用 `inflate()` 时：
1. 加载 layout 属性指向的布局
2. 把 stub 从父容器移除
3. 用新 inflate 的 View 替换 stub 位置
4. 只能 inflate 一次，第二次抛 IllegalStateException

**注意**：inflate 后原 stub 引用失效，要保存返回的 View。

### Q4：为什么 RelativeLayout 性能比 LinearLayout 差？

RelativeLayout 需要按子 View 之间的相对关系求解，**一次 measure 不够**：先水平方向、再垂直方向，各做一次。子 View 数量 N 的情况下复杂度偏向 O(N²)。

LinearLayout 是顺序 measure 一次即可，除非遇到 weight。

### Q5：AsyncLayoutInflater 有什么坑？

1. 布局里不能有 `<fragment>`
2. 布局里的 View 必须所有构造器都线程安全（大部分系统 View 没问题，但第三方 View 可能不）
3. 不支持 `LayoutInflater.Factory`（如 AppCompat 主题会失效）
4. 回调在主线程，inflate 发生在子线程

生产上很多人用 X2C 或手写 View 替代。

### Q6：Layout Inspector 里看到 measure count = 2 代表什么？

代表这个 View 被 measure 了两次。常见原因：
- 父容器是 RelativeLayout / LinearLayout with weight
- ScrollView + wrap_content 子节点
- ConstraintLayout 的特殊约束

多次 measure 本身不致命，但层级深时会指数级恶化。

### Q7：RecyclerView 的 setHasFixedSize 作用？

告诉 RecyclerView **adapter 里 item 增删不会改变 RecyclerView 自身尺寸**（只改变内容）。这样 item 变化时跳过 `requestLayout()` 的昂贵操作，只 invalidate 相关区域。

### Q8：过度绘制多少层算需要优化？

- 1x（蓝）：可以接受
- 2x（绿）：正常范围
- 3x（粉）：开始警惕
- 4x+（红）：必须优化

一般要求**关键页面红色区域 < 10%**。全屏白底就是典型的 2x（Activity windowBackground + 布局背景）。

### Q9：merge 标签使用后层级真的少一层吗？

是。`<merge>` 不会产生实际 View，它的子 View 直接挂在父容器下。Layout Inspector 中也看不到 merge 节点。但如果布局没有父容器（如直接 setContentView），merge 就不能用。

### Q10：Jetpack Compose 布局性能比 View 系统好吗？

- **首帧**：Compose 略慢（需要初始化 composition runtime）
- **重组**：Compose 更快（精确重组，跳过未变化的节点）
- **层级**：Compose 天然扁平
- **列表**：LazyColumn 和 RecyclerView 性能接近

现阶段 Compose 已经可以做生产应用，**新项目推荐**。
