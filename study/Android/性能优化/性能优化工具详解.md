# Android 性能优化工具详解

## 一、工具全景图

```
                   Android 性能优化工具
                          │
   ┌──────────────────────┼──────────────────────┐
   │                      │                      │
 系统级                  方法级                线上监控
   │                      │                      │
   ├─ Systrace(旧)       ├─ Android Profiler    ├─ LeakCanary
   ├─ Perfetto(新)       ├─ TraceView(旧)       ├─ BlockCanary
   ├─ Simpleperf         ├─ Method Tracing      ├─ Matrix(腾讯)
   └─ AGI(GPU)           └─ Benchmark(Jetpack)  └─ ANR-WatchDog

   ┌──────────────────────┼──────────────────────┐
   │                      │                      │
 内存分析                UI/渲染               网络
   │                      │                      │
   ├─ Memory Profiler    ├─ Layout Inspector    ├─ Network Profiler
   ├─ MAT               ├─ GPU Profiler         ├─ Charles
   ├─ LeakCanary        ├─ Overdraw(开发者选项) └─ Stetho
   └─ HPROF             └─ Choreographer
```

**选型原则**：
- **本地开发调优** → Android Studio Profiler
- **系统级深度分析** → Perfetto（新）/Systrace（旧）
- **线上监控** → Matrix / LeakCanary
- **方法级耗时** → Trace.beginSection + Profiler / Simpleperf

## 二、Systrace（旧版系统 Trace）

### 原理

Systrace 通过 **ftrace**（Linux 内核跟踪机制）收集内核、HAL、系统服务、应用层的关键事件，生成可视化 HTML。

```
    ┌────────────────────────────────────┐
    │  Java 层 Trace.beginSection()      │
    │  Native 层 ATRACE_BEGIN()          │
    │  系统服务内置埋点（SF/AMS/WMS）    │
    └───────────────┬────────────────────┘
                    │ 写入 /sys/kernel/debug/tracing/trace_marker
                    ▼
    ┌────────────────────────────────────┐
    │  Linux Kernel ftrace              │
    └───────────────┬────────────────────┘
                    │ systrace.py 抓取并转 HTML
                    ▼
    ┌────────────────────────────────────┐
    │  Chrome Tracing 可视化             │
    └────────────────────────────────────┘
```

### 使用方法

```bash
# Python 版（AndroidSDK/platform-tools/systrace/）
cd $ANDROID_HOME/platform-tools/systrace
python systrace.py -t 10 -o trace.html -a com.example.app \
    sched gfx view wm am app dalvik binder_driver

# 参数说明
-t 10               采集 10 秒
-o trace.html       输出文件
-a com.example.app  指定应用（开启 app 级 tag）
后面是 category：sched调度 gfx渲染 view wm窗口 am四大组件 dalvik虚拟机 binder
```

### 代码埋点

```kotlin
Trace.beginSection("loadUserList")
try {
    loadData()
} finally {
    Trace.endSection()  // 必须成对
}
```

### 看什么

- **主线程时间轴**：找"橙色/红色长条"（卡顿帧）
- **Frames 行**：绿色=正常，黄/红=丢帧
- **Alerts**：系统直接给出卡顿原因提示

**缺点**：
- Python 脚本，Android 10+ 已废弃
- UI 是 HTML，大文件打开慢
- 最多抓几十秒

## 三、Perfetto（推荐，Systrace 替代品）

### 原理

Perfetto 是 Google 新一代 trace 系统，**向后兼容 Systrace**，但采用二进制 protobuf 格式，支持更长时间、更多维度采集。

```
┌──────────────────────────────────────────┐
│  Perfetto Tracing Service (traced)       │  ← 常驻系统服务
│  ┌──────────────┐  ┌──────────────┐      │
│  │ ftrace probe │  │ atrace probe │      │
│  └──────┬───────┘  └──────┬───────┘      │
│         └────────┬─────────┘             │
│                  ▼                       │
│         共享内存环形缓冲区               │
└──────────────────┬───────────────────────┘
                   │ 输出 .perfetto-trace (protobuf)
                   ▼
          ui.perfetto.dev 可视化
```

### 三种抓取方式

#### 方式 1：开发者选项（最简单）

```
设置 → 开发者选项 → System Tracing → 记录跟踪
     → 下拉通知栏，点停止 → 分享到电脑
```

#### 方式 2：adb 命令

```bash
# 快捷方式：和 systrace 参数一样
adb shell perfetto -o /data/misc/perfetto-traces/trace.pftrace \
    -t 10s \
    -a com.example.app \
    sched freq idle am wm gfx view binder_driver hal dalvik input res memory

# 拉到本地
adb pull /data/misc/perfetto-traces/trace.pftrace ./
```

#### 方式 3：配置文件（最灵活）

```bash
# 写一份 trace_config.pbtx
cat > trace_config.pbtx <<EOF
buffers { size_kb: 63488 fill_policy: DISCARD }
data_sources {
    config {
        name: "linux.ftrace"
        ftrace_config {
            ftrace_events: "sched/sched_switch"
            ftrace_events: "power/cpu_frequency"
            atrace_categories: "gfx"
            atrace_categories: "view"
            atrace_apps: "com.example.app"
        }
    }
}
duration_ms: 10000
EOF

adb shell perfetto --txt -c - -o /data/misc/perfetto-traces/trace.pftrace \
    < trace_config.pbtx
```

### 可视化分析

打开 **https://ui.perfetto.dev**，拖入 `.pftrace` 文件。

**核心面板**：

```
┌─────────────────────────────────────────────────┐
│ CPU 时间轴（每个 CPU 核心）                      │ ← 看核心是否满载
├─────────────────────────────────────────────────┤
│ 进程/线程泳道                                    │
│  ├─ UI 线程     [==onCreate==][=measure=][draw] │ ← 主线程分析重点
│  ├─ RenderThread [        ==sync==][==draw==]   │ ← 渲染线程
│  └─ Binder 线程                                  │
├─────────────────────────────────────────────────┤
│ SurfaceFlinger 合成                              │ ← 丢帧分析
├─────────────────────────────────────────────────┤
│ Expected/Actual Frames                           │ ← 绿/黄/红帧状态
└─────────────────────────────────────────────────┘
```

**常用快捷键**：
- `W/S`：放大/缩小
- `A/D`：左移/右移
- `M`：标记区间测量耗时
- 点击 slice → 看详细堆栈

### SQL 查询（高级）

Perfetto 支持用 SQL 查询 trace 数据：

```sql
-- 查询主线程 > 16ms 的方法
SELECT name, dur/1e6 AS ms
FROM slice
WHERE thread_name = 'main' AND dur > 16000000
ORDER BY dur DESC LIMIT 20;
```

## 四、Android Studio Profiler（日常首选）

### 四大模块

| 模块    | 用途                       | 替代的老工具    |
| ------- | -------------------------- | --------------- |
| CPU     | 方法耗时、线程状态         | TraceView       |
| Memory  | 内存分配、GC、泄漏         | MAT + Allocation|
| Network | HTTP/WebSocket 请求        | Charles         |
| Energy  | 电量消耗（定位唤醒锁）     | Battery Historian|

### CPU Profiler 四种采样模式

```
┌──────────────────────┬──────────┬──────────────┬─────────────┐
│ 模式                 │ 开销     │ 精度         │ 场景        │
├──────────────────────┼──────────┼──────────────┼─────────────┤
│ Sample Java Methods  │ 低       │ 采样，会漏   │ 常规分析    │
│ Trace Java Methods   │ 高       │ 每个方法都记 │ 短时精细    │
│ Sample C/C++         │ 低       │ Simpleperf   │ Native 分析 │
│ System Trace         │ 极低     │ 系统级       │ 类 Perfetto │
└──────────────────────┴──────────┴──────────────┴─────────────┘
```

### Memory Profiler 操作

```kotlin
// 代码里手动触发 GC + Dump
adb shell am dumpheap com.example.app /data/local/tmp/heap.hprof

// 或 Profiler 里点 "Dump Java heap" 按钮
```

**排查内存泄漏步骤**：
1. 执行可疑操作（如反复打开关闭某 Activity）
2. 手动 GC（Profiler 里的垃圾桶图标）
3. Dump Heap
4. 按 Class 排序 → 搜 Activity 名 → 看 GC Root 引用链

## 五、LeakCanary（内存泄漏自动检测）

### 原理

```
     Activity.onDestroy
           │
           ▼
  ┌────────────────────┐
  │ 包装成 WeakReference │
  └──────────┬─────────┘
             │
             ▼
   ┌────────────────────┐
   │ 5 秒后检查是否回收  │──── 已回收 ──▶ 正常
   └──────────┬─────────┘
              │ 未回收
              ▼
   ┌────────────────────┐
   │ GC + 再检查         │──── 已回收 ──▶ 正常
   └──────────┬─────────┘
              │ 仍未回收
              ▼
   ┌────────────────────┐
   │ Dump Heap + 解析   │
   │ 找到 GC Root 引用链 │
   │ 通知栏提示          │
   └────────────────────┘
```

### 接入

```gradle
dependencies {
    debugImplementation 'com.squareup.leakcanary:leakcanary-android:2.12'
}
```

**不需要任何代码**，ContentProvider 自动初始化。

### 监控其他对象

```kotlin
AppWatcher.objectWatcher.expectWeaklyReachable(
    myObject, "详细描述"
)
```

## 六、BlockCanary / Matrix（卡顿监控）

### BlockCanary 原理

利用 **Looper 的 Printer 机制**：

```
Looper.loop() {
    ...
    if (logging != null) logging.println(">>> Dispatching ...")  ← A
    msg.target.dispatchMessage(msg)                              ← 执行消息
    if (logging != null) logging.println("<<< Finished ...")     ← B
}
```

**思路**：注册自定义 `Printer`，A 点记时间戳，B 点比较，若超过阈值（如 500ms），dump 堆栈。

```kotlin
Looper.getMainLooper().setMessageLogging(object : Printer {
    override fun println(x: String) {
        if (x.startsWith(">>>>> Dispatching")) {
            startTime = System.currentTimeMillis()
        } else if (x.startsWith("<<<<< Finished")) {
            if (System.currentTimeMillis() - startTime > 500) {
                // dump 堆栈上报
            }
        }
    }
})
```

### Matrix（腾讯，生产级）

相比 BlockCanary 更全：
- **Trace Canary**：基于字节码插桩，方法级耗时
- **Resource Canary**：Bitmap 冗余、内存泄漏
- **IO Canary**：主线程 I/O
- **SQLite Lint**：SQL 性能问题

```gradle
dependencies {
    implementation 'com.tencent.matrix:matrix-android-lib:2.1.0'
    implementation 'com.tencent.matrix:matrix-trace-canary:2.1.0'
}
```

## 十一、面试题

### Q1：Systrace 和 Perfetto 的区别？

| 维度       | Systrace         | Perfetto             |
| ---------- | ---------------- | -------------------- |
| 采集机制   | Python 脚本+ftrace | 常驻服务 traced+ftrace |
| 输出格式   | HTML（Chrome Tracing）| protobuf            |
| 最大时长   | 几十秒           | 几分钟~小时           |
| 查询能力   | 肉眼看           | 支持 SQL             |
| 版本支持   | 已废弃           | Android 9+ 官方推荐   |

简答：**Perfetto 是 Systrace 的继任者，同内核机制（ftrace），但更强大、支持 SQL、时长更久。**

### Q2：Profiler 的 Sample 和 Trace 有什么区别？

- **Sample**：固定间隔（如 1ms）采样线程栈，**开销低**，可能漏掉短方法
- **Trace**：插桩每个方法的 enter/exit，**精确但开销大**，会影响真实耗时

线上定位用 Sample，本地精细分析用 Trace。

### Q3：BlockCanary 的原理是什么？

利用 `Looper.loop()` 里对消息分发前后调用 `Printer.println` 的机制，注册自定义 Printer 记录消息开始和结束时间，超过阈值就 dump 主线程堆栈。

**局限**：
- 只能监控主线程 Looper 消息
- 异步消息（postAtFrontOfQueue）可能遗漏
- 精度受 Printer 打印影响

生产级更推荐用 **Choreographer.FrameCallback** + **字节码插桩**（如 Matrix）。

### Q4：LeakCanary 如何判断对象泄漏？

1. 监听 Activity/Fragment 的 onDestroy
2. 用 `WeakReference` + `ReferenceQueue` 包装对象
3. 5 秒后检查 queue，若 WeakReference 没进 queue，说明没回收
4. 手动触发 GC 再检查
5. 仍未回收 → dump heap → 用 Shark 解析 → 找 GC Root 引用链

### Q5：怎么用 Perfetto 定位启动慢？

1. 开启 System Tracing，启动应用
2. 导入 ui.perfetto.dev
3. 找到目标进程的 UI 线程
4. 按时间顺序看：
   - `bindApplication` → Application 初始化耗时
   - `activityStart` → Activity 启动耗时
   - `Choreographer#doFrame` → 首帧渲染
5. 哪段长就点进去看子 slice / 方法名

### Q6：怎么优化过度绘制（Overdraw）？

1. **移除默认背景**：主题里 `windowBackground` 空白、`<item name="android:windowBackground">@null</item>`
2. **clipRect**：自定义 View 裁剪不需要绘制的区域
3. **减少层级**：`<merge>`、ConstraintLayout 扁平化
4. **透明度慎用**：alpha < 1.0 会触发离屏缓冲
5. **开发者选项"调试 GPU 过度绘制"** 验证：目标是**蓝/原色**为主

### Q7：如何在代码里插入 Systrace/Perfetto 可见的自定义 Section？

```kotlin
// Java 层
Trace.beginSection("myTask")
try { doWork() } finally { Trace.endSection() }

// AndroidX（支持 API < 29）
androidx.tracing.Trace.beginSection("myTask")
```

Native 层：

```cpp
#include <android/trace.h>
ATrace_beginSection("myTask");
doWork();
ATrace_endSection();
```

### Q8：生产环境的卡顿监控怎么做？

**思路**：本地工具不能上线，需要轻量级无侵入监控方案。

典型方案（如腾讯 Matrix）：
1. **字节码插桩**：在每个方法前后插入打点（编译期）
2. **Choreographer 监听**：`FrameCallback` 监听帧率
3. **环形缓冲**：保留最近 N 秒的方法调用栈
4. **卡顿触发**：帧率低于阈值时，dump 缓冲区上报
5. **聚合分析**：后台按堆栈相似度聚合卡顿点

比 BlockCanary 更准，也比 Trace 开销低。
