## 二、DNS 优化

### 2.1 DNS 解析耗时问题

```
常规 DNS 解析链路:

  App ──── getaddrinfo() ────▶ system DNS resolver ──── UDP ────▶ DNS Server
   │                             (LocalDNS缓存)                         │
   │                                                                   │
   └──────────────────── 50ms ~ 3000ms ────────────────────────────────┘
     （弱网/跨网场景下可能超时数秒）
```

**问题：**
- UDP 协议无连接，丢包后超时重试，默认超时很长
- 运营商 LocalDNS 劫持 / 转发，返回错误 IP
- 域名首次解析无缓存，冷启动延迟高

### 2.2 HTTPDNS

```
                    HTTPDNS 方案

  ┌──────┐    HTTPS (携带客户端IP)     ┌──────────────┐
  │ App  │ ─────────────────────────▶ │ HTTPDNS 服务  │
  │      │ ◀───────────────────────── │ (自建/阿里云等) │
  └──────┘    返回最优IP+TTL           └──────────────┘
      │                                      │
      │ 直接用IP建连                          │
      ▼                                      ▼
  ┌─────────┐                       ┌──────────────┐
  │ 业务服务器 │                       │ 权威DNS同步    │
  └─────────┘                       └──────────────┘
```

**优势：**
- 绕过运营商 LocalDNS，防劫持
- 基于客户端 IP 做精准调度（就近接入）
- 请求使用 HTTPS，安全防篡改

```java
/**
 * HTTPDNS 接入示例（以阿里云 HTTPDNS SDK 为例）
 */
public class HttpDnsHelper {

    private static HttpDnsService httpDns;

    public static void init(Context context) {
        httpDns = HttpDns.getService(context, "your-account-id");
        // 预解析：已知域名在初始化时提前解析
        httpDns.setPreResolveHosts(new ArrayList<>(Arrays.asList(
            "api.example.com",
            "cdn.example.com"
        )));
    }

    /**
     * 同步获取 IP（建议在子线程调用）
     */
    public static String getIpByHost(String host) {
        // 优先走 HTTPDNS
        String ip = httpDns.getIpByHostAsync(host);
        if (!TextUtils.isEmpty(ip)) {
            return ip;
        }
        // 兜底走 LocalDNS
        return null;
    }
}
```

### 2.3 DNS 预解析

```java
// OkHttp 层面：创建时配置自定义 DNS
OkHttpClient client = new OkHttpClient.Builder()
    .dns(new Dns() {
        @Override
        public List<InetAddress> lookup(String hostname) throws UnknownHostException {
            // 1. 先查 HTTPDNS 缓存
            String ip = HttpDnsHelper.getIpByHost(hostname);
            if (ip != null) {
                return Arrays.asList(InetAddress.getByName(ip));
            }
            // 2. 兜底系统 DNS
            return Dns.SYSTEM.lookup(hostname);
        }
    })
    .build();
```

## 三、连接优化

### 3.1 连接复用（Connection Pool）

```
              无连接池                           有连接池

  请求1:  TCP握手+TLS握手+HTTP      请求1:  TCP握手+TLS握手+HTTP
         完成后CLOSE                        保持连接(Keep-Alive)
  请求2:  TCP握手+TLS握手+HTTP      请求2:  复用连接直接发HTTP
         完成后CLOSE                        保持连接
  请求3:  TCP握手+TLS握手+HTTP      请求3:  复用连接直接发HTTP
         完成后CLOSE                        超时后CLOSE

  耗时:   3×(RTT×2 + TLS)           耗时:   1×(RTT×2 + TLS)
```

```java
// OkHttp 默认连接池：5个空闲连接，存活5分钟
// 可根据业务调整
OkHttpClient client = new OkHttpClient.Builder()
    .connectionPool(new ConnectionPool(
        10,           // maxIdleConnections: 最大空闲连接数
        5, TimeUnit.MINUTES  // keepAliveDuration: 空闲连接存活时间
    ))
    .build();
```

### 3.2 长连接 / Keep-Alive

```
  HTTP/1.1 默认开启 Keep-Alive

  请求头:  Connection: keep-alive
  响应头:  Keep-Alive: timeout=5, max=100

  timeout: 空闲连接最长保持时间
  max:     此连接上最多可发送的请求数

  注意：服务端也可能主动关闭，客户端要做好重试
```

### 3.3 连接预热

```java
/**
 * 应用启动时对核心域名建立预连接
 */
public class ConnectionWarmer {

    public static void warmUp(OkHttpClient client, String... hosts) {
        for (String host : hosts) {
            // 发起一个 HEAD 请求预热连接
            Request request = new Request.Builder()
                .url("https://" + host)
                .head()
                .build();
            client.newCall(request).enqueue(new Callback() {
                @Override
                public void onFailure(Call call, IOException e) {
                    // 忽略失败，预热本来就是 best-effort
                }

                @Override
                public void onResponse(Call call, Response response) {
                    response.close();
                }
            });
        }
    }
}
```

## 四、TLS 握手优化

### 4.1 TLS 握手开销

```
  完整 TLS 1.2 握手: 2-RTT

  Client                                  Server
    │ ── ClientHello ──────────────────────▶ │
    │                                        │
    │ ◀── ServerHello + Certificate ──────  │  RTT1
    │     + ServerKeyExchange                │
    │     + ServerHelloDone                  │
    │                                        │
    │ ── ClientKeyExchange ────────────────▶ │
    │    + ChangeCipherSpec                  │  RTT2
    │    + Finished                          │
    │                                        │
    │ ◀── ChangeCipherSpec + Finished ────  │
    │                                        │
    │ ── Application Data ────────────────▶ │
```

### 4.2 Session Ticket / Session ID 复用

```
  TLS 会话复用 —— 1-RTT

  Client (缓存了上次的 Session Ticket)         Server
    │ ── ClientHello + Session Ticket ──────▶ │ ← 携带上次的 ticket
    │                                         │
    │ ◀── ServerHello + Finished ──────────  │ RTT1 (跳过证书交换和密钥协商)
    │                                         │
    │ ── ChangeCipherSpec + Finished ──────▶ │
    │                                         │
    │ ◀── Application Data ────────────────  │
```

OkHttp 默认支持 Session Ticket，无需额外配置，只需确保 OkHttpClient 单例以复用连接即可自动生效。

### 4.3 TLS 1.3 —— 降到 1-RTT

```
  TLS 1.3 完整握手: 1-RTT（PSK模式下甚至0-RTT）

  Client                                  Server
    │ ── ClientHello + KeyShare ───────────▶ │  ← 客户端直接发送密钥参数
    │                                        │
    │ ◀── ServerHello + KeyShare ──────────  │
    │     + Certificate                      │  RTT1
    │     + Finished                         │
    │                                        │
    │ ── Finished ────────────────────────▶ │
    │                                        │
    │ ◀── Application Data ──────────────── │

   已经比 TLS 1.2 节省了 1-RTT
```

服务端配置 TLS 1.3 即可，客户端（OkHttp）自动协商。

### 4.4 证书校验优化


## 五、协议层面优化

### 5.1 HTTP/2 多路复用

```
  HTTP/1.1 (串行)                    HTTP/2 (多路复用)

  Connection 1: [REQ1]──[RESP1]
  Connection 2: [REQ2]──[RESP2]      Connection 1:
  Connection 3: [REQ3]──[RESP3]        Stream1: [REQ1]──[RESP1]
  ...                                  Stream2: [REQ2]──[RESP2]
  最多6个并发连接（浏览器限制）           Stream3: [REQ3]──[RESP3]
                                       ...
                                       一条连接承载所有请求

  优势：
  - 单连接多路复用，避免多连接竞争带宽
  - 头部压缩(HPACK)，减少重复头部传输
  - 服务端推送(Server Push)
```

```java
// OkHttp 默认支持 HTTP/2，前提是服务端开启了 HTTP/2
// 协议协商通过 ALPN（Application-Layer Protocol Negotiation）自动完成
OkHttpClient client = new OkHttpClient.Builder()
    .protocols(Arrays.asList(Protocol.HTTP_2, Protocol.HTTP_1_1))
    .build();
```

### 6.2 请求去重

```java
/**
 * 防止短时间内发起相同请求
 */
public class RequestDeduplicator {

    private final Set<String> pendingRequests = new HashSet<>();

    public synchronized boolean shouldExecute(String requestKey) {
        // key = method + url + body_hash
        if (pendingRequests.contains(requestKey)) {
            return false; // 已有相同请求在进行中
        }
        pendingRequests.add(requestKey);
        return true;
    }

    public synchronized void onComplete(String requestKey) {
        pendingRequests.remove(requestKey);
    }
}
```

### 6.4 请求重试策略

```java
OkHttpClient client = new OkHttpClient.Builder()
    .retryOnConnectionFailure(true) // 连接失败自动重试（默认true）
    // 自定义拦截器实现更精细的重试
    .addInterceptor(new Interceptor() {
        @Override
        public Response intercept(Chain chain) throws IOException {
            Request request = chain.request();
            int maxRetries = 2;
            int retryCount = 0;
            IOException lastException = null;

            while (retryCount <= maxRetries) {
                try {
                    Response response = chain.proceed(request);
                    // 服务端返回 5xx 也重试
                    if (response.code() >= 500 && retryCount < maxRetries) {
                        response.close();
                        retryCount++;
                        continue;
                    }
                    return response;
                } catch (IOException e) {
                    lastException = e;
                    if (retryCount >= maxRetries) throw e;
                    retryCount++;
                    // 指数退避
                    try {
                        Thread.sleep((long) Math.pow(2, retryCount) * 1000);
                    } catch (InterruptedException ignored) {}
                }
            }
            throw lastException;
        }
    })
    .build();
```

```java
        dnsStart = System.currentTimeMillis();

    @Override
    public void dnsEnd(Call call, String domainName, List<InetAddress> list) {
        long cost = System.currentTimeMillis() - dnsStart;
        // 上报 dnsCost
    }

    @Override
    public void connectStart(Call call, InetSocketAddress address, Proxy proxy) {
        connectStart = System.currentTimeMillis();
    }

    @Override
    public void secureConnectStart(Call call) {
        tlsStart = System.currentTimeMillis();
    }

    @Override
    public void secureConnectEnd(Call call, Handshake handshake) {
        long tlsCost = System.currentTimeMillis() - tlsStart;
        // 上报 tlsCost
    }

    @Override
    public void connectEnd(Call call, InetSocketAddress address,
                           Proxy proxy, Protocol protocol) {
        long tcpCost = System.currentTimeMillis() - connectStart;
        // 上报 tcpCost（含TLS）
    }

    @Override
    public void requestBodyStart(Call call) {
        requestStart = System.currentTimeMillis();
    }

    @Override
    public void responseBodyEnd(Call call, long byteCount) {
        long requestCost = System.currentTimeMillis() - requestStart;
        // 上报 totalCost、byteCount
        // 计算网速: byteCount / requestCost
    }
}
```

## 十一、面试题

**Q1: 一个 APP 首页接口耗时 2 秒，如何排查优化？**

> 1. **量化分解**：通过 EventListener 拿到 DNS / TCP / TLS / TTFB / 传输 各阶段耗时
> 2. **DNS 慢** → 上 HTTPDNS + 预解析
> 3. **建连慢** → 检查连接池复用率，做连接预热
> 4. **TLS 慢** → 确认是否启用 Session Ticket / TLS 1.3
> 5. **TTFB 高** → 服务端问题，推动后端优化接口 / 加缓存
> 6. **传输慢** → 数据量是否过大？改用 Protobuf + GZIP，或分页

**Q2: HTTPDNS 原理？和 LocalDNS 有什么区别？**

> LocalDNS 走 UDP 协议到运营商 DNS 服务器，容易被劫持、调度不准。HTTPDNS 走 HTTPS 直连自建 DNS 服务，基于客户端真实 IP 做就近调度，绕过运营商，安全且准确。

**Q3: HTTP/2 相比 HTTP/1.1 做了哪些优化？为什么还不够好？**

> HTTP/2 引入多路复用（单连接多 Stream）、头部压缩（HPACK）、服务端推送。但它底层还是 TCP，TCP 层面的队头阻塞问题依然存在——一个 TCP 包丢失，该连接上所有 Stream 都会被阻塞。HTTP/3 基于 QUIC(UDP) 彻底解决了这个问题。

**Q4: OkHttp 连接池的工作原理？**

> OkHttp 内部维护一个 `RealConnectionPool`，按 host:port 分组。请求完成后连接不会立即关闭，而是放入空闲队列。下次同 host 的请求可以直接复用。空闲超过 keepAliveDuration（默认5分钟）或数量超过 maxIdleConnections（默认5）则清理。清理通过一个后台线程定期执行。

**Q5: 弱网环境下如何优化用户体验？**

> 分层策略：① 网络质量检测 → ② 自适应超时（弱网给更长等待）→ ③ 图片/视频降级 → ④ 减少单次请求数据量 → ⑤ 本地缓存兜底（先展示旧数据，新数据到了再更新）→ ⑥ 离线请求排队。

**Q6: 如何做网络请求的优先级调度？**

> 利用 OkHttp 的异步队列 + 自定义 Dispatcher。关键业务请求（如支付确认）直接进入 running 队列，低优先级请求（如日志上报、预加载）在 pending 队列中等待。页面退出时取消该页面未完成的请求，避免无效网络消耗。
