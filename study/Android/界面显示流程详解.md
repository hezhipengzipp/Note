# Android 界面显示流程详解

## 全景图

```
Activity.onCreate()
    │
    ▼
setContentView(R.layout.xxx)
    │
    ▼
XML 解析 → View 树创建
    │
    ▼
Activity.onResume()
    │
    ▼
WindowManager.addView()
    │
    ▼
ViewRootImpl 创建
    │
    ▼
requestLayout() 触发
    │
    ├── measure（测量每个 View 多大）
    ├── layout（确定每个 View 放在哪）
    └── draw（绘制内容）
    │
    ▼
Surface 提交给 SurfaceFlinger
    │
    ▼
SurfaceFlinger 合成所有图层
    │
    ▼
屏幕显示（VSync 信号驱动刷新）
```

---

## 一、Activity 启动与 View 树创建

```
ActivityThread.handleLaunchActivity()
    │
    ▼
Activity.onCreate()
    │
    ▼
setContentView(R.layout.activity_main)
    │
    ├── 1. LayoutInflater 解析 XML
    │      把每个标签实例化为对应的 View 对象
    │      <LinearLayout> → new LinearLayout()
    │      <TextView>     → new TextView()
    │
    ├── 2. 构建 View 树
    │      DecorView (FrameLayout)
    │        └── LinearLayout
    │              ├── ActionBar
    │              └── FrameLayout (android.R.id.content)
    │                    └── 你的布局（activity_main.xml）
    │
    └── 3. 此时只是创建了 View 对象，还没有测量、布局、绘制
         View 还不可见，宽高都是 0
```

**注意**：`onCreate()` 中调用 `view.getWidth()` 返回 0，因为还没有 measure。

```
Activity.onStart()   ← Activity 对用户"可见"（但还不能交互）
    │
    ▼
Activity.onResume()  ← Activity 可交互
    │
    ▼
此时才真正开始把 View 显示到屏幕上
```

## 二、Window 和 ViewRootImpl 的连接

`onResume()` 之后，`ActivityThread` 调用 `handleResumeActivity()`：

```
handleResumeActivity()
    │
    ▼
WindowManager.addView(decorView, layoutParams)
    │
    ▼
WindowManagerImpl.addView()
    │
    ▼
WindowManagerGlobal.addView()
    │
    ├── 1. 创建 ViewRootImpl
    │      ViewRootImpl 是 View 树的管理者
    │      连接 View 树和 WindowManagerService（WMS）
    │
    ├── 2. ViewRootImpl.setView(decorView)
    │      │
    │      ├── 向 WMS 注册窗口，获取 Surface
    │      │
    │      └── requestLayout()  ← 触发第一次绘制流程
    │
    └── 3. 关系建立：
         ViewRootImpl ←→ DecorView ←→ 你的 View 树
         ViewRootImpl ←→ WMS（跨进程 Binder）
         ViewRootImpl ←→ Surface（绘图画布）
```

**ViewRootImpl 是核心枢纽**：

```
┌──────────┐     ┌───────────────┐     ┌─────────────────┐
│  View 树  │ ←→ │ ViewRootImpl   │ ←→ │ WMS (系统服务)    │
│ (DecorView)│    │ - 调度测量布局绘制│    │ - 窗口管理        │
│            │    │ - 接收触摸事件  │    │ - 分配 Surface    │
│            │    │ - 管理 Surface │    │ - Z 轴排序        │
└──────────┘     └───────────────┘     └─────────────────┘
```

## 三、measure → layout → draw

`requestLayout()` 会向 Choreographer 注册一个 VSYNC 回调，等下一个 VSYNC 信号到来时执行：

```
VSync 信号到达
    │
    ▼
Choreographer.doFrame()
    │
    ▼
ViewRootImpl.performTraversals()    ← 整个绘制流程的入口
    │
    ├── 1. performMeasure()
    │      │
    │      └── DecorView.measure()
    │            └── 递归调用子 View 的 measure()
    │                 每个 View 确定自己的宽高
    │                 onMeasure() ← 你可以重写这个方法
    │
    ├── 2. performLayout()
    │      │
    │      └── DecorView.layout()
    │            └── 递归调用子 View 的 layout()
    │                 每个 View 确定自己的位置 (left, top, right, bottom)
    │                 onLayout() ← ViewGroup 重写这个方法来摆放子 View
    │
    └── 3. performDraw()
           │
           └── DecorView.draw()
                 │
                 ├── 硬件加速（默认）：
                 │    构建 DisplayList（绘制指令列表）
                 │    交给 RenderThread 用 GPU 渲染到 Surface
                 │
                 └── 软件绘制（fallback）：
                      Canvas 直接绘制到 Surface 的 Buffer
                      onDraw() ← 你在这里用 Canvas 画东西
```

### MeasureSpec

```
MeasureSpec = 模式 + 大小（32 位 int，高 2 位是模式，低 30 位是大小）

三种模式：
┌──────────────┬────────────────────────────────────┐
│ EXACTLY      │ 精确大小（match_parent 或具体 dp）     │
│ AT_MOST      │ 最大不超过（wrap_content）             │
│ UNSPECIFIED  │ 不限制（ScrollView 对子 View）         │
└──────────────┴────────────────────────────────────┘

父 View 的约束 + 子 View 的 LayoutParams → 子 View 的 MeasureSpec
```

## 四、Surface → SurfaceFlinger → 屏幕

```
View 绘制完成
    │
    ▼
内容写入 Surface 的 Buffer（GraphicBuffer）
    │
    ▼
Surface 通过 BufferQueue 提交 Buffer 给 SurfaceFlinger
    │
    ▼
SurfaceFlinger（系统进程）
    │
    ├── 收集所有窗口的 Layer（状态栏、导航栏、App 窗口...）
    ├── 按 Z 轴排序
    ├── 合成最终画面
    │    ├── GPU 合成（复杂场景）
    │    └── HWC 硬件合成（Hardware Composer，更高效）
    │
    ▼
显示到屏幕（下一个 VSync 信号时刷新）
```

### 双缓冲 / 三缓冲机制

```
┌─────────────────────────────────────────────────┐
│  App 绘制到 Back Buffer                          │
│  SurfaceFlinger 读取 Front Buffer 显示到屏幕      │
│  两者交替，避免画面撕裂                             │
│                                                  │
│  三缓冲：多一个 Buffer，App 不用等 SF 释放就能继续画 │
│  减少掉帧，但增加一帧延迟                           │
└─────────────────────────────────────────────────┘
```

## 五、VSync 机制

```
VSync 信号（60Hz 屏幕 = 每 16.6ms 一次）
    │
    ├── 通知 App：该绘制下一帧了（Choreographer 回调）
    │
    └── 通知 SurfaceFlinger：该合成并显示了

一帧的时间线（16.6ms）：
┌──────────────────┬──────────────────┐
│  App 绘制第 N+1 帧 │  SF 合成第 N 帧    │  ← 流水线并行
└──────────────────┴──────────────────┘
VSync                                VSync
```

## 六、完整时间线总结

```
时间 ──────────────────────────────────────────────────────────→

onCreate()         onResume()      VSync到达         VSync到达
    │                  │              │                │
    ▼                  ▼              ▼                ▼
 创建 View 树    ViewRootImpl     measure/layout   SurfaceFlinger
 (XML → 对象)    连接 WMS          /draw            合成显示
                 申请 Surface     渲染到 Surface     屏幕刷新
                                                      │
                                                      ▼
                                                  用户看到画面
```

**从 `onCreate()` 到用户看到画面，至少经历 2-3 帧（约 32-48ms）**。

## 七、面试高频问题

### Q1: 为什么 onCreate() 里获取不到 View 的宽高？

- `onCreate()` 只是创建了 View 对象，measure 要等到 `onResume()` 之后 ViewRootImpl 触发 `performTraversals()` 才执行
- 获取宽高的正确方式：`view.post { view.width }` 或 `ViewTreeObserver.OnGlobalLayoutListener`

### Q2: onResume() 之后界面就可见了吗？

- 不是。`onResume()` 之后才开始创建 ViewRootImpl、申请 Surface、执行 measure/layout/draw
- 真正可见要等 draw 完成并且 SurfaceFlinger 合成后的下一个 VSync
- `onResume()` 只是表示 Activity 进入"可交互"状态

### Q3: View.post() 为什么能获取到宽高？

```kotlin
view.post {
    // 此时 measure/layout 已完成
    val width = view.width
}
```

- `post()` 把 Runnable 放到 View 关联的 Handler 消息队列中
- 如果 View 已经 attach 到窗口，Runnable 会在当前消息（包含 performTraversals）执行完之后执行
- 如果 View 还没 attach，Runnable 会暂存，等 `performTraversals()` 完成后再执行
- 所以执行时 measure/layout 一定已经完成

### Q4: invalidate() 和 requestLayout() 的区别？

| | `invalidate()` | `requestLayout()` |
|--|----------------|-------------------|
| 触发 | 只触发 draw | 触发 measure + layout + draw |
| 场景 | 内容变了（颜色、文字） | 大小或位置可能变了 |
| 性能 | 轻量 | 较重（整棵树重新测量布局） |

### Q5: 硬件加速和软件绘制的区别？

| | 硬件加速 | 软件绘制 |
|--|---------|---------|
| 渲染器 | GPU（RenderThread） | CPU（主线程） |
| 绘制方式 | 构建 DisplayList，GPU 执行 | Canvas 直接画像素 |
| 性能 | 快，不阻塞主线程 | 慢，阻塞主线程 |
| 局部刷新 | 只重绘脏区域的 DisplayList | 需要重绘整个 View |
| 默认 | Android 4.0+ 默认开启 | 需要手动关闭硬件加速 |

### Q6: 掉帧（卡顿）发生在哪个环节？

```
一帧 16.6ms 的时间分配：
├── 主线程：Input 处理 + 动画 + measure + layout + draw 构建 DisplayList
├── RenderThread：GPU 渲染
│
│  如果主线程超过 16.6ms → 掉帧
│  常见原因：
│  - 布局层级太深（measure/layout 慢）
│  - onDraw() 里做了耗时操作
│  - 主线程有其他耗时任务（IO、计算）
│  - 过度绘制（同一像素被画多次）
```
