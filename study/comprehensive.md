## 一、计算机网络（必考）

### 1. HTTP / HTTPS

#### HTTP 请求流程
```
DNS 解析 → TCP 三次握手 → (TLS 握手) → 发送 HTTP 请求 → 接收响应 → (四次挥手)
```

#### HTTP 版本对比

| 版本 | 特点 |
|------|------|
| HTTP/1.0 | 短连接，每次请求新建 TCP |
| HTTP/1.1 | 长连接（Keep-Alive）、管线化（但有队头阻塞） |
| HTTP/2 | 多路复用、头部压缩、二进制帧、服务端推送 |
| HTTP/3 | 基于 QUIC（UDP），解决 TCP 队头阻塞 |

#### HTTPS = HTTP + TLS

```
Client                              Server
  │                                    │
  │ ──── ClientHello ──────────────>   │  支持的 TLS 版本、加密套件
  │ <─── ServerHello ──────────────    │  选定的加密套件 + 证书
  │                                    │
  │  验证证书（CA 链）                   │
  │  生成 Pre-Master Secret            │
  │  用服务端公钥加密                    │
  │ ──── 加密的 Pre-Master ────────>   │
  │                                    │
  │  双方各自计算 Master Secret          │
  │  生成对称加密密钥                    │
  │                                    │
  │ <═══ 对称加密通信 ═══════════════>  │
```

**核心**：非对称加密交换密钥，对称加密传输数据

#### 常见状态码

| 状态码 | 含义 |
|--------|------|
| 200 | 成功 |
| 301 | 永久重定向 |
| 302 | 临时重定向 |
| 304 | 未修改（走缓存） |
| 400 | 请求错误 |
| 401 | 未认证 |
| 403 | 禁止访问 |
| 404 | 未找到 |
| 500 | 服务器内部错误 |
| 502 | 网关错误 |
| 503 | 服务不可用 |

### 2. TCP vs UDP

| | TCP | UDP |
|------|-----|-----|
| 连接 | 面向连接（三次握手） | 无连接 |
| 可靠性 | 可靠（确认重传、流量控制、拥塞控制） | 不可靠 |
| 有序 | 有序 | 无序 |
| 速度 | 较慢 | 快 |
| 场景 | HTTP、文件传输 | 视频直播、DNS、游戏 |

#### TCP 三次握手

```
Client              Server
  │ ── SYN ────────> │    1. 客户端发起连接
  │ <─ SYN+ACK ──── │    2. 服务端确认并发起连接
  │ ── ACK ────────> │    3. 客户端确认
```

**为什么要三次？** 防止已失效的连接请求到达服务端，导致服务端误开资源。

#### TCP 四次挥手

```
Client              Server
  │ ── FIN ────────> │    1. 客户端请求关闭
  │ <─ ACK ────────  │    2. 服务端确认（可能还有数据要发）
  │ <─ FIN ────────  │    3. 服务端数据发完，请求关闭
  │ ── ACK ────────> │    4. 客户端确认（等待 2MSL 后关闭）
```

### 3. DNS 解析

```
浏览器缓存 → 系统缓存(hosts) → 路由器缓存 → ISP DNS → 根域名 → 顶级域名 → 权威域名
```

### 4. Cookie / Session / Token

| | 存储位置 | 适用场景 |
|------|---------|---------|
| Cookie | 客户端 | 自动携带，有大小限制 |
| Session | 服务端（Cookie 存 SessionID） | 传统 Web |
| Token（JWT） | 客户端 | 无状态，适合移动端和分布式 |

## 二、设计模式（结合 Android 源码）

### 高频设计模式

| 模式 | Android 中的实例 | 核心思想 |
|------|-----------------|---------|
| 观察者 | LiveData、RxJava、BroadcastReceiver | 一对多依赖，状态变化通知 |
| 单例 | Application、各种 Manager | 全局唯一实例 |
| 工厂方法 | BitmapFactory、Fragment.instantiate | 创建对象不暴露逻辑 |
| 建造者 | AlertDialog.Builder、OkHttpClient.Builder | 分步骤构建复杂对象 |
| 代理 | Retrofit（动态代理）、Binder（远程代理） | 控制对象访问 |
| 策略 | 动画插值器、RecyclerView.LayoutManager | 封装算法族，可互换 |
| 责任链 | OkHttp 拦截器、View 事件分发 | 请求沿链传递 |
| 适配器 | RecyclerView.Adapter、ListView.Adapter | 接口转换 |
| 模板方法 | Activity 生命周期、AsyncTask | 定义骨架，子类实现步骤 |
| 装饰器 | ContextWrapper、InputStream 体系 | 动态添加功能 |

### 六大设计原则（SOLID + 迪米特）
- **S** 单一职责：一个类只做一件事
- **O** 开闭原则：对扩展开放，对修改关闭
- **L** 里氏替换：子类可以替换父类
- **I** 接口隔离：接口最小化
- **D** 依赖倒置：依赖抽象而非具体
- **迪米特法则**：最少知道，降低耦合

#### 3. 树
- 二叉树前/中/后序遍历（递归 + 迭代）
- 层序遍历（BFS + 队列）
- 最大/最小深度

#### 4. 动态规划

- 最长递增子序列（LIS）
- 最长公共子序列（LCS）

- 快排（O(nlogn)，不稳定）
- 归并排序（O(nlogn)，稳定）

#### 6. 常考数据结构
- **HashMap**：数组+链表+红黑树，O(1) 查找
- **LRU Cache**：HashMap + 双向链表，Android 的 LruCache
- **栈和队列**：括号匹配、最小栈

## 四、架构设计（资深考察重点）

### 1. App 架构模式

| 模式 | 特点 | 缺点 |
|------|------|------|
| MVC | 简单，Activity 既是 C 也是 V | Activity 臃肿 |
| MVP | 接口隔离，Presenter 处理逻辑 | 大量接口，Presenter 也会臃肿 |
| MVVM | 数据绑定，ViewModel + LiveData/Flow | 学习曲线 |
| MVI | 单向数据流，状态统一管理 | 模板代码多 |

当前主流：**MVVM + Clean Architecture**

### 2. 组件化 / 模块化

```
┌─────────────────────────────────────┐
│              App Shell               │
├──────┬──────┬──────┬────────────────┤
│ 首页  │ 商城  │ 消息  │  我的          │
│Module│Module│Module│  Module        │
├──────┴──────┴──────┴────────────────┤
│           公共业务层                   │
│   （登录、支付、分享、推送）              │
├─────────────────────────────────────┤
│           基础架构层                   │
│   （网络、图片、路由、日志、埋点）        │
└─────────────────────────────────────┘
```

**关键问题**：
- **页面路由**：ARouter / 自研路由，通过 URI 跳转，解耦模块依赖
- **模块间通信**：接口下沉到公共层 + 实现注册（SPI / Hilt）
- **独立编译运行**：application / library 切换

### 3. 系统设计面试题

**设计一个图片加载库**：
- 三级缓存：内存（LRU）→ 磁盘 → 网络
- 线程池管理下载任务
- 图片压缩和采样
- 生命周期绑定，防止泄漏
- BitmapPool 复用

**设计一个长连接保活方案**：
- WebSocket / 自定义 TCP 协议
- 心跳机制：智能心跳（根据网络环境动态调整间隔）
- 重连策略：指数退避
- 前台：正常心跳；后台：JobScheduler / WorkManager 定期唤醒
- 多进程：独立进程运行长连接服务

**设计一个启动框架**：
- DAG（有向无环图）表达任务依赖
- 拓扑排序确定执行顺序
- 线程池并行执行无依赖任务
- 主线程任务同步等待（CountDownLatch）
- 支持锚点任务（某些任务必须在某个生命周期前完成）

## 七、软技能面试

### STAR 法则整理项目经验
- **Situation**：背景，项目处于什么阶段
- **Task**：你的职责和目标
- **Action**：你具体做了什么
- **Result**：结果和数据（量化）

### 建议准备的项目经验
1. **一个性能优化案例**：有数据对比（启动时间从 X 降到 Y）
2. **一个架构升级案例**：为什么要升级、如何推动、踩了什么坑
3. **一个疑难 Bug 排查**：排查思路和定位过程
4. **一个跨端/跨团队协作**：体现沟通协调能力
