# WebRTC 信令详解：Janus 到底在“信”什么？

## 1. 核心定义：什么是“信令”？

在 WebRTC 的世界里，**“信令”就是帮助两个设备在正式视频通话前，“相互认识”并“商量好怎么连接”的一套“暗号”和流程。**

它不传输实际的音视频数据，而是负责“打前站”，解决点对点通信中最棘手的三个问题：寻址、协商和建立连接。

## 2. 为什么需要信令服务器？（三大核心问题）

WebRTC 的目标是实现浏览器之间的 **点对点（P2P）** 直接通信。但在直接“私聊”之前，两个陌生的浏览器必须通过一个“中间人”来解决以下问题：

| 问题 | 描述 | 解决方案（信令负责的部分） |
| :--- | :--- | :--- |
| **1. 找到对方** | 在浩瀚的网络中，Browser A 怎么知道 Browser B 的地址？ | 提供一个“号码簿”服务，让双方交换网络地址。 |
| **2. 协商“语言”** | A 支持 H.264，B 支持 VP8，它们需要确定一种共通的编解码器。 | 交换 **SDP** (Session Description Protocol) 信息。 |
| **3. 交换“住址”** | 大部分设备在 NAT/防火墙后，有好几个可能的网络地址（本地IP、公网IP等）。 | 交换 **ICE Candidates**（通过 STUN 服务器找到的候选地址）。 |

## 3. 信令服务器的具体工作流程

信令服务器就像一个“接线员”，其工作流程分为三个阶段：

1.  **交换“能力”（SDP）**：A 和 B 将各自的媒体能力（编解码器、分辨率等）打包成 SDP “简历”，通过服务器交换。
2.  **交换“地址”（ICE Candidate）**：A 和 B 各自通过 STUN 服务器找到自己的网络地址（Candidate），再通过服务器交换。
3.  **帮助建立连接**：当双方都有了对方的“简历”和“地址”后，开始尝试直接建立 P2P 连接。

> **关键点**：一旦连接成功建立，信令服务器的任务就完成了。后续的音视频数据直接在 A 和 B 之间的 P2P 通道传输，不再经过服务器。
