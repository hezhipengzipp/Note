**查看方式**
- **Android Studio / IntelliJ**：安装 `PlantUML Integration` 插件后，打开任一 `.puml` 文件，右侧即可看图
- **VS Code**：安装 `PlantUML` 扩展（jebbs），打开 `.puml` 后 `Alt+D` 预览
- **依赖**：`brew install graphviz`（首次使用必须）

## 1. 模块架构

📎 [01_模块架构.puml](./diagrams/01_模块架构.puml)

SDK 分五层：**对外层 → 业务层 → 信令层 → 媒体层 + 网络层**。

- **对外层 `RtcInit`**：对业务方暴露统一 API（`initialize / videoCall / hangup / close / switchMicrophone` 等）
- **业务层 `JanusVideoCallUseCase`**：编排 SSE → Janus → WebRTC 的全流程
- **信令层 `JanusClient + WebSocketChannel`**：封装 Janus 协议（create / attach / register / call / trickle / hangup）
- **媒体层 `RTCPeerConnection + DataChannelHandler + KeepAliveChannel`**：管理音视频流、业务消息、心跳
- **网络层**：SSE 用于首次握手下发 token/wss；HttpEngin 用于挂断通知和状态查询

---

## 2. API 入口与生命周期

📎 [02_API入口与生命周期.puml](./diagrams/02_API入口与生命周期.puml)

`initialize` 阶段一次性完成 PeerConnection 构建、本地媒体采集、事件绑定、DataChannel 创建，之后仅通过 `videoCall` / `hangup` 驱动状态变化。

关键点：
- 本地音频在 init 阶段即获取并 addTrack，不是每次通话重新采集
- DataChannel 在 init 阶段由主叫侧创建（id=0 有序、id=1 无序），对端自动通过 SCTP 协商接入

---

## 3. 完整通话全景时序

📎 [03_完整通话流程.puml](./diagrams/03_完整通话流程.puml)

端到端涉及 8 个参与者。核心阶段：

1. **SSE 握手**：拉取 `token / wssUrl / randUserFrom / randUserTo / hashId / connId`
2. **Janus 会话**：`create → attach(videocall) → register(localName)`，每步 15s 超时
3. **呼叫**：本地 offer → Janus 转发 → 对端 accept → 回传 answer
4. **ICE 协商**：trickle candidate 逐个上报 + complete 结束
5. **媒体就绪**：`webrtcup` → `media:receiving=true` → Connected

## 4. SSE 连接与消息分支

📎 [04_SSE连接与消息分支.puml](./diagrams/04_SSE连接与消息分支.puml)
📎 [04.1_SSE成功数据处理.puml](./diagrams/04.1_SSE成功数据处理.puml)

容错策略：
- `onError + statusCode=200`：延迟 3s 重连，最多 5 次
- `onError + 其它`：直接 `DisConnected(SSE Fail:code)`
- `onConnectionClose`：延迟 500ms，若此时 `connId` 仍为空则上报断开（解决"close 消息比最后一条数据先到"的时序问题，对应 commit `b2d14b8`）

4.1 中展示了收到成功数据后如何根据 `status` 派发 `JanusConnect` / `JanusCall` 任务。

## 5. Janus 信令状态机

📎 [05_Janus信令状态机.puml](./diagrams/05_Janus信令状态机.puml)

主链路状态单向前进。重连特例：
- Socket 重开 → `_sessionId` 存在则 `claim` 而非 `create`
- 连接计数 > 1 → 回调 `onSocketReopen` → 触发 `restartOffer + resumeAlive`

## 6. WebRTC 媒体协商

📎 [06_WebRTC媒体协商.puml](./diagrams/06_WebRTC媒体协商.puml)
📎 [06.1_重协商_Socket重连.puml](./diagrams/06.1_重协商_Socket重连.puml)
📎 [06.2_重协商_UPDATE事件.puml](./diagrams/06.2_重协商_UPDATE事件.puml)

- 主流程使用 Trickle ICE：candidate 逐个发送，`gatheringState=Complete` 时发送 `{completed:true}` 收尾
- `onTrack` 对每路 video 流创建独立 `RTCVideoRenderer`，存入 `remoteVideoMap[streamId]`
- 重协商有两种触发路径：**Socket 重连** 主动 `restartOffer`；**收到 EVENT UPDATE** 被动 setRemoteDescription + createAnswer

## 7. 数据通道（业务 + 心跳）

📎 [07_数据通道.puml](./diagrams/07_数据通道.puml)
📎 [07.1_心跳异常重协商.puml](./diagrams/07.1_心跳异常重协商.puml)

两条 DataChannel 物理隔离：
- `id=0` 有序可靠：用于业务消息（麦克风状态、自定义文本）
- `id=1` 无序：每 2s 心跳，`_issuelessStat` 计数对端无回应次数
  - ≥3：显示 loading 提示 UI
  - ≥10：触发 `onHeartbeatTimeout` → `DisConnected(OTHER, KeepAliveTimeout)`

## 8. 超时与重试策略

📎 [08_超时与重试策略.puml](./diagrams/08_超时与重试策略.puml)
📎 [08.1_呼叫超时任务.puml](./diagrams/08.1_呼叫超时任务.puml)

四层超时叠加：**SSE(5次×3s) → Janus信令(15s/步) → 业务呼叫(15/35s) → 媒体协商(8s) → 心跳(10次×2s)**。

`_startCallingTimeoutTask` 每 5s 轮询后端 `queryDeviceAcceptStatus`，既能在服务端已判定结果时立即返回，也能兜底兜超时。

## 9. 错误码与 DisConnected

📎 [09_错误码与DisConnected.puml](./diagrams/09_错误码与DisConnected.puml)

所有异常最终收敛到 `DisConnected(code, msg)`。特殊处理：
- `USER_HANGUP` → 对外转为 `USER_REFUSE`
- `SSE_TIMEOUT` → 根据远端是否已唤醒，分别映射为 `INIT_FAILURE` / `CALLEE_OFFLINE`
- `JANUS_UNAUTHORIZED`（403）→ 使用 `oldToken` 携带 `forceNewToken=true` 自动重发 `requestVideoCall`

## 10. 媒体控制与业务消息

📎 [10_媒体控制与业务消息.puml](./diagrams/10_媒体控制与业务消息.puml)

- `switchMicrophone`：切换本地音轨 `enabled` + 通过 DataChannel 下发状态给对端
- `switchVoice(cameraId)`：按摄像头 id 切换远端音频轨 `enabled`
- `sendMessage` + `setOnMessageListener`：业务自定义消息双向通道

## 11. 挂断与销毁流程

📎 [11_挂断与销毁流程.puml](./diagrams/11_挂断与销毁流程.puml)

- `hangup`：仅发 Janus message{hangup}，保留连接资源，等待对端响应
- `close(reason)`：完整销毁 — 停 timer、通知服务端 HTTP hangup、close PC、destroySession、关闭 WS、关闭 DataChannel
- 被动挂断：收到 Janus HANGUP 后先查后端状态，按结果分类派发事件

## 12. 对外事件状态机

📎 [12_对外事件状态机.puml](./diagrams/12_对外事件状态机.puml)

UI 层只需订阅 `RtcInit.videoCallEvent` 即可驱动整条通话生命周期：

```
     → Connecting → StreamOpen → Connected
     → DisConnected(code, msg)
```

## 14. 总结

整套 SDK 在一次通话中经历四个阶段：

1. **握手阶段**：SSE 拉取 Janus 接入信息（token、wss、hashId、对端用户名），带 5 次重连兜底。
2. **信令阶段**：Janus WebSocket 建立 session → attach videocall 插件 → register 本地用户；每步 15s 超时，10s 心跳保活。
3. **媒体阶段**：createOffer → trickle ICE → 收到 answer → setRemoteDescription → onTrack 渲染 → webrtcup + media:receiving 完成建连；协商 8s 超时。
4. **保活与运行**：DataChannel(id=0) 传业务消息、KeepAliveChannel(id=1) 每 2s 心跳，异常 ≥10 次触发挂断；Socket 重连时 restartOffer。

异常在任意环节均回落为 `DisConnected(code, msg)`，对外统一暴露事件状态机，调用方只需监听 `RtcInit.videoCallEvent` 即可驱动 UI。
