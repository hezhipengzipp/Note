# 简历关键词逐个拆解

> 目标：把简历里那句「基于 Janus Gateway + flutter_webrtc 自研车机 1v1 实时视频通话 SDK：设计 SSE 下行推呼叫状态 + WebSocket 上行 Janus 信令的非对称信令架构，完成 Session/Handle 三级寻址、SDP Offer/Answer + Trickle ICE（host/srflx/relay）NAT 穿透；在 WebRTC DataChannel 上独立实现...」拆成 8 个独立概念讲透，按**从问题到解决方案**的逻辑顺序。

---

## 概念 1：什么是 Janus？

**Janus** = 一个开源的 **WebRTC 信令服务器**（C 语言写的）。

### 形象理解

把它当成**电话交换机**：

- 你（客户端 A）不知道对方（客户端 B）的网络地址
- 你们俩都先连到这个"交换机"（Janus）
- Janus 帮你们转发 SDP、ICE、挂断等协议消息
- 一旦两端通过 ICE 打通，**媒体流直连两端，不再经过 Janus**

### 本项目用什么

用的是 Janus 的 **`videocall` 插件**——专门处理 1v1 视频通话。Janus 还有很多其他插件（多方会议 `videoroom`、直播推流 `streaming`、SIP 网关 `sip`），本项目只用了一个。

### 代码位置

`lib/janus/janus_client.dart` 就是和 Janus 对话的客户端。

## 概念 2：什么是"非对称信令架构"？

"非对称"指的是**上行和下行走不同的通道**：

```
业务服务器 ─── SSE ───→ 客户端      （下行，单向推送）
客户端    ─── WebSocket ──→ Janus  （上行，双向）
```

### 对比：对称方案

一条 WebSocket 双向搞定所有事。

### 为什么本项目非对称

- **SSE**（基于 HTTP 长连接）比 WebSocket **更容易穿透 CDN / 反向代理**，适合被叫端常驻监听"有人要呼叫我了吗"
- **WebSocket** 留给真正需要双向密集交互的 Janus 协议（SDP/ICE 频繁往返）

### 类比

- **SSE** 像**电视广播**——电视台发信号，家家户户的电视都能收，单向
- **WebSocket** 像**电话**——可以打进来也可以打出去，双向

### 代码位置

- SSE：`janus_video_call_usecase.dart:_startSSEConnect`
- WebSocket：`wss/web_socket_channel.dart`

## 概念 3：Session/Handle 三级寻址

Janus 协议里一个消息要带**三层身份**才能路由到对的地方：

```
session_id    → 哪个客户端
  └── handle_id   → 这个客户端挂了哪个插件实例
        └── body.request  → 插件里的哪个具体动作
```

### 类比：去医院看病

- **session_id** = 你的病历号（"我是张三"）
- **handle_id** = 你挂了哪个科的号（"我挂了骨科 3 号诊室"）
- **body.request** = 具体要做什么（"我要拍 X 光"）

### 本项目对应

- `_sessionId` = 客户端与 Janus 的会话 ID（`createSession` 拿到）
- `_handlerId` = 客户端在 videocall 插件上的实例 ID（`attachPlugin` 拿到）
- `body.request` = `register` / `call` / `hangup` / `set` 等具体动作

### 为什么要三级

一个客户端（一个 session）可能同时挂多个 Janus 插件（多个 handle），比如一边打视频通话一边听广播。如果只有 `session_id`，服务器不知道消息是给哪个插件处理的。

### 代码位置

每条 `_channel.sendMessage(obj)` 里都能看到 `session_id` + `handle_id` 两个字段。

## 概念 4：SDP Offer / Answer

**SDP**（Session Description Protocol）= 一段**文本**，描述"我这边能接受什么格式的媒体"。

### Offer/Answer 就是规格谈判

```
主叫 → "我支持 opus 音频、H264 视频、VP8 视频"  （Offer）
被叫 → "那我们用 opus + H264 吧（VP8 我不支持）" （Answer）
```

谈完后两端知道要用什么 codec、用什么密钥、走哪个端口。

### 代码位置

- 生成 Offer：`doCall` 里 `_peerConnection.createOffer()`
- 发送：`_janusClient.call(handleId, sdp, remoteName)`
- 接收 Answer：`onCallAccepted` 里 `_peerConnection.setRemoteDescription(sdp)`

### 重要前提

SDP 只是协商规格，**协商完还不能通话**——还要解决"怎么找到对方"的问题，就是下面的 ICE。

## 概念 5：NAT 穿透是啥难题？

**NAT** = 你家路由器把你的内网 IP（`192.168.x.x`）转换成公网 IP 的机制。

### 问题场景

```
你家                      对方家
192.168.1.5      ???     10.0.0.8
      │                       │
   [家用路由器]           [家用路由器]
      │                       │
      └───────── 互联网 ──────┘
```

两个内网设备**互相看不到对方的 IP**，没法像服务器那样"你直接连我 IP 端口就行"。

### 解决方案

就是下面讲的 **ICE**。

## 概念 6：ICE 和三种 Candidate

**ICE**（Interactive Connectivity Establishment）= WebRTC 找路径的**框架**，它会列出多条可能的通路，一条条试。

### 三种"候选地址"（Candidate）

| 类型 | 地址来源 | 啥时候能通 |
| --- | --- | --- |
| **host** | 你本机网卡 IP（比如 192.168.1.5）| 两端在同一个局域网时 |
| **srflx**（server reflexive）| 经 **STUN 服务器**探测到的你的公网 IP | 大多数家用 NAT 能穿透 |
| **relay** | **TURN 服务器**的地址（中继转发）| 前两种都不行时的兜底 |

### 类比：两个人要见面

- **host** = "来我办公室"（同公司可以）
- **srflx** = "来我家小区门口"（外人可以到）
- **relay** = "我们在咖啡馆见"（谁都能去，但要付咖啡钱 = 带宽成本）

### 协商过程

```
本端                                   对端
 │                                      │
 │ 收集到 3 个 candidate（host/srflx/relay）
 │ ─── 通过 Janus 转发给对端 ─────────→ │
 │ ← 对端也收集 3 个发回来 ──────────── │
 │                                      │
 │ ═══ 两端互相 ping 所有组合 ═══════  │
 │ ═══ 选出能通且延迟最低的一条 ═════  │
```

### 代码位置

- 本端收集：`onIceCandidate` 回调 → `_janusClient.trickleCandidate()`
- 接收对端：`onIceCandidate(handleId, candidate)` → `_peerConnection.addCandidate()`

## 概念 7：什么是 Trickle ICE？

**Trickle** = **"涓滴"**，意思是"一点点流出来"。

### 对比两种 ICE 模式

- **传统 ICE**：等所有 candidate 都收集完（可能 2~5 秒），打包塞进 SDP 一起发
- **Trickle ICE**：每收集到一个 candidate 就**立刻单独**发给对端

### 本项目用 Trickle

```dart
// 每发现一个 candidate 就立刻 trickle
_peerConnection.onIceCandidate = (candidate) {
  _janusClient.trickleCandidate(_handlerId!, candidate);
};

// 全部收集完再发个 completed:true 告诉对方"没有更多了"
_peerConnection.onIceGatheringState = (state) {
  if (state == RTCIceGatheringStateComplete) {
    _janusClient.trickleCandidateComplete(_handlerId!);
  }
};
```

### 好处

对端提前几秒就能开始尝试打洞，**建连时间能省 2~5 秒**。

## 概念 8：WebRTC DataChannel

**DataChannel** = WebRTC 除了音视频外，**还能传任意二进制数据**的一条 P2P 通道。

### 特点

- **端到端**（E2E）：两个客户端直连，不经过服务器
- **加密**：自动走 DTLS 加密
- **可配置顺序**：可选 `ordered`（保序）或 `unordered`（不保序）

### 本项目建了两条

| 通道 | 用途 | 配置 |
| --- | --- | --- |
| **业务 DataChannel** | 发麦克风状态、自定义消息 | `ordered=true`（要保序）|
| **KeepAlive DataChannel** | 2s 一次心跳保活 | `ordered=false`（心跳丢了无所谓）|

### 为什么要自己跑心跳

WebRTC 底层能告诉你"网络通不通"，但**告诉不了你"对端的 App 还活着吗"**。比如对端 App 卡死了，网络层还显示通，但业务已经不响应了——只有业务心跳才能检测到。

### 代码位置

- `data_channel_handler.dart`（业务通道）
- `keep_alive_channel.dart`（心跳通道）
