# 专业术语解释

本文档解释本项目涉及的所有关键技术术语，分五类：**信令传输**、**WebRTC 核心**、**Janus 协议**、**SDP / ICE 协商**、**本项目特有概念**。每个术语给出：定义、在本项目中的位置、关键代码引用。

---

## 一、信令传输类

### SSE（Server-Sent Events）

**定义**：基于 HTTP 长连接的**单向**服务器推送协议（`Content-Type: text/event-stream`）。客户端发一次 GET 请求，服务器保持连接不断开，持续向客户端推送文本事件，直到一方关闭。

**与 WebSocket 的区别**：
- SSE 只能服务器 → 客户端，单向；WebSocket 双向
- SSE 走标准 HTTP，天然穿透 CDN / 反向代理；WebSocket 需要 Upgrade 握手
- SSE 自带重连机制（浏览器原生支持 `EventSource`）；WebSocket 要自己实现

**在本项目**：业务服务器 → 客户端的**呼叫状态推送通道**。通过第三方库 `eventflux` 接入。
- 建立：`janus_video_call_usecase.dart:_startSSEConnect()`
- 解析：`_parseSseData()` 按 `code` 分发 13 种业务消息
- 关键消息：`SSE_SUCCESS(200)` 下发 token / wssUrl / connId，推进 Janus 握手

### WebSocket（wss）

**定义**：基于 TCP 的**全双工**通信协议，通过 HTTP Upgrade 握手升级。`ws://` 明文、`wss://` TLS 加密。

**在本项目**：
- **Janus 信令主通道**——客户端 ↔ Janus Gateway 的双向 JSON 消息通道
- 使用 `dart:io` 的原生 `WebSocket`，封装在 `lib/wss/web_socket_channel.dart`
- 握手头：
  - `Sec-WebSocket-Protocol: janus-protocol`（必需，Janus 子协议标识）
  - `x-hash-id: <consistentHashId>`（集群一致性哈希路由）

### HTTP

**定义**：标准请求-响应协议，无状态。

**在本项目**：业务上行的**独立通道**，由宿主 App 注入实现（`HttpEngin` 抽象接口），只用于三类操作：
- `hangup()` — 挂断上报
- `queryDeviceAcceptStatus()` — 轮询对端接听状态
- `uploadVideoCallStatus()` — 埋点

## 二、WebRTC 核心类

### WebRTC（Web Real-Time Communication）

**定义**：W3C 标准的浏览器 / 客户端实时音视频通信技术，包含三个主要 API：
- `MediaStream` — 媒体采集与传输
- `RTCPeerConnection` — P2P 连接管理
- `RTCDataChannel` — 任意数据点对点传输

**底层协议栈**：UDP + ICE（打洞）+ DTLS（握手加密）+ SRTP（媒体加密）+ SCTP（DataChannel）

**在本项目**：通过 `flutter_webrtc` 库使用，见 [信令通道架构说明.md](./信令通道架构说明.md)。

### PeerConnection（RTCPeerConnection）

**定义**：WebRTC 的核心对象，代表一条**端到端的媒体 / 数据连接**。负责：
1. SDP 协商（offer / answer）
2. ICE 打洞（candidate 交换）
3. DTLS 握手
4. 媒体编解码 + SRTP 加密传输

**状态机**（4 套，互相独立）：
- `signalingState` — SDP 协商阶段（stable / have-local-offer / have-remote-offer）
- `iceGatheringState` — ICE candidate 收集状态
- `iceConnectionState` — ICE 连通性状态（connected / disconnected / failed）
- `connectionState` — 整体连接状态（聚合以上三个）

**在本项目**：`JanusVideoCallUseCase._peerConnection`
- 创建：`createPeerConnection(ICE_SERVERS)` (:76)
- 添加本地流：`addTrack(track, stream)` (:115)
- 接收远端流：`onTrack` 回调 (:568)

### MediaStream / MediaStreamTrack

**定义**：
- `MediaStream` — 媒体流容器，装多条 track（音频 / 视频）
- `MediaStreamTrack` — 单条媒体轨，代表一路音频或一路视频

**采集方式**：`navigator.mediaDevices.getUserMedia({audio: true, video: true})`

**在本项目**：
- `_localSteam` — 本端麦克风采集流（只有 audio track，不采摄像头）
- `onTrack` 回调里的 `rtcTrackEvent.streams[0]` — 远端传过来的流

### RTCVideoRenderer

**定义**：`flutter_webrtc` 提供的渲染载体，内部是一个 Flutter Texture（硬件加速贴图）。用法：
1. `await renderer.initialize()` — 初始化 textureId
2. `renderer.srcObject = stream` — 绑定 MediaStream
3. 把 `RTCVideoView(renderer)` 作为 Widget 插入 UI 树

**为什么不直接渲染 MediaStream**：Flutter UI 框架是声明式 Widget 树，WebRTC 的视频是原生层 Texture，中间需要一个桥接对象串起来，这个对象就是 `RTCVideoRenderer`。

**在本项目**：
- `remoteAudio` — 远端音频 renderer（无可视画面，仅承载 audio track）
- `remoteVideoMap: Map<String, RTCVideoRenderer>` — 多路摄像头，cameraId → renderer
- 在 `onTrack` 里按 `track.kind == "video"` 创建 renderer 并绑定 (`:583-587`)
- UI 层通过 `RtcRenderView(renderer)` 薄包装上屏

### RTCDataChannel

**定义**：基于 SCTP 协议、在 PeerConnection 上建立的**任意二进制数据**传输通道，端到端加密，**不经服务器中转**。

**关键配置**：
- `ordered: bool` — 是否保序（true 类似 TCP，false 类似 UDP）
- `id: int` — SCTP 流 ID（手动指定避免冲突，或由 SDP 协商决定）

**在本项目**：建了**两条**：

| 通道 | id | ordered | 用途 | 文件 |
| --- | --- | --- | --- | --- |
| `dataChannel` | 0 | true | 业务消息（麦克风状态同步、自定义消息）| `data_channel_handler.dart` |
| `keepAliveChannel` | 1 | false | P2P 心跳保活（每 2s 一次）| `keep_alive_channel.dart` |

### RTCIceCandidate

**定义**：ICE 候选项，描述**一个可能的网络地址**（IP:端口 + 协议 + 优先级），用于两端尝试打洞。

三种类型：
- **host** — 本机网卡地址
- **srflx**（server reflexive）— 经 STUN 服务器看到的公网地址
- **relay** — 经 TURN 服务器中转的地址（兜底）

**三元组**：`candidate + sdpMid + sdpMLineIndex` 定位到 SDP 中某一路媒体的打洞候选。

**在本项目**：通过 Janus `trickle` 消息转发（`janus_client.dart:trickleCandidate`）。

### RTCRtpTransceiver

**定义**：RTP 收发器，管理 PeerConnection 上**一路**媒体流的发送 + 接收方向。

**为什么需要预占**（`_addStubTrack`）：主叫发 SDP offer 前必须在 SDP 里声明要接收多少路视频。业务服务器下发 `cameras` 列表后，按数量 `addTransceiver(video)` 预建对应的接收端 m= 行，否则对端推过来的视频会被 SDP 协商丢弃。

## 三、Janus 协议类

### Janus Gateway

**定义**：一个通用的开源 WebRTC 服务器（C 实现），本身不处理业务逻辑，所有功能通过**插件**实现。常见插件：
- `janus.plugin.videocall` — 1v1 视频通话（本项目用的）
- `janus.plugin.videoroom` — 多方会议
- `janus.plugin.streaming` — 单向直播
- `janus.plugin.sip` — SIP 网关

**客户端接入方式**：HTTP Long Polling / WebSocket / RabbitMQ。本项目用 **WebSocket + `janus-protocol` 子协议**。

### Janus message

**定义**：发往 Janus 服务器的所有 JSON 请求的统称。**不是**一个特定字段，而是泛指协议消息。

**通用消息格式**：
```json
{
  "janus": "<动作类型>",
  "transaction": "<12位随机串>",
  "session_id": <Janus session id>,
  "handle_id": <插件 handle id>,
  "token": "<鉴权token>",
  "body": {"request": "<插件API>", ...},
  "jsep": {"type": "offer|answer", "sdp": "..."}
}
```

**动作类型**（`janus` 字段值）：

| 值 | 含义 | 方向 |
| --- | --- | --- |
| `create` | 创建 session | 请求 |
| `claim` | 复用已有 session（重连）| 请求 |
| `attach` | 绑定插件 | 请求 |
| `message` | 插件业务消息（register / call / set / hangup）| 请求 |
| `trickle` | 上报 ICE candidate | 请求 |
| `keepalive` | 心跳 | 请求 |
| `destroy` | 销毁 session | 请求 |
| `success` | 请求成功响应 | 响应 |
| `ack` | 异步请求已收到 | 响应 |
| `error` | 请求失败 | 响应 |
| `event` | 服务器推送的业务事件 | 推送 |
| `webrtcup` | ICE 打通，媒体已通 | 推送 |
| `media` | 某路媒体状态变化 | 推送 |
| `hangup` | 对端或服务器挂断 | 推送 |
| `detached` | 插件已拆除 | 推送 |

**在本项目**：封装在 `janus_client.dart`，每个 `sendMessage` 方法顶部都注释了对应的 Janus 动作。

### Session / Handle / Plugin

Janus 三层身份标识：

| 概念 | 含义 | 生命周期 |
| --- | --- | --- |
| **Session** | 客户端与 Janus 的会话 | WebSocket 连接级，60s 无 keepalive 就被回收 |
| **Plugin** | 功能模块（videocall / videoroom 等）| 服务器端永久加载 |
| **Handle** | Session 在某个 Plugin 上的绑定实例 | Session 生命周期内 |

类比：一次通话就是 `[Session: {Handle A on videocall}, {Handle B on videoroom}]`——一个 Session 可以挂多个插件 Handle。

**在本项目**：`JanusClient._sessionId`、`JanusVideoCallUseCase._handlerId`。

### Transaction（事务）

**定义**：Janus 协议里每条**请求**带一个客户端生成的 12 位随机 `transaction` 字段，服务器回包时原样带回，客户端据此匹配响应。

**作用**：异步 WebSocket 上实现请求-响应模式（类似 RPC 的 requestId）。

**在本项目**：`janus/transaction.dart` 封装了 Promise 模式，`_transactionMap: tid → Transaction` 存住发出去的请求，收到响应时摘除并回调。15s 未响应视为超时。

### janus.plugin.videocall

**定义**：Janus 官方提供的 **1v1 视频通话插件**，负责：
- 用户注册（`register`）—— 以用户名寻址
- 呼叫（`call`）—— 交换 SDP offer
- 接听（`accept`）—— 交换 SDP answer
- 挂断（`hangup`）

**插件 API**（`body.request` 字段值）：

| 值 | 含义 |
| --- | --- |
| `register` | 注册用户名 |
| `call` | 主叫发起，带 SDP offer |
| `accept` | 被叫接听，带 SDP answer |
| `set` | 通用 SDP 更新（restart ICE）|
| `hangup` | 挂断 |

## 四、SDP / ICE 协商类

### SDP（Session Description Protocol）

**定义**：会话描述协议，纯文本格式，描述「**我这边能接受什么媒体 / 用什么 codec / 网络地址在哪**」。

**片段示例**：
```
v=0
o=- 4611732721611386476 2 IN IP4 127.0.0.1
s=-
m=audio 9 UDP/TLS/RTP/SAVPF 111 ...
c=IN IP4 0.0.0.0
a=rtpmap:111 opus/48000/2
a=setup:actpass
```

**关键行**：
- `m=` 媒体行，每路媒体一行（audio / video / application）
- `a=rtpmap:` 编解码器（opus、vp8、h264）
- `a=candidate:` ICE 候选（trickle 模式下这里可能为空）

### Offer / Answer 协商

**WebRTC 的标准协商流程**（以主叫为例）：

```
主叫                              被叫
  │ createOffer() → SDP             │
  │ setLocalDescription(offer)     │
  │ ─── offer ───────────────────→ │
  │                           setRemoteDescription(offer)
  │                           createAnswer() → SDP
  │                           setLocalDescription(answer)
  │ ←───────────────── answer ──── │
setRemoteDescription(answer)
  │                                 │
  │ ←──── ICE trickle ─────────→   │
  │                                 │
  │ ═════════ 媒体流通 ══════════  │
```

**在本项目**：
- 主叫 offer：`doCall()` 中 `_peerConnection.createOffer()` → `_janusClient.call()`
- 收到 answer：`onCallAccepted()` 中 `_peerConnection.setRemoteDescription()`

### JSEP（JavaScript Session Establishment Protocol）

**定义**：SDP 的 JSON 包装，格式：`{type: "offer|answer", sdp: "<纯文本 SDP>"}`。

**作用**：让 SDP 可以作为结构化字段嵌入信令消息。

**在本项目**：所有 Janus `call / set` 请求的 `jsep` 字段就是这个结构。转换函数 `jsepToSdp()` (`janus_video_call_usecase.dart:556`)。

### ICE（Interactive Connectivity Establishment）

**定义**：NAT 穿透框架，通过交换 candidate 找出两端能互通的最佳路径。

**三阶段**：
1. **Gathering** — 收集本端所有可能地址（host / srflx / relay）
2. **Checking** — 两端互相 ping 对方的 candidate，测试连通性
3. **Completed** — 选出最佳路径，打洞成功

### Trickle ICE

**定义**：ICE candidate 的**增量交换**模式。传统 ICE 要等 gathering 完成后一次性塞进 SDP，trickle 模式可以边 gathering 边发，缩短建连时间。

**在本项目**：每发现一个 candidate 就调 `_janusClient.trickleCandidate()`，gathering 完成时发 `completed: true` 收尾。

### STUN / TURN

- **STUN** — 帮客户端探测自己的公网 IP:Port（生成 srflx candidate）
- **TURN** — 在公网上做中继（生成 relay candidate），当两端都在对称 NAT 后面无法打洞时兜底

**在本项目**：`ICE_SERVERS` 常量定义（位于 `data/janus_api.dart`），创建 PeerConnection 时传入。

### DTLS / SRTP

- **DTLS**（Datagram TLS）— UDP 上的 TLS，用于 WebRTC 握手 + DataChannel 加密
- **SRTP**（Secure RTP）— RTP 流的加密，密钥通过 DTLS 协商

**作用**：保证媒体流端到端加密，中间节点（包括 Janus 服务器）看不到明文媒体内容。

## 五、本项目特有概念

### RtcInit

SDK 门面（Facade Pattern）。宿主 App 唯一接触的对象，所有对外 API 都在这里，内部代理到 `JanusVideoCallUseCase`。

### JanusVideoCallUseCase

**全局时序编排器**，SDK 最核心的类。把 SSE / Janus / WebRTC / DataChannel 四条链路组合成完整呼叫流程。

### HttpEngin

**依赖倒置**的抽象接口。SDK 不自带 HTTP 栈，由宿主注入实现，好处是复用宿主的鉴权 / 拦截器。

### VideoCallEvent

对上层 UI 的**事件契约**。SDK 通过 `rtcInit.videoCallEvent` 回调把内部状态抛给 UI 层，事件类型见 `event/video_call_event.dart`（`Connecting` / `Connected` / `StreamOpen` / `DisConnected` 等）。

### Transaction（本地）

不同于 Janus 协议层的 transaction，本项目 `janus/transaction.dart` 是客户端**本地的 Promise 对象**，持有 success / error 回调，配合 `_transactionMap` 实现 RPC 模式。

### KeepAliveChannel

本项目特有的**端到端**心跳机制。**注意**：不同于 Janus session 级 keepalive（那个保服务器不回收 session），这个是跑在 DataChannel 上检测 P2P 通道健康度。

- 2s 一次心跳
- 连续 10 次无应答判死
- 连续 3~9 次无应答上报 loading 状态给 UI

### consistentHashId

一致性哈希路由标识，由业务服务器在 SSE 里下发。WebSocket 建连时放在 `x-hash-id` 请求头，保证同一个用户始终打到集群里同一个 Janus 节点，避免跨节点时 session / handle 丢失。

### randUserFrom / randUserTo

**随机化用户名**。Janus videocall 插件以 username 寻址，为了避免用户名被扫描、猜测或重放攻击，业务服务器每次呼叫都下发一对随机用户名，一次性使用。

### accOn

车机场景下的 **ACC 开关状态**（ACC 即车辆点火状态）。影响呼叫超时策略：
- ACC 开 → 驾驶员可能忙着开车，呼叫超时 35s（7×5s）
- ACC 关 → 一般在休息，呼叫超时 15s（3×5s）

### forceNewToken

token 失效后的重登标志。当 SSE 返回 `SSE_FORCE_NEW_TOKEN` 或 Janus 返回 403 时，带着旧 token 重入呼叫流程，由服务器颁发新 token。

### traceId

**全链路追踪 ID**，由业务服务器在 SSE 首包下发。用于在埋点日志里把「业务服务器事件 + Janus 事件 + 客户端事件」串联起来，排查问题时定位同一次通话的完整链路。
