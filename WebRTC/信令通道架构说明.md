# JanusVideoCallUseCase 信令通道架构说明

本文档说明 `JanusVideoCallUseCase` 在 WebRTC 远程视频通话中使用的各个信令通道，及其分工。

---

## 一、`eventflux` 在远程视频连接中的作用

`eventflux` 是一个 **SSE（Server-Sent Events）客户端库**，在 `JanusVideoCallUseCase` 中**不直接参与 WebRTC 音视频流传输**，而是充当**信令通道的下行推送**。

### 作用定位（`lib/janus/janus_video_call_usecase.dart:178-206`）

它是业务服务器 → 客户端的**单向事件通道**，用于接收呼叫状态变更通知：

- `_startSSEConnect()`：通过 `EventFlux.instance.connect()` 建立 GET 长连接到 `rtcInit.sseUrl`，附带 `devicesUniqueId`、`forceNewToken`、`cameraId` 参数和 token header。
- `_parseSseData()`：解析推送的 `VideoCallRsp`，处理 `SSE_SUCCESS`、`SSE_TIMEOUT`、`SSE_UNAUTHORIZED`、`DEVICE_TOKEN_EXPIRED`、`CALLEE_OFFLINE` 等事件。
- `onDestroy()` 中 `EventFlux.instance.disconnect()` 释放连接。
- 自带重连机制 `_attemptSSEReconnect`（最多 5 次，间隔 3s）。

### 与 WebRTC 的分工

| 通道 | 协议 | 职责 |
| --- | --- | --- |
| **eventflux (SSE)** | HTTP 长连接 | 业务服务器推送呼叫状态（被叫在线 / 超时 / token 过期等），触发 Janus 会话创建 |
| **Janus Client (WebSocket)** | Janus API | 真正的 WebRTC 信令（SDP offer/answer、ICE candidate） |
| **WebRTC PeerConnection** | DTLS / SRTP | 音视频媒体流 P2P 传输 |

`eventflux` 本质上是**呼叫唤醒和状态同步的业务信令**，在 WebRTC 连接建立**之前**驱动整个呼叫流程启动；媒体流本身走的是 WebRTC 的 `_peerConnection`。

---

## 二、客户端 → 服务器的上行通道

SSE 只负责下行推送，上行走的是**另外三条独立通道**，分工明确。

### 1. WebSocket（Janus 信令主通道）—— `lib/wss/web_socket_channel.dart`

承担 **WebRTC 建立连接所需的所有上行信令**。`JanusClient` 通过 `_channel.sendMessage(obj)` 发送 JSON 消息，服务器也从 WebSocket 回包（不走 SSE）。

- **建立**：`WebSocket.connect(_wssUrl, headers: {Sec-WebSocket-Protocol: janus-protocol, x-hash-id: ...})`（`web_socket_channel.dart:65`）
- **发送的消息类型**（`lib/janus/janus_client.dart`）：
  - `CREATE` / `CLAIM` session（:92, :113）
  - `ATTACH` 插件（:134）
  - `register` 用户（:163）
  - `setOffer` / `setAnswer` —— **SDP 交换**（:189, :218）
  - `trickleCandidate` —— **ICE candidate**（:240, :258）
  - `hangup` / `detach` / `destroy`（:278, :295, :449）
  - 保活 keepalive（Timer 周期触发）

### 2. HTTP（业务侧上行）—— `lib/core/http_engin.dart`

抽象接口 `HttpEngin`，由宿主 App 注入实现（SDK 不关心用 dio 还是 http）。只用于**业务级**动作：

- `hangup(connId, deviceId, reason)` —— 通知业务服务器挂断（`janus_video_call_usecase.dart:779`）
- `queryDeviceAcceptStatus(connId, deviceId)` —— 轮询对端接听状态（`janus_video_call_usecase.dart:659`）
- `uploadVideoCallStatus(...)` —— 埋点上报

### 3. WebRTC DataChannel（P2P 点对点，不经服务器）

`DataChannelHandler` 和 `KeepAliveChannel`（`janus_video_call_usecase.dart:128-147`）建立在 `_peerConnection` 上，连接建立后直接与对端设备互通业务消息和保活心跳。

## 三、整体通道汇总

| 方向 | 通道 | 用途 |
| --- | --- | --- |
| ↓ 下行 | **SSE (eventflux)** | 呼叫状态推送 |
| ↑ 上行 | **WebSocket** | Janus 信令（SDP / ICE / 会话） |
| ↑↓ 双向 | **HTTP** | 挂断、状态查询、埋点 |
| ⇄ P2P | **DataChannel** | 端到端业务消息 |

### 设计要点

这是一个 **"SSE 下行 + WebSocket 上行" 的非对称信令架构**：

- **SSE 比 WebSocket 更轻量、更容易穿透 CDN / 反向代理**，适合被叫端常驻监听唤醒；
- **真正的实时信令密集交互仍用 WebSocket**，保证 SDP/ICE 交换的低延迟与双向可靠；
- **业务级动作（挂断、状态查询、埋点）解耦到 HTTP**，交由宿主 App 复用已有的网络栈；
- **端到端业务数据走 DataChannel**，不占用服务器带宽，也不受服务器转发延迟影响。
